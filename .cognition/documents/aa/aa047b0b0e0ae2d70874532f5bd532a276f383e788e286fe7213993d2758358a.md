# Scope: Personalized Session-Start Prime (per-git-user context)

**Status:** APPROVED by Colton 2026-07-07 (all four open questions answered — see Finalized decisions). Vorpid implements; Vince gates.
**Author:** Vince (manager). Date: 2026-07-07.

## Finalized decisions (Colton, 2026-07-07)

1. **Auto-detect** is the default posture: solo graphs (≤1 distinct non-empty stamped email) get byte-identical output; multi-user graphs personalize automatically. `PRIME_PERSONALIZE=auto|on|off` env override still ships.
2. **"Your recent activity" includes episodes, decisions, AND discoveries — capped at 5 of each** (per-type caps, config knobs). Note: this exceeds my original flat-token recommendation; the growth is Colton-approved. Acceptance changes from "≤ ~700 tok" to: measure and report multi-user prime size at the gate; solo prime stays byte-identical.
3. **Task-claiming semantics are pulled IN scope**: when `cognition_update_task` moves a task to `in_progress` (a claim), the server stamps `metadata.claimed_by = {name, email}` from git identity — same mechanism as `created_by`, file-reads only. Free-text `owner` is untouched (stays display-only). "Your Open Tasks" matches `created_by.email` OR `claimed_by.email`.
4. **The `_format_constraints` HEAD-filter fix (open task 0d7e84d52537) folds into WP-P13n-2.**

## Problem

`hooks/session-start.sh` → `python -m vibe_cognition.cognition.prime` injects a single global digest of the graph (constraints, open tasks, workflows, patterns, decisions, incidents). On a multi-user team (e.g. 20 people sharing one graph via git), every section is an amalgamation of *everyone's* work. A returning user gets a digest dominated by coworkers' activity; their own open tasks and recent episodes — the "where did I leave off" signal — may be pushed out entirely by the section caps (task cap 5, decisions 3, patterns 3).

## Goal

Prime output personalized to the **current git user** (resolved server-side, same mechanism tasks already use), showing *their* open tasks and recent activity first, while keeping genuinely shared knowledge (constraints, workflows, high-severity incidents) global. Token budget stays ~flat (v0.13.0 trimmed prime to ~634 tok — a settled decision; we reallocate, not grow).

## What already exists (verified in code)

- `git_identity.resolve_git_identity(repo_path)` — file-read-only (NO subprocess; settled v0.12.1 P0 constraint), returns `{name, email}`, never raises. Prime's `main()` already has `REPO_PATH`; it can call this directly.
- Task nodes: `metadata.created_by = {name, email}` **server-resolved** at creation (`_add_task`, cognition_tools.py:1207-1211) + `metadata.owner` (free text), `metadata.status`, transitions audit log. This is trustworthy attribution.
- All other nodes: `author` is **caller-provided free text** via `cognition_record` — in practice a mix of human names ("Colton Dyck") and agent names ("Vince", "Vorpid"). NOT reliable for per-user filtering. There is an open attribution-honesty task (73f750d8d528) about exactly this entity-vs-task trust asymmetry.
- `PrimeConfig` frozen dataclass (prime.py) + env-var knobs in config.py — the sanctioned place for new toggles.
- Prime is journal-read-only and light-import — it must stay that way (hook timeout; the whole WP-Sidecar/Wedge program exists because heavy imports wedge processes).

## Design

### Phase 1 — Provenance foundation: server-stamped identity on ALL node writes

Extend `_record_node()` to stamp `metadata.recorded_by = {name, email}` (server-resolved via `resolve_git_identity`, exactly as `_add_task` does). The free-text `author` field is untouched (it stays "who/what dictated the record" — agent or human); `recorded_by` is "whose machine/git-identity recorded it." Additive metadata only — no journal schema break, old readers ignore unknown metadata keys. Historical nodes simply lack the key.

This also directly serves the open attribution-honesty task: the trust asymmetry is closed going forward for all node types, not just tasks.

### Phase 2 — Personalize the prime

In `prime.main()`: resolve current identity once (cheap file reads). Then:

**Multi-user auto-detect:** count distinct **non-empty** `recorded_by.email` (fallback `created_by.email`) values in the graph. If ≤ 1 non-empty distinct email (solo graph, or no stamped nodes yet), emit today's global digest **unchanged** — zero regression for every existing solo user, no config needed. (Non-empty matters: `resolve_git_identity` returns `email=""` when unconfigured — a solo user who set `user.email` mid-history would otherwise count as two "users" and flip to multi-user. Peer-review blocker, fixed here.)

**When multi-user, per section:**

| Section | Treatment | Rationale |
|---|---|---|
| Active Constraints | Global, unchanged | Constraints bind everyone |
| Open Tasks | Split: "Your Open Tasks" (match `created_by.email` == current email ONLY) up to N, then "Team critical" (severity=critical/high only) up to M | The core personalization win; tasks have trustworthy attribution TODAY |
| Workflows | Global, unchanged | Shared procedures |
| Documents count | Global, unchanged | One line |
| Recent Patterns / Decisions | Global, unchanged in v1 | Team knowledge; relevance-biasing needs semantics (rejected, below) |
| Recent Incidents | Global, unchanged | High-severity incidents matter to everyone |
| **NEW: "Your recent activity"** | Last N episodes/nodes where `recorded_by.email` == current email | The "where did I leave off" signal; grows in value as Phase 1 stamps accumulate |

**Matching rule:** email is the ONLY match key (stable, server-stamped: `created_by.email` for tasks, `recorded_by.email` for stamped nodes). `metadata.owner` and `author` are free text (agent names, unvalidated) and are **display-only — never used for matching** (peer-review should-fix: an owner-name fallback would both leak tasks to the wrong person on loose name matches and silently drop agent-owned tasks). No email resolvable (unconfigured git identity) → global digest.

**Token budget:** personal task lines replace (not add to) global task lines; "Your recent activity" (~4 lines) is funded by the team-task cap reduction. Acceptance criterion: multi-user prime ≤ ~700 tok measured; solo prime byte-identical to today.

**Config:** new `PrimeConfig` fields + env vars (e.g. `PRIME_PERSONAL_TASK_CAP`, `PRIME_TEAM_TASK_CAP`, `PRIME_RECENT_ACTIVITY_LIMIT`, `PRIME_PERSONALIZE` override: auto|on|off). Defaults = auto-detect.

### Rejected alternatives (record with decision on approval)

- **Semantic relevance-biasing of patterns/decisions to the user's recent work** — requires embeddings/chromadb in the prime path; prime must stay light-import (Wedge/Sidecar program). Rejected for v1; could later go through the sidecar.
- **Per-user config file mapping agent names → humans** — manual bookkeeping that rots; server-stamped `recorded_by` gets the same result automatically.
- **Client-passed identity parameter** — caller-provided identity is exactly the trust gap tasks already solved by server-resolving; don't reintroduce it.
- **Filtering constraints/incidents per user** — actively harmful; these exist to bind/warn everyone.

### Known limits (document, don't hide)

- Historical nodes (pre-Phase-1) lack `recorded_by`; "Your recent activity" starts sparse and fills in with use. Tasks personalize immediately (created_by exists since tasks shipped).
- `recorded_by` = "recorded on a machine with your git identity." Agent-recorded nodes on your machine count as yours — for the manager-records-everything workflow this is the desired semantic, but it conflates on shared machines/CI, and a manager recording a teammate's relayed fact stamps the *manager's* identity, not the teammate's. Acceptable for v1; must be stated in user-facing docs, not just here.
- Two graph writers with the same email on different machines are the same "user" — correct behavior.
- **Pre-existing composition hazard (peer-review finding):** `_format_constraints` has NO supersession/HEAD filter (unlike workflows and documents, which both have one) — the known "superseded constraints injected" defect (open task 0d7e84d52537). This plan keeps constraints "unchanged," which inherits the duplication and inflates the baseline the token math is measured against. Open question 4 below asks whether to fold the HEAD-filter fix in (cheap: same file, same pattern used twice already) or keep it separate scope.

## Work packages (Vorpid implements; Vince gates)

### WP-P13n-1 — Provenance stamps (Phase 1 + claiming)

Scope: (a) `metadata.recorded_by = {name, email}` server-stamped in `_record_node` on every node write; (b) `metadata.claimed_by = {name, email}` server-stamped in the `cognition_update_task` path when status moves to `in_progress`. Both use `resolve_git_identity` (file-reads only — subprocess is BANNED, v0.12.1 P0).

Acceptance criteria:
- Every node created via `cognition_record` / `_record_node` carries `metadata.recorded_by` with name (non-empty) and email (may be "").
- Claiming a task (status → `in_progress`) stamps `metadata.claimed_by`; re-claiming re-stamps; other status transitions do not touch it.
- Identity-resolution failure degrades to `{"name": <getpass or "unknown">, "email": ""}` — never raises, never blocks the write.
- Old journals (nodes without the new keys) load and behave unchanged.
- Fix + proof in the same commit; tests assert the specific stamped values, not just presence.

Known-intentional (do NOT "fix"):
- `author` stays caller-provided free text — no validation/normalization.
- `owner` stays free text and display-only.
- `created_by` on tasks stays exactly as-is.

### WP-P13n-2 — Prime personalization (Phase 2 + constraints HEAD-filter)

Scope: identity resolve in `prime.main()`; multi-user auto-detect (distinct NON-EMPTY emails > 1); task-section split ("Your Open Tasks" matched on `created_by.email` OR `claimed_by.email`, then "Team critical" high/critical only); new "Your recent activity" section (your episodes/decisions/discoveries, 5 each, `recorded_by.email` match); `PRIME_PERSONALIZE=auto|on|off` + per-cap env knobs in config.py/PrimeConfig; AND fold in the `_format_constraints` supersession HEAD-filter fix (task 0d7e84d52537), mirroring `_format_workflows`' existing HEAD-filter pattern.

Acceptance criteria:
- Solo graph (≤1 distinct non-empty stamped email): output byte-identical to current main, EXCEPT superseded constraints no longer duplicate (the folded fix — assert both directions).
- Multi-user fixture graph: personal sections appear, matched strictly by email; no name-string matching anywhere.
- Unconfigured git identity (empty email): global digest, no personal sections, no crash.
- Superseded constraint (incoming SUPERSEDES edge) never renders; its HEAD does.
- No new imports in prime.py beyond stdlib/existing (no torch/chromadb — hook path stays light).
- Measured multi-user prime token size reported in the PR description.

Known-intentional (do NOT "fix"):
- v0.13.0 trim knob defaults are settled decisions — don't rebalance them.
- Constraints/workflows/incidents/documents remain GLOBAL sections by design.
- `_format_workflows` HEAD-filter is the pattern to mirror; its known limits (if any) come to me first, don't diverge silently.
