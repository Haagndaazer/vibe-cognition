## Problem — reframed after review
Main Claude Code agents frequently skip the edge-analyzer subagent during /vibe-curate and hand-author
edges, missing the context the analyzer fan-out surfaces.

Root cause (honest version): this is NOT an information-leak problem. As of v0.14.0 (task 7d1c151b0372,
commit fb57594 — deliberate, high-severity fix) the full edge-semantics table lives in the
cognition_add_edge / cognition_add_edges_batch docstrings, so every agent holding those tools knows how to
author edges regardless of what any SKILL.md says. The failure mode is a PROCESS-COMPLIANCE problem: the
sanctioned path is a 4-step pipeline the main agent must drive (assess → batch → spawn analyzers → review →
commit → cluster), and agents shortcut multi-step processes. The fix is to collapse the sanctioned path to
ONE tool call, so the shortcut IS the sanctioned path.

## Evidence gathered
1. **Prior art in this repo**: v0.7.0 (commit 7395672) REMOVED a background local-LLM auto-curator
   (CognitionCurator, Ollama qwen3:8b) as "unreliable". Implementation flaws: num_ctx 4096, one node at a
   time, `format:json` without schema-constrained decoding, top-8 similarity candidates, in-process daemon
   thread inside the MCP server, no checkpointing/audit trail.
2. **Sibling projects** (memcog + linglang-teacher): BOTH use a cheap hosted API model (Gemini 3 Flash),
   NOT a local model. MemCog evaluated a Claude-based curator and pivoted to Gemini on cost. Both succeed
   via deterministic engineering: trigger-on-write / session-end, checkpointed passes, strict
   responseSchema JSON, similarity pre-filtering (≥0.65), confidence gates (≥0.8), caps (≤3 edges/node),
   fail-open, cost ledgers, pass logs. Shared scar: "bad edges are worse than missing edges."
3. **Local LLM landscape (July 2026; 3900X / 64GB / RTX 4080 16GB)**: GPT-OSS-20B (~12GB MXFP4,
   ~140 tok/s on a 4080) is the first-trial candidate; Qwen3.6-35B-A3B (MoE CPU-offload into 64GB RAM) the
   quality ceiling; Gemma 4 12B the co-resident option. JSON *validity* is solved locally
   (grammar-constrained decoding). Semantic *judgment* is not: ~2x composite-intelligence gap vs Haiku
   class, weakest exactly on supersedes/contradicts/led_to direction. Cluster naming is fine locally.
4. **Claude Code mechanics**: subagent→subagent nesting since v2.1.172 (depth ≤5), needs Agent in the
   subagent's tools. Plugins ship custom agents (`agents/*.md`) with pinned model + tool list — proven in
   this repo (`agents/plan.md` is live as `vibe-cognition:Plan`); MCP tools must be ENUMERATED in the
   frontmatter tools list (as plan.md does). Backgrounded subagents die with the session; hooks cannot
   spawn detached work; headless `claude -p` daemons are unsupported/fragile.
   UNVERIFIED (spike required, WP-0): (a) `run_in_background` behavior when launching a plugin agent that
   itself spawns subagents; (b) whether an `Agent(...)` allowlist in a subagent's frontmatter actually
   restricts which agent types it can spawn (docs suggest it may not — treat any restriction as
   prompt-level until proven); (c) behavior on Claude Code < v2.1.172 (error vs silent no-op).

## Option A — Local-model background curator daemon (PARK)
Out-of-process worker (GPT-OSS-20B via Ollama, structured outputs, batched, checkpointed, confidence-gated,
audit-logged), triggered on record via debounced queue or timer.

Pros: zero marginal token cost; survives session close; keeps the plugin's "fully local, no API keys"
promise; a Python pipeline can never skip its own steps; the hardware is adequate.
Cons: re-litigates the v0.7.0 removal — the implementation flaws are fixable, but the model-inherent
judgment gap sits exactly on the hardest edge types; heavyweight dependency for OTHER plugin users
(distributed plugin → they'd need Ollama + ~12GB model + GPU); VRAM contention; per-model prompt
maintenance; a bad background curator silently poisons the graph.
If ever pursued: memcog-style hybrid — local model proposes with a confidence field; low-confidence and all
supersedes/contradicts candidates escalate to a Haiku review queue; A/B on ~50 real node batches vs the
current pipeline BEFORE committing; provenance tag `source: "local-curator"` so edges are sweepable.
Decision: park. Record as a decision node with the A/B gate as the reopening condition.

## Option B — Self-contained backgrounded curation subagent (RECOMMENDED)
Plugin ships three custom agents; SKILL.md shrinks to a one-call launcher.

- `agents/curate-orchestrator.md` — model: sonnet. Tools: Agent + enumerated cognition_* MCP tools (+ Read).
  System prompt absorbs today's SKILL.md steps 1–4 verbatim in spirit: assess uncurated, batch 5–10, spawn
  edge-analyzers on Haiku, review proposals (self-refs, retired duplicate_of, task part_of ban, vague
  reasons), commit via add_edges_batch with per-edge source, mark_curated per batch, cluster-analyzer pass,
  final report. CONTAINMENT (Colton, approval round): the orchestrator FINALIZES all edges/summary nodes
  itself; its report back to the main instance is success/failure + bare counts ONLY (no edge content,
  reasons, or node IDs) — the main instance never interacts with the curation process. This deliberately
  retires the current SKILL.md Step 4 narrated-report mandate; detailed narration lives in the
  orchestrator's transcript and the graph/dashboard. Graceful degradation: if the Agent tool is unavailable (old Claude Code), perform the
  analysis itself INLINE — the essential analyzer protocol (edge-type rules, review criteria, caps) is
  embedded in the orchestrator's own system prompt (it needs most of it for the review step anyway), so the
  degraded path does not depend on locating sibling files. A Sonnet doing the full protocol inline still
  beats today's failure mode.
- `agents/curate-edge-analyzer.md` — model: haiku (pinned); body = current edge-analyzer.md.
- `agents/curate-cluster-analyzer.md` — model: haiku (pinned); body = current cluster-analyzer.md.
- `skills/vibe-curate/SKILL.md` — thin launcher: call get_status; if uncurated == 0 report and stop; else
  launch `vibe-cognition:curate-orchestrator` in the background, tell the user "curation launched — N
  uncurated nodes; completion will be reported; ground truth is get_status's uncurated count", done. Keeps
  the existing concurrency note (two sessions curating = documented wasted-work case, unchanged) plus one
  new line: "if a curation run was just launched and hasn't reported completion, don't launch another"
  (background mode widens the concurrency window — the main session keeps recording while the orchestrator
  runs; nodes recorded mid-run stay uncurated until the next launch, which is fine).

Why this fixes the failure mode (honest framing): not by hiding edge knowledge (impossible — see root
cause) but by (a) making the sanctioned path SHORTER than the shortcut — one launch call vs hand-authoring
a batch of edges — and (b) putting pipeline compliance inside a fixed agent system prompt that the
executing agent receives whole, rather than a skill the main agent skims while juggling other context.
Residual risk that a main agent still hand-authors edges unprompted is accepted and mitigated by docstring
demarcation (WP-3), not claimed to be eliminated.

Pros: same quality as today (Sonnet judgment + Haiku fan-out); main session not blocked; zero new
infrastructure or hardware requirements; rides the user's existing Claude Code auth (no API keys — keeps
the plugin promise, unlike the sibling-projects' Gemini path); small diff (mostly relocating existing
prompt text).
Cons/limits: still costs API tokens; dies with the session — mitigated by idempotent per-batch
mark_curated (next run resumes) plus the launch-report telling the user how to check; observability of a
silently-dead orchestrator is get_status's uncurated count + the background-agent completion notification
(WP-4 verifies both); Claude Code < v2.1.172 degrades to inline-orchestrator mode (see agent definition);
the main agent still physically holds cognition_add_edge (callers indistinguishable server-side) but the
tool surface forbids it — ONLY the curate-orchestrator may use add_edge/add_edges_batch/mark_curated
(Colton's forceful demarcation, WP-3b item 3; supersedes the earlier targeted-manual-edge carve-out).
