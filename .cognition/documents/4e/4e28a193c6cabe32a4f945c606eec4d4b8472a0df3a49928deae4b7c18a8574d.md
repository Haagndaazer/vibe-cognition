# WP-Nudge-1: Session-start update nudge with call-to-action

**Status:** APPROVED (Colton, 2026-07-23) — peer-reviewed (Sonnet, approve-with-edits, all findings folded in)
**Implementer:** Vorpid
**Manager:** Vince
**Epic/train:** upgrade-UX (siblings: stale-first-session race 43d3c3dab10f, MCP process leak d2a62f38cc88 — NOT dependencies, but see Seams)

## Problem

Claude Code's plugin auto-update is **off by default for third-party marketplaces** and cannot be enabled by the marketplace maintainer (verified against official docs 2026-07-23: no `autoUpdate` field in the marketplace.json schema; `extraKnownMarketplaces.autoUpdate` is managed-settings-only). So our users get **no signal at all** that a new vibe-cognition version exists unless they manually poll `/plugin`. Colton's ruling: build a **nudge-only** alert — surface "new version available" with a clear call to action at session start. **The user stays in control at all times: we never run an update on their behalf.**

## Approach (one direction, chosen)

A new module `src/vibe_cognition/update_check.py`, invoked from `hooks/session-start.sh` as a new step between migrate-mcp (Step 3) and prime (Step 4), mirroring the migrate_mcp pattern exactly: it prints a one-line note (or nothing) to stdout; the hook captures it into `VIBE_UPDATE_NOTE` and passes it to prime via env; prime appends it to the injected context the same way it surfaces `VIBE_MIGRATION_NOTE`.

### Version discovery (verified viable 2026-07-23 on Colton's machine)

1. **Installed version:** read `${CLAUDE_PLUGIN_ROOT}/.claude-plugin/plugin.json` → `version`. (The hook already has `PLUGIN_ROOT`; pass it to the module via env or argv.)
2. **Latest released version:** two unauthenticated HTTPS GETs (both verified 200 without auth):
   - `https://raw.githubusercontent.com/Haagndaazer/colton-claude-plugins/main/.claude-plugin/marketplace.json` → find the `vibe-cognition` entry → `source.sha` (the release pin).
   - `https://raw.githubusercontent.com/Haagndaazer/vibe-cognition/<sha>/.claude-plugin/plugin.json` → `version`.
3. **Compare version strings, never SHAs.** Ground truth from Colton's machine: `installed_plugins.json` records `gitCommitSha: 811350d…` while the marketplace pin is `5199ee3…` — *both are v0.28.0*. SHA comparison would false-nudge today. Parse versions as dotted-int tuples (all our versions are plain `X.Y.Z`; on any unparsable version → treat as equal, i.e. **no nudge**). Nudge **only when remote > installed** — remote == or < installed is silence (a pin rollback must not nudge).

Why the pin and not the code repo's `main`: main is routinely ahead of the pin (code lands, Loki pins later — "pin-relay batching" pattern). Nudging off main would announce unreleased versions.

### Throttle — zero cost on the steady path

- A stamp file `${CLAUDE_PLUGIN_DATA}/update-check.json` (plugin-data dir: survives updates, machine-local, never in git): `{checked_at, remote_version}`.
- **Bash-side gate — keyed off the stamp file's mtime, NOT its JSON payload** (peer-review finding: parsing `checked_at` in bash needs jq/python and would reintroduce the process spawn the gate exists to avoid). Use `find -mtime`/`stat`-style mtime comparison against the file itself; the JSON payload is for the Python side only. Fresher than 24h → skip the step entirely (no uv process, no network). This keeps the common session start at zero added cost — important given the WP-A hook-timing work.
- **`set -e` hazard specific to this gate:** the "stamp is stale → proceed" branch is the *routine* false-exit-code path, not an error path — a bare `[ ... ]` age test outside an `if`/`||` construct kills the hook on the common case. Structure it as an explicit `if`; this is a different hazard shape from the `|| true` guards on unexpected failures.
- When the check does run: hard network timeout **3s per request** (urllib, stdlib only — no new deps), total worst-case ≤ ~6s once per 24h. Any failure (timeout, 404, DNS, JSON parse, our entry missing) → empty note, exit 0, stamp still written (so a flaky network doesn't retry every session).
- Nudge cadence = check cadence: at most one nudge per 24h while outdated. No per-version dismissal state in v1 (keep state minimal; the 24h cadence is the anti-nag).

### The nudge text (exact, an acceptance criterion)

> vibe-cognition v{remote} is available (you have v{installed}). To update: run `/plugin update vibe-cognition@{marketplace}`, then restart Claude Code. Updating is always your call — this notice is informational only. Disable it with `VIBE_UPDATE_NUDGE=off`.

- `{marketplace}` is **derived, not hardcoded** (peer-review finding: a fork/mirror install would get a wrong CTA): the plugin cache path is `.../cache/<marketplace>/vibe-cognition/<version>`, so the marketplace name is the grandparent directory of `${CLAUDE_PLUGIN_ROOT}` — derive it from the path; fall back to plain `/plugin update vibe-cognition` phrasing if derivation fails. The *version-check URL* stays pinned to our canonical marketplace repo (it is our distribution channel); on a non-canonical install the fetch simply 404s → silence, which is correct.
- CTA says **restart**, not `/reload-plugins`: reload leaves the prior MCP server running (open task d2a62f38cc88) and skips SessionStart hooks (open task 43d3c3dab10f evidence). Vorpid: verify the exact `/plugin update` argument syntax against current Claude Code docs during implementation; the `name@marketplace` form is believed correct.

### User control & privacy (Colton's core requirement)

- **Nudge-only.** The module performs GETs and writes its own stamp file — nothing else. It never invokes `claude`, never touches the plugin cache, never mutates the user's install.
- **Kill switch:** config knob named `vibe_update_nudge: bool = True` in `config.py` — the `vibe_` prefix is load-bearing (peer-review finding): pydantic-settings derives the env var from the field name, so `vibe_update_nudge` → `VIBE_UPDATE_NUDGE`, making the config knob and the user-facing env var in the nudge text literally the same variable. This repo already uses exactly this trick (`vibe_cognition_no_git_hygiene` → `VIBE_COGNITION_NO_GIT_HYGIENE`). A bare `update_nudge` field would derive `UPDATE_NUDGE` and silently ignore the documented `VIBE_UPDATE_NUDGE`. Also honored bash-side: if `VIBE_UPDATE_NUDGE` is `off`/`0`/`false` in env, the hook skips the step without spawning Python. **Do NOT copy from the prime knobs:** the `Literal["auto","on","off"]` tri-state (`prime_personalize`) — there is no "auto" here; plain bool.
- **RULED (Colton, 2026-07-23): default ON with opt-out.** Stated intent: "this is really meant to ensure users always have latest." README gets the prominent disclosure ("checks GitHub for the released version at most once per day; nothing about you or your project is sent; disable with `VIBE_UPDATE_NUDGE=off`") and the session-start notice text itself names the off switch. Opt-in was rejected (near-zero adoption defeats the purpose).

## Known-intentional (do NOT "fix" these)

- `uv run --no-sync` everywhere in the hook: deliberate (venv-lock rationale); the stale-first-session task owns re-examining it.
- `generate_prime()` is pure read-only; stamps (last-seen.json) are written only by hook-side code. The update stamp follows the same doctrine: written by the update_check step, never by prime.
- Hook stdout is reserved for the final JSON; all diagnostics to stderr via the `_bc` breadcrumb pattern (add `update_check_start/done` breadcrumbs).
- Hook must exit 0 on every path (`set -euo pipefail` is live — guard every new command with `|| true` capture, as migrate_mcp does).
- The marketplace schema has no version field — that's upstream reality, not a gap to patch.

## Acceptance criteria

1. Remote pin version > installed: session-start context contains exactly the nudge line above; everything else in the prime output unchanged.
2. Remote == installed, remote < installed, unparsable versions: no nudge line.
3. Network down / timeout / 404 / malformed JSON / plugin entry absent from marketplace.json: no nudge, hook exits 0, prime context otherwise intact, total added latency bounded by the request timeouts.
4. Stamp fresher than 24h: no update-check `uv`/python invocation appears in the fake-uv invocation log (the established `test_hooks_shell.py` pattern — a fake `uv` on PATH logging every call; peer review: this is the assertable form of "no process spawned").
5. `update_nudge=false` (config) or `VIBE_UPDATE_NUDGE=off` (env): no network, no nudge, no stamp write.
6. Concurrent session starts (two hooks racing on the stamp): no crash, no torn stamp (write-temp-then-rename or tolerate-and-rewrite; either is fine, pick one and test it).
7. Tests: unit tests for version parse/compare (incl. unequal-length `0.9`/`0.28.0`, junk input), fetch-layer mocked (NO live network in tests), stamp throttle logic, bash-path skip covered in `test_hooks_shell.py` style. Windows-safe (thread-based timeout norm applies).
8. Docs: README section (what it does, exactly what is fetched, how to disable), CHANGELOG entry. **Also edit the two existing absolute "no cloud services" claims (README lines ~8 and ~67)** to stay truthful next to the new disclosure — narrow them to "all processing and storage happens on your machine" (peer-review finding: adding a disclosure without touching those lines leaves the README self-contradicting). One contributor-facing line noting dev sessions on this repo itself will make the daily check unless `VIBE_UPDATE_NUDGE=off`. Version bump per release procedure (user-facing → bump both `pyproject.toml` and `plugin.json`).

## Seams checked (pairwise with inherited constraints)

- **Stale-first-session race (43d3c3dab10f):** the nudge reads versions from disk + network, never from the running MCP server — so it stays correct even when the server serves stale code. CTA says restart (not reload) partly because of this.
- **Process leak (d2a62f38cc88):** no interaction — the check spawns no server and its process exits after printing.
- **Venv-lock constraint:** check runs `--no-sync` in the existing venv; no sync added.
- **Hook timing (WP-A breadcrumbs):** bash-side 24h gate keeps steady-state cost at ~zero; breadcrumbs make the once-daily cost visible.
- **Journal/graph:** untouched — this WP writes no cognition state.
