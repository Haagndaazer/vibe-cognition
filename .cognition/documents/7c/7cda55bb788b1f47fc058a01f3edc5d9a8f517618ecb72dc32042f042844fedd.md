## FINALIZED RULINGS (Colton, 2026-07-14 — not up for relitigation)

1. Person nodes model HUMANS ONLY. Agent identity never persists in the graph (agents live in teammate-comms). Fields: name, role, seniority, reports-to (direct manager). Patterned after teammate-comms profiles but fully separate.
2. Person nodes are updated IN PLACE (not supersession-versioned) with an audit trail of who changed what, when.
3. Email is the identity key (established WP-P13n match key).
4. Agent-origin bool on every node write; client-declared; DEFAULT TRUE ("via agent") when undeclared — human-origin is the deliberate, marked assertion. Human input will outweigh agent input in retrieval (that weighting is WP-TC9, later — this WP ships the DATA, not the weighting).
5. Onboarding (prime-triggered introduce-yourself) is WP-TC7, separate — but this WP's tools must be sufficient for TC7 to build on without further schema change.

## PROBLEM

The graph stamps unforgeable human identity (recorded_by/created_by/claimed_by) but knows nothing ABOUT those humans — no role, seniority, or reporting line — and cannot distinguish "Colton dictated this" from "an agent recorded this on Colton's machine". Everything downstream in the epic (seniority weighting, role-aware prime, onboarding, dashboard person chips) consumes this layer; per Colton it ships first.

## DESIGN — WP-TC5: person node type

**Node type.** New `CognitionNodeType.PERSON` ("person"). Follow the workflow-node-type precedent for full node-type survival across the stack — the COMPLETE touchpoint list (peer-review verified):
- models.py enum; storage/journal replay.
- **`_INERT_TYPES` in storage.py: add PERSON explicitly** — the deterministic-edge matcher classifies unknown types as "entity" by default and would auto-mint part_of edges; task/workflow both gate this explicitly rather than relying on empty-refs-by-convention (workflow WP treated this as a blocker; same here). Test case in tests/test_deterministic_edges.py.
- Embedding path (_embed_entity_node — verified no changes needed beyond wiring the type).
- Dashboard TYPE_COLORS (one line; unknown types already degrade to gray).
- cognition_record docstring's NODE TYPES table: person is NOT creatable via cognition_record — dedicated tools only, like document.
- **Hardcoded node_type lists in tool docstrings** (cognition_search, cognition_get_history, cognition_get_edgeless_nodes, and any sibling with a prose type list): add "person" — acceptance promises person is searchable, so the tool contracts must say so.
- **readme.py `COGNITION_GUIDE`** (Node types section + Tool groups table): required by the write-trigger/retrieval-trigger survival pattern. Verified safe: COGNITION_GUIDE is served only by the on-demand cognition_readme tool, NOT injected at session start — updating it does not violate the no-prime-changes rule.
- SKILL.md tool table + README (test_doc_drift.py already force-fails until both mention the new type).
- **NOT SERVER_INSTRUCTIONS** — instructions.py is surfaced every session and re-injected after compacts; editing its standing-practices prose would be a behavioral/onboarding change, which is TC7's territory. Leave it untouched this WP.

**Shape.** summary = "Name — role" (display); detail = free-text bio/notes (optional). Structured profile lives in metadata (server-controlled write path):
- `metadata.person = {email, name, role, seniority, reports_to_email}` — email casefolded at write; **seniority vocabulary PINNED: "owner" > "senior" > "mid" > "junior"** (stored as string, validated against this closed set, casefolded; ordering is a module-level constant so TC9 imports it rather than re-declaring); reports_to_email may be "" (top of chain).
- `metadata.profile_history = [{changed: {field: {from, to}}, at, by}]` — append-only; `by` server-resolved (same mechanism as task transitions).
- `metadata.recorded_by` stamped as usual.

**Invariant: one person node per email.** Registering an email that already has a person node returns the existing node with a clear "already_registered" signal (never a silent duplicate, never an error that loses the caller).

**New MCP tools** (dedicated, like task tools — cognition_update_node stays narrative-only):
1. `cognition_register_person(name, role, seniority, reports_to_email?, email?, detail?)` — creates the node. **Identity rule: when `email` is omitted, the server resolves the CURRENT git identity's email and registers self** (the onboarding path — impersonation-resistant). Passing an explicit `email` registers someone else (e.g. manager pre-registering a teammate) — allowed, trust-based per Colton, but the profile_history/recorded_by records who did it. Empty resolvable email + no explicit email → clean error (established WP-P13n empty-email edge case).
2. `cognition_update_person(email_or_id, name?, role?, seniority?, reports_to_email?, detail?)` — in-place field updates; every call appends one profile_history entry; pass "" to clear reports_to. Anyone may update anyone (local trust domain; audit trail is the control) — docstring says so plainly.
3. `cognition_get_person(email_or_id)` — full node incl. profile_history; plus `cognition_list_people()` — roster view {email, name, role, seniority, reports_to_email}. (If implementation finds get/list foldable into one tool cleanly, propose it at gate — surface minimalism preferred.)

**reports_to semantics.** Self-reporting rejected. Cycles rejected (A→B→…→A). **Implementation note (peer-review): this is NOT the task-parent guard's shape** — task's guard walks real PART_OF edges and hard-errors on a missing parent; reports_to is a plain metadata email string with dangling explicitly legal. Walk the email-keyed chain via person-node scan (casefolded match, WP-P13n linear-scan convention — no email index exists), and STOP GRACEFULLY at any email with no backing node (dangling terminates the walk, it is not an error). reports_to_email need not resolve to an existing person node (a manager may be registered later) — dangling is legal, surfaced in get/list output as `reports_to_registered: false`.

**Summary regeneration.** `cognition_update_person` regenerates `summary` ("Name — role") whenever name or role changes, so display never desyncs from the profile. Known accepted drift: a generic `cognition_update_node` narrative edit on a person node can hand-alter summary; next update_person re-normalizes it. Document, don't guard.

**Embedding.** Person nodes embed like entities (searchable: "who is Vorpid's human", "who owns releases"). Re-embed on update (established update-path pattern — both vector and metadata).

**Prime.** NO prime changes in this WP (onboarding notice is TC7). Solo prime output byte-identical to current main — assert it.

## DESIGN — WP-TC6: agent-origin bool

**Param.** `from_agent: bool = True` added to `cognition_record`, `cognition_add_task`, `cognition_store_document`, **and the new `cognition_register_person`/`cognition_update_person`** (person nodes ship in the same batch — without this, every person node would be born "legacy/unknown provenance", defeating the point of pairing TC5+TC6). Stamped as `metadata.from_agent`. Default TRUE per ruling — an undeclared write is honestly "via agent"; `from_agent=false` is the marked human-dictation case. Docstrings must state: set false ONLY when the human explicitly dictated/authored the content; when in doubt leave default.

**Reads.** Surfaced wherever provenance already surfaces: cognition_search result rows, get_node, list_tasks rows. Missing key (all pre-TC6 nodes) is rendered as absent/unknown — NEVER coerced to true or false (TC9's ruled fallback: visible "unverified/legacy" treatment; don't pre-empt it here with a fake value).

**No weighting in this WP.** Data only.

## BACK-COMPAT (hard requirements)

- Additive metadata only; old journals load unchanged.
- **Cross-version reader behavior — ALREADY TRACED at peer review (verify with a test, but the finding is established):** journal replay on old versions is safe (type stored as raw string, no enum validation); live-search catch-up is safe (per-node try/except skips the person node); BUT the old version's **startup embedding sync has NO per-node guard** — the first person node raises inside `_sync_cognition_embeddings`, the outer handler logs "Background initialization failed", and every node behind it in iteration order silently never embeds, recurring on every restart until upgrade. Also `cognition_update_node` on a person node from an old client returns a raw enum-error. NO design change needed on our side — but (a) add a new-version regression test pinning that OUR sync has per-node tolerance for unknown future types (learn from the old version's defect, don't inherit it), and (b) CHANGELOG/README rollout note: teammates sharing a graph should upgrade promptly once person nodes land, or their local search coverage silently degrades.
- `.cognition/` layout unchanged; no per-user files; no config changes; env-var surface unchanged.

## REJECTED ALTERNATIVES (recorded — do not resurrect)

Supersession-versioned person nodes (Colton: in-place); committed roster/config file (rots; fails tool-surface self-sufficiency); agent names persisted in graph (agents stay in teammate-comms); client-passed caller identity for self-registration (server resolves; explicit-email path is data about a third party, not caller identity); person creation via generic cognition_record (dedicated tools keep the invariant + audit trail enforceable).

## KNOWN LIMITS (document, don't fix here)

reports_to and seniority are trust-declared and peer-editable (audit trail is the control — local trust domain). from_agent is client-declared and unverifiable by the server. Legacy nodes have no from_agent. Person data is as stale as the team keeps it; onboarding (TC7) reduces cold-start, nothing enforces freshness.
