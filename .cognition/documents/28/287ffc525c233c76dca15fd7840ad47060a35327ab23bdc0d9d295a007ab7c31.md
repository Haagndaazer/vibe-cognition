## RULINGS (Colton, recorded — not up for relitigation)

- Trigger is FROM THE PRIME (decision 8f3612ad9583): prime detects the missing person node and injects a notice directing the agent to run the introduce-yourself conversation. We trust the human's answers (name, role, seniority, who they directly report to).
- Sub-question delegated to this brief: the solo-project decline/snooze path.

## PROBLEM

The graph can now model humans (TC5), but nothing creates person nodes organically. Colton's ruling: when a user first interacts with vibe-cognition, auto-detect that they're new and prompt them (via their agent) to introduce themselves. Without this, TC9 weighting and TC16 role-aware prime run on an empty roster.

## DESIGN

**Detection (in `generate_prime`, light-import safe).** All primitives already exist in prime.py's import set: `current_email` is already resolved and casefolded (prime.py:395); person nodes are reachable via `storage.get_nodes_by_type(CognitionNodeType.PERSON)` with email at `metadata.person.email` (already casefolded at write by TC5). The notice fires when ALL hold:
1. `current_email` is non-empty (empty git email → no notice; matches the established empty-email edge-case handling — never key on a blank).
2. No person node has `metadata.person.email == current_email`.
3. The email is not declined (see decline file below).
4. The `prime_onboard` kill switch is on (see config below).

Fires for BOTH solo and multi-user graphs — Colton's ruling is "auto detect a new user", not "a new teammate"; solo users get the decline path instead of a carve-out. **Empty-graph interaction (peer-review corrected): the notice and the empty-graph ONBOARDING_BLOCK are MUTUALLY EXCLUSIVE by construction** — main() (prime.py:521-529) appends ONBOARDING_BLOCK on the empty branch and only calls generate_prime on the else branch, and detection lives inside generate_prime. On a genuinely empty graph the block is the whole onboarding surface; the person notice first fires the session AFTER the first node lands. This is the intended design (do not add a detection path to main()'s empty branch). Ordering with other prepends: migration note and rehydrate warning stay first (test_prime.py:580-597 ordering pins must keep passing).

**Notice position (pinned for byte-testability):** the notice is the FIRST section of generate_prime's returned string, before Active Constraints — it's an action item; ambient context follows. In the full hook payload that yields: migration note → rehydrate warning → [notice → constraints → …].

**Decline-file path resolution:** generate_prime locates the file via `storage.cognition_dir` (public property, storage.py:126-129) — `storage.cognition_dir / "onboard-declined"`. No signature change to generate_prime needed.

**Notice content (one short section, exact text owned by implementation, contract here).** Must direct the agent to:
- Tell the human the graph doesn't know them yet, and ask for: name, role, seniority (owner | senior | mid | junior), and who they directly report to (optional — may be nobody or someone not yet registered).
- Then call `cognition_register_person` OMITTING `email` (server resolves — impersonation-resistant) with **`from_agent=false`** — the profile content is human-dictated by construction, which is exactly the marked case TC6 defined. The notice says this explicitly so agents don't leave the default.
- If the human declines or wants to be asked later: append their email to the decline file (below) — one line, done; do NOT register a placeholder person node.
- Keep it short: this rides in every session start until resolved; target ≤ 10 lines of injected text.

**Decline/snooze path (the ruled sub-question — my resolution).** A per-machine, git-ignored plain-text file: `.cognition/onboard-declined` — one casefolded email per line; prime skips the notice for any listed email. Rationale:
- Per-MACHINE is the right scope: git identity is per-machine config, and the file never syncs — a human who declines on their laptop isn't silently declining for a teammate.
- Plain file written by the AGENT (the notice instructs it), read by prime with stdlib only — no new MCP tool (keeps Gate B surface unchanged), no graph write (a decline is not knowledge), no new state mechanism (precedent: `.cognition/.git-hygiene-managed` marker, gitignored, .cognition/.gitignore:3).
- "Snooze" == decline (remove the line to be asked again; the notice's decline sentence mentions this). No timed re-ask in v1 — timers need per-user local state richer than a marker and TC14's digest marker can revisit.
- Gitignore entry (peer-review corrected — hand-editing this repo's `.cognition/.gitignore` is NOT enough): that file is generated per-project by `git_hygiene.py::_write_gitignore` (git_hygiene.py:171-232) behind the version-stamped one-shot pass. Add the `onboard-declined` entry to the managed-gitignore writer AND bump `GIT_HYGIENE_VERSION` (currently 2 → 3) so every downstream project's already-run hygiene pass re-fires and picks it up — otherwise other projects would accidentally track the file. Follow the existing versioned-writer precedent (git_hygiene.py:37,47,183-184,197-198,219-220).

**Config.** New `prime_onboard: bool = True` in Settings + PrimeConfig (mirrored per the PrimeConfig↔Settings default-consistency pin, test_prime.py:82). Env-overridable like every knob (config is env-var-only by decision — no new file). This is the project-level kill switch (e.g. a solo project that will never want a roster); the decline file is the per-human path.

**What does NOT change.**
- No MCP tool or schema changes (TC5's tools are the contract; if implementation finds them insufficient, STOP and report — that voids a TC5 acceptance line and I need to know). NOTE: `git_hygiene.py` IS in scope (gitignore writer + version bump above) — the only non-prime code change in this WP.

TC5 deferred instructions.py to "TC7's territory," but the onboarding flow is prime-injected and one-shot per user — standing-practices prose (re-injected every compaction, ~433 tokens, token-costed on every session) is the wrong vehicle for it. If the implementation disagrees, propose at gate with rationale; default is untouched.
- Multi-user personalization logic (`_should_personalize`, section assembly) untouched.
- The registration conversation itself is agent-driven — no interactive code in prime.

**Test evolution (peer-review corrected — RETAIN, don't replace):** `test_person.py::test_solo_prime_byte_identical_with_and_without_person_nodes` (test_person.py:642) passes NO current_email, so under detection rule 1 (empty email → no notice) its assertions still hold verbatim after TC7 — keep it as-is as the no-email pin; only update its now-stale docstring ("onboarding surfacing is WP-TC7, out of scope here" → TC7 has shipped, no-email sessions never see the notice). ## REJECTED ALTERNATIVES (recorded)

- Fire only for multi-user graphs (solo carve-out): rejected — ruling says new USER; solo users are exactly who seeds the roster before a team forms; decline path covers the uninterested.
- Decline recorded in the graph (placeholder person node / decline node): rejected — a decline is not team knowledge, and it would sync one person's local preference to everyone.
- New MCP tool for decline: rejected — widens Gate B surface for a file append the agent can do directly.
- SERVER_INSTRUCTIONS onboarding prose: rejected for v1 — per-session token cost for a one-shot flow; prime notice is targeted and disappears once resolved.
- Timed snooze/re-ask: deferred — needs richer per-user local state; revisit with TC14's marker mechanism.

## KNOWN LIMITS (document, don't fix)

Decline is per-machine (a human with two machines declines twice — or registers once and both go quiet). The notice trusts the agent to relay the conversation faithfully and set from_agent=false (client-declared, unverifiable — same trust class as TC6). A human who ignores the notice sees it every session until they register or decline — intended nag, by ruling.
