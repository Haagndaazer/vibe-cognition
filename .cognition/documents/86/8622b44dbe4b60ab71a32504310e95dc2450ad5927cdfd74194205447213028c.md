## PROBLEM

(1) No way to exclude specific people's nodes from search when a human explicitly wants that (e.g. "search without my own notes cluttering it"). (2) Search results silently truncate at `limit` — the LLM can't tell "returned 10, that's all there is" from "returned 10 of 60"; M4 ruled that disclosure in.

## DESIGN — filter

**Param.** `cognition_search` gains `exclude_people: str | None = None` — comma-separated emails. DEFAULT NONE and **docstring marked USER-INVOKED ONLY**: "Include this parameter ONLY when the human explicitly asked to filter people out this session; never volunteer it" (the ruled guard against accidental LLM use). Casefolded at parse (single-normalization convention).

**Matching.** A hit is excluded iff its node's SERVER-RESOLVED identity email matches: `metadata.recorded_by.email` (non-task) or `metadata.created_by.email` (task) — the established identity key (doctrine + pattern: src/vibe_cognition/cognition/prime.py:137-159, `_node_email`). Free-text `author` is NEVER matched. A node with NO stamp (pre-P13n) cannot be attributed and is NEVER excluded — unverifiable ≠ excludable.

**Where (peer-review corrected).** Inside `_format_search_results` — but note the N1 ghost filter only does a MEMBERSHIP check (`search_hit_is_live` → `has_node`, storage.py:144-153); a full node fetch exists today only for document hits. Implementing this requires ADDING an unconditional `storage.get_node(node_id)` per hit — cheap (O(1) dict read under _synced(), same cost class as the existing document path), but it's new code, not a reuse. Exclusion happens BEFORE the limit fill so the adaptive over-query loop keeps widening to fill `limit` with non-excluded hits — filtering must not starve the result count when other matches exist. **excluded_count: NEVER accumulate across widening rounds** — adaptive_vector_search re-runs the dedupe callback FROM SCRATCH each round and discards earlier rounds' output (embeddings/storage.py:403-422); take only the terminating round's count, exactly as `formatted` itself is replaced (a stateful running counter across calls is the double-count bug).

**Disclosure.** When the filter excluded anything: result gains `excluded_count` (distinct nodes) + `excluded_for` (the casefolded email list as applied). Silent-exclusion is forbidden — the human who asked for the filter still sees its size. When `exclude_people` is not passed, neither key appears.

**Scope.** cognition_search only (single + multi-project paths — filter applies uniformly per project entry). NOT list_tasks/get_history/prime (prime personalization is TC16; retrieval weighting is TC9).

## DESIGN — "returned N of M" (M4)

**Semantic search (`cognition_search`).** Result gains `total_found: int` — the number of DISTINCT LIVE nodes the adaptive over-query discovered before stopping — plus `exhaustive: bool`: true when the loop stopped because Chroma ran out of hits (M is exact for the index), false when it stopped at limit/cap (M is a floor: "at least M matched"). Docstring explains both semantics in one sentence each — honesty about what M means for vector search is the point of the feature, do not oversell exactness.
**Implementation reality (peer-review):** this CHANGES adaptive_vector_search's dedupe contract — `_format_search_results` today breaks early at limit (cognition_tools.py:593-594) and never counts the full distinct set; remove that early break (bounded — the round's raw hits are capped at _SEARCH_OVERQUERY_CAP=500, already fetched) and return list + count. `exhaustive` derives from the terminating round's `len(results) < n`, already computed inline. The contract has THREE call sites: the MCP path AND **dashboard/api.py's own _dedupe closure (api.py:238-265) — which gets a MECHANICAL CONFORMANCE update to the new return shape (plus its tests, e.g. test_dashboard.py:372,382), explicitly in scope as plumbing, NOT a dashboard feature change**; the dashboard's JSON output does not expose the new fields.
**Multi-project aggregation (project="*", cognition_tools.py:2850-2909):** `total_found` = SUM across entries (no cross-project id collisions, per the existing no-dedup comment at :2900); `exhaustive` = AND-reduction (one non-exhaustive project makes the aggregate a floor); note in the docstring that post-merge truncation to the outer limit can make `count < total_found` independent of exclusion/exhaustion — one line so N-of-M doesn't read self-contradictory.

**Deterministic lists.** `cognition_get_history`: the `context_term` branch is genuinely list-then-limit (queries.py slices at the call site, cognition_tools.py:3141) — total_found free; the NO-context_term branch calls storage.get_recent_nodes, which slices INTERNALLY and discards the pre-slice length (storage.py:451-473) — needs an ADDITIVE, backward-compatible change (opt-in flag or second return value) that leaves its other callers (prime.py:353,363; dashboard/api.py:449) untouched. `exhaustive` always true here, included for shape consistency. **`cognition_get_edgeless_nodes`: OUT OF SCOPE — already compliant** via its existing `total_edgeless` field (cognition_tools.py:3347,3370,3386); do NOT add a duplicate key.

**No prose changes to counts.** Existing `count` (returned N) keeps meaning returned-row count everywhere; nothing renames.

## REJECTED ALTERNATIVES (recorded)

Include-only filter (author allowlist): rejected — the backlog ask was exclusion ("filter OUT nodes from certain people"); allowlisting is a different feature with weighting interactions (TC9 territory). Matching free-text author: rejected (prime.py:118 doctrine — free text is not identity). Filtering in Chroma metadata (where clause): rejected — recorded_by email is deliberately NOT in vector metadata; adding it duplicates identity into a second store and drifts. Exact M for vector search via full-graph scan per query: rejected — O(all nodes) per search to sharpen a disclosure number; the floor+exhaustive-flag contract is honest and cheap.

## KNOWN LIMITS (document)

exclude_people trusts the caller's restraint (the docstring guard is behavioral, unverifiable server-side — same trust class as from_agent). total_found is a floor unless exhaustive. Excluding a prolific author on a small graph can legitimately return fewer than limit results when nothing else matches — disclosed via excluded_count, not padded.
