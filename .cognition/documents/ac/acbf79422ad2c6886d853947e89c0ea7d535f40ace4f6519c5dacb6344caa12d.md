## PROBLEM

Retrieval ranks purely by vector similarity. The graph now knows who recorded a node (recorded_by), whether a human drove it (from_agent), and how senior that human is (person nodes, SENIORITY_LEVELS owner>senior>mid>junior) — but a drive-by agent note and the owner's considered decision rank identically. Colton ruled hierarchy into retrieval: seniority context is part of the cognitive story (junior finds a better solution, senior later revises — both must surface, weighted and labeled).

## DESIGN

**Core principle — penalties, never boosts.** All weighting is a multiplicative penalty from a 1.0 baseline: `weighted_score = score × multiplier`, `multiplier ∈ (0, 1.0]`. Nothing is ever boosted above its raw similarity. Consequences that make the ruled invariants STRUCTURAL rather than special-cased:
- A constraint/incident (pinned multiplier 1.0, exempt) can only ever RISE relative to down-weighted neighbors — "never outranked by seniority" holds by construction.
- An unverifiable node (no stamp / no person node / no bool) gets neutral 1.0 — it is never penalized for missing provenance ("unverifiable ≠ junior"), mirroring every prior doctrine (TC10 unstamped-never-excluded, dashboard unverified chip).
- Nothing is filtered: weighting reorders within the discovered set; total_found/exhaustive/excluded_count semantics untouched (never-wiped, ruled).

**Weighting axes (invariants pinned; exact numbers are Vorpid's, checked at gate against the invariants):**
1. *Seniority* (from the person node matching the hit's server-resolved identity email — `_node_email`, casefolded): monotone non-increasing multipliers down SENIORITY_LEVELS, owner = 1.0 floor bounded (suggest owner/senior 1.0, mid ~0.95, junior ~0.9 — spread deliberately small; this is a tiebreaker-plus, not a gag).
2. *Agent origin* (from_agent true): a penalty STRICTLY STRONGER than the entire seniority spread, so ANY human-attributed node outweighs ANY agent-attributed node at equal raw score (ruled: "human input always outweighs agent input"). E.g. with junior 0.9, agent must be < 0.9 (suggest ~0.85). Axes compose multiplicatively (agent-recorded-by-senior = 1.0 × agent penalty).
3. *Exempt types*: constraint + incident pinned 1.0 regardless of both axes (reuse `_NEVER_EXCLUDED_SEARCH_TYPES` — same doctrine set, single source).
Multipliers live as a module-level table (named constant, docstringed) — no config knob, no env var (weighting is a product behavior, not a preference; matches the no-persistent-muting precedent).

**Visibility (ruled: never silent).** `score` stays the RAW similarity — never mutated. Each hit gains:
```
"weight": {
  "multiplier": 0.9,                       # 1.0 = neutral
  "seniority": "junior" | null,            # from the person node; null = no registered person
  "from_agent": true | false | null,       # echo of the provenance bool; null = pre-TC6 node
  "basis": "human:junior" | "human:unregistered" | "agent" | "exempt:constraint" | "unverified"
}
"weighted_score": 0.81                     # score × multiplier — what the ordering used
```
**Basis vocabulary (peer-review corrected — the dashboard mirror is EXACT, not approximate).** The dashboard's "unverified" treatment (api.py:372-374, app.js:81-99) means precisely NO server-resolved identity at all (pre-P13n, no recorded_by/created_by); a resolved identity with no person node renders as a VERIFIED chip there. TC9 mirrors that boundary:
- `"human:<seniority>"` — stamped identity + registered person node (multiplier from the seniority table).
- `"human:unregistered"` — stamped identity, NO person node: a KNOWN author who isn't in the roster yet. Neutral 1.0, seniority null. NOT "unverified" — conflating "we don't know who" with "we know who, they're unrostered" is the improvisation the battle plan forbids.
- `"agent"` — from_agent true (agent penalty; seniority axis still reported when resolvable).
- `"exempt:constraint"` / `"exempt:incident"` — pinned 1.0 regardless of other axes.
- `"unverified"` — NO stamp at all (the dashboard's exact case). Neutral 1.0, both axes null.
Missing from_agent key (pre-TC6): `from_agent: null`, no agent penalty; basis falls to whichever of human:*/unverified applies from the stamp axis. The `weight` key is ALWAYS present on every hit (silence is exactly what the ruling forbids; a 1.0 multiplier with its basis is information — consistent with the total_found always-present precedent).

**Where it runs.** Inside the search result path so weighting decides WHICH hits surface, not just their order within an already-cut top-N: the dedupe callback (`_format_search_results`) already fetches the full node per hit (TC10's get_node) — it additionally computes multiplier + fields, then sorts its (uncapped, per TC10) return list by weighted_score desc so `adaptive_vector_search`'s `[:limit]` slice takes the weighted top (verified: no downstream re-sort on the single-project path). **Multi-project fan-out: the outer merge sort at cognition_tools.py:3254 (`all_results.sort(key=...score...)`) MUST switch to weighted_score** — peer-review flagged this as THE safety-critical edit: left unchanged, it silently re-sorts everything back to raw score and no-ops weighting for project="*" while single-project works, the exact silent-divergence failure mode. Raw score kept per hit; per-entry weighting uses each entry's OWN person registry (a foreign project's seniority is its own graph's business). Dashboard `_dedupe` closure: NO weighting, and verified ZERO change needed (it's a fully separate implementation sharing only the tuple contract — weighting in _format_search_results cannot leak there); assert at gate via unmodified dashboard tests.

**Person resolution — cost model + memo location (peer-review corrected).** `_find_person_by_email` (cognition_tools.py:1888-1898) is a LINEAR SCAN over all person nodes — calling it per distinct email is O(D × P_total), the accidentally-quadratic shape C2 would flunk. Instead: **once per top-level search call**, `get_nodes_by_type(PERSON)` → build an email→seniority dict → **thread it into the dedupe lambda from `_search_with_embedding`'s enclosing scope** (the lambda at :719). NOT a local inside `_format_search_results` — that resets every widening round and silently degrades per-call to per-round. Cross-round persistence here is CORRECT and does not repeat the TC10 excluded_count lesson: this is a read-only CACHE of data immutable for the call's duration (person nodes don't change mid-search), whereas excluded_count was per-round OUTPUT that accumulating double-counted; cache the inputs, never the outputs. Multi-project: one map per entry (per-entry registries), built at fan-out time. No persistent/session cache — no invalidation surface.

**Scope boundary.** cognition_search ONLY (single + multi-project). NOT get_history (structural/recency semantics — weighting a deterministic list distorts its contract), NOT list_tasks (backlog ordering is priority+recency by design), NOT prime (TC16's territory), NOT dashboard JSON (V3's territory). **get_workflow's limit=1 internal search (cognition_tools.py:3404-3407) inherits weighting via the shared path — this is a DELIBERATE, material behavior change, not a footnote** (peer-review): on a multi-topic graph, weighting can change WHICH workflow procedure is matched (e.g. an agent-authored near-similar workflow no longer beats a senior-authored one) — that changes what steps a caller executes. Accepted as consistent with the ruling's intent, disclosed UNCONDITIONALLY in the get_workflow docstring (a docstring is static; no graph-shape-conditional wording), and given its own acceptance fixture. Note also `matched` can change within a single supersession chain even when `head` doesn't.

**Performance (Gate C2 hook).** Added cost per hit: one casefolded-email person lookup + one multiplication. Person resolution builds a per-CALL email→seniority map (memoize per search call, NOT a persistent cache — no invalidation surface) so N hits cost ≤ N dict lookups + P person-node reads (P = distinct emails in results). C2 measures search latency with weighting on the 1k-node harness; the brief pins the budget: weighting must not add more than a few ms at harness scale (C2 gates the number; TC9 just must not do anything O(n²) or I/O-per-hit).

## ACCEPTANCE

- Ordering: junior-authored hit with higher raw score STILL SURFACES (never wiped) but sorts below a senior hit whose weighted_score exceeds it; fixture proves reordering across the limit boundary — an UNPENALIZED hit enters the top-N when a higher-raw-score neighbor is penalized below it (relative rise only; no hit's weighted_score ever exceeds its raw score — do NOT add a boost path).
- Invariants (fixture-tested, not number-tested): multipliers ≤ 1.0 everywhere; constraint/incident hits carry multiplier 1.0 + basis exempt regardless of author/agent axes and are never displaced downward by any weighted neighbor (ruled; provably holds even at weighted-score ties — a tie requires the neighbor's raw score to already meet the exempt hit's, so a tie is never weighting-caused; stable sort keeps input order on ties, one line in the fixture notes this); every agent-attributed multiplier < every human-attributed multiplier (strict dominance, tested programmatically against the shipped table).
- Fallbacks (battle-plan mandate, basis per fixture EXPLICIT): (a) pre-TC6 node (stamped, no from_agent key, registered person) → from_agent null, no agent penalty, basis "human:<seniority>"; (b) stamped email, no person node → seniority null, neutral, basis "human:unregistered"; (c) fully unstamped → both axes null, multiplier 1.0, basis "unverified"; (d) COMPOUND (peer-review): stamped + no from_agent key + no person node (the realistic post-P13n/pre-TC6 unrostered case) → neutral, basis "human:unregistered".
- Visibility: weight key present on EVERY hit incl. neutral ones; score AND weighted_score both present as separate fields (equality EXPECTED whenever multiplier is 1.0 — do not assert inequality); docstring explains basis vocabulary and that ordering uses weighted_score.
- get_workflow (peer-review): multi-topic fixture — agent-authored workflow vs senior-authored near-similar; weighting changes which is matched, asserted deliberately; docstring notes the weighting unconditionally.
- TC10 composition: exclude_people + weighting compose (excluded set unaffected by weights); total_found/exhaustive unchanged by weighting (weighting never changes the discovered count); excluded_count fixture passes unchanged.
- Multi-project: outer merge at :3254 ordered by weighted_score (fixture: cross-project raw-vs-weighted inversion proves the re-sort switched — the silent-no-op failure mode); per-entry person resolution uses that entry's graph (fixture: same email, different seniority in two projects → each entry's hits weighted by its own registry).
- Dashboard: zero behavior change (existing dashboard search tests unmodified and passing).
- Performance shape: ONE `get_nodes_by_type(PERSON)` scan per top-level search call, asserted with a fixture that FORCES a widening round (flood past round 1) — the counter must still read 1 across rounds (catches the memo-in-wrong-scope degradation); no persistent cache.
- Docs: cognition_search docstring (weight/weighted_score/basis vocabulary + never-wiped framing); README search section; SKILL.md search bullet; readme.py search line; CHANGELOG. Tool-surface self-sufficiency audit (67751ebc39bd) over cognition_search (return-shape change → Gate B re-run per battle plan :61).

## REJECTED ALTERNATIVES (record)

Annotation-only (no reordering): explicitly overruled by Colton. Boost-based weighting (seniors > 1.0): breaks the constraints-never-outranked invariant structurally and inflates scores above similarity (a lie in the score field). Filtering low-seniority below a threshold: ruled out ("weighted, never wiped"). Config-knob weights: no preference surface for a product-ruled behavior; also a foot-gun (a team silently zeroing juniors violates the ruling's spirit). Persistent person cache: no invalidation surface — per-call memo only. Weighting get_history/prime: distorts deterministic contracts / TC16's lane. Storing multiplier into Chroma metadata: weights derive from CURRENT person data at query time; baking them into the index goes stale on every seniority edit.

## KNOWN LIMITS (document)

Seniority is trust-declared on person nodes (peer-editable; audit trail is the control) — weighting inherits that trust model. The seniority spread is deliberately small: it reorders near-ties, it does not bury strong matches (Gate D scenario 4 is the acceptance story: the junior's better find still surfaces on top when its raw score dominates). Foreign-project hits weight by that project's registry — an unregistered-there teammate is neutral there. Agent-origin dominance is per the ruled axis, even though an agent note may transcribe a human's words — the stamp records the channel, not the wisdom.
