## DESIGN

**Nav + views.** Single-page, hash-routed or state-toggled views (implementation's choice; mockup uses state-toggle). Overview per doc §4.1: stat tiles (open / in progress / blocked / done-this-week / documents / workflows), Active Constraints strip (severity≠low, HEAD-filtered), Needs Attention list (stale in-progress claims > 5 days; blocked tasks), Recent Episodes (5), Recent Incidents (high+, 14d). **HEAD-filter implementation (peer-review): do NOT import prime.py** (private markdown functions + CLI deps) — the reusable unit is the one-line storage predicate prime already uses: `not storage.get_predecessors(node_id, CognitionEdgeType.SUPERSEDES)`. Call that directly in the dashboard handler. Board per doc §4.2: 4 columns (open / in_progress / blocked / done capped to last 7 days; cancelled behind a toggle); cards show summary, priority chip, parent breadcrumb, creator chip, claimant chip + claim age; tree-view toggle renders the parent hierarchy (depth-indented) as the alternative to columns.

**Detail drawer** per doc §4.7: full node fields + metadata rendered — provenance block (recorded_by / created_by / claimed_by / author with trust-class labeling), task transition timeline, references, related nodes grouped by edge type (supersedes/contradicts visually loud). **Trust-class chip rule (spec, not optional):** person chips backed by server-resolved identity render normally; a node with no recorded_by falls back to free-text author and MUST be visually distinct (dashed outline + "unverified"). **from_agent badge:** if `metadata.from_agent` is present render a small "via agent" badge (TC6 ships before this WP — the data will exist); absent key renders nothing (never coerce). **Conflict banner SLOT:** ship the banner component rendering from a `conflicts` field in the node payload if present; server population of that field is Wave-2 TC2, so in V1 it simply never fires — the slot exists, zero fabricated data.

**Graph tab.** Existing Cytoscape canvas moves behind a "Graph" nav item; Cytoscape init deferred until first tab activation (lazy — do not pay 600KB parse + layout on dashboard open). Functionality otherwise unchanged. **Peer-review precision — lazy means the BUILD, and the POLL must be decoupled:** today init() unconditionally calls loadGraph()→buildCy() AND the 30s setInterval rebuilds a brand-new Cytoscape instance every cycle. Required shape: stats/overview polling always runs (header pills, view data); the graph fetch is a cheap JSON call other views may still use for feeds; `buildCy()`/layout runs ONLY while the Graph tab is active (first activation initializes; subsequent polls rebuild only if the tab is active or on next activation). "Lazy graph fetch" and "lazy Cytoscape build" are different things — the acceptance criterion is about the BUILD.

**API.** New `GET /api/tasks`: token-gated like siblings; returns `{tasks: [{id, summary, status, priority, owner, parent_id, depth, created_by, claimed_by, timestamp, claimed_at, last_transition_at, transitions_count}], count}`. **Peer-review corrections baked in:** (a) `claimed_by` and the transition fields are NOT in `_list_tasks`'s existing row shape — implement the dashboard handler independently against `storage.get_nodes_by_type(TASK)` (own filter/sort/depth), do NOT extend the MCP tool's return contract (this WP touches no MCP surface); (b) claim age and the done-column 7-day cap are computable ONLY from transition timestamps — `claimed_at` = at of the latest →in_progress transition, `last_transition_at` = metadata.transitions[-1].at; the node's top-level timestamp is creation-only and never updated. New `GET /api/overview` (REQUIRED, not optional): server-computed aggregate for the Overview view — task counts by status, done-this-week count, active constraints (severity≠low, HEAD-filtered via the storage predicate above, with severity + recorded_by), needs-attention list (stale claims >5d from claimed_at; blocked), recent episodes (5), recent incidents (high+, 14d) — one call, no N-fetch client assembly, no client-side reimplementation of graph semantics. `/api/node/{id}` already passes metadata through (verified) — NO backend change; the drawer renders what's already there. `/api/graph`, search, stats, documents, delete, download: unchanged.

**Threadpool risk (task 4163f54f2848) — resolve or accept HERE.** The dashboard runs run_in_threadpool on its own uvicorn loop; V1 adds one endpoint + the same 30s polling. Assess the actual spawn risk; either fix it in this WP (if small) or report a written accept-rationale at gate. Not silent.

**Token hygiene (shipped, don't regress):** no URL+token INFO logging; new endpoint uses the same header-token middleware.
