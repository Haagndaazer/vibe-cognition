# WP-WhatsNew-1: Post-update "what's new" notice in the session-start prime

**Status:** APPROVED (Colton, 2026-07-23) — peer-reviewed (Sonnet, needs-rework verdict, both blockers + all should-fixes folded in). Colton additionally directed: the release-procedure workflow node MUST gain the whats-new step (Vince updates by supersession at merge time, before the 0.30.0 release).
**Implementer:** Vorpid
**Manager:** Vince
**Train:** upgrade-UX (complements WP-Nudge-1: the nudge says "update"; this says "here's what you got")

## Problem

After a plugin update, nothing tells the user what's newly available — features ship silently and go unused (Colton: "goal is to ensure users know what is available to use now", modeled on vibe-memory's since-your-last-version alerts). Constraint from Colton: **no spam** — track that the user has seen the news, then silence.

## Approach (one direction, chosen)

### Content source: a curated `whats-new.json`, NOT the CHANGELOG

New file `.claude-plugin/whats-new.json` shipping with the plugin:

```json
{ "0.29.0": ["Update nudge: you now get a one-line notice at session start when a newer plugin version is released (disable: VIBE_UPDATE_NUDGE=off)."],
  "0.30.0": ["What's-new notices: after an update, your first session lists the new features once (this message!). Disable: VIBE_WHATS_NEW=off."] }
```

- 1–3 short, user-facing, **pure-ASCII** bullets per version, maintainer-written at release time.
- Why not parse CHANGELOG.md: entries are long dev-oriented prose (the 0.29.0 entry alone is ~150 words); injecting or auto-summarizing them is token bloat and fragile parsing. A curated file is bounded, deliberate, and cheap.
- A version with no entry is silently skipped — the maintainer opted out of announcing it. Unknown/malformed file → silence, never a crash.
- Seed the file with the 0.29.0 and 0.30.0 entries above (so the first rollout announces both the nudge and itself to anyone jumping 0.28 → 0.30).
- **Release-procedure change:** writing the new version's whats-new bullets becomes a release step. (Vince updates the "vibe-cognition plugin release procedure" workflow node by supersession post-merge — manager duty, not Vorpid's.)

### Seen-tracking: per-MACHINE marker in the plugin-data dir

- Marker `${CLAUDE_PLUGIN_DATA}/whats-new-seen` — **plain text**, containing exactly the version string last shown (plain text so the bash gate can read it without jq/python).
- **Per-machine, not per-project** — deliberately NOT `.cognition/` (the TC7 decline file's location): a user with five projects would otherwise see the same notice five times. First session in ANY project after an update shows it once; every other project stays quiet. (CLAUDE_PLUGIN_DATA survives updates — same reason update-check.json lives there.)
- Semantics: "seen" = "shown once" (matches Colton's spec: show, mark seen, silence — we can't verify a human read it, same as vibe-memory).
- **Fresh-install vs existing-user disambiguation (peer-review BLOCKER — the naive "no marker = fresh = silence" rule would have shown nothing to ANYONE on rollout day, since the marker is new and nobody has one):** when the marker is absent, check whether `${CLAUDE_PLUGIN_DATA}/update-check.json` exists (written every session since 0.29.0). Present → this machine ran the plugin before → treat seen as "0.0.0" and show the (capped) backlog once. Absent → truly fresh machine → write marker = installed, show NOTHING (onboarding owns new users). **This signal is only valid because the whats-new step runs BEFORE the update-check step in the hook** (see Mechanism) — otherwise a fresh install's first session would write update-check.json first and then misread it as evidence of prior use. Documented limitation (accepted): a user who set VIBE_UPDATE_NUDGE=off since 0.29.0 and never accrued a stamp is misread as fresh ONCE, misses that one backlog, and self-heals (marker exists thereafter).
- Lifecycle: marker == installed → silence, zero cost. marker < installed → show bullets for every version in (marker, installed] that has an entry, then write marker = installed. marker > installed (downgrade/rollback) → rewrite marker = installed, silence. Marker write is atomic (temp + os.replace), never raises, and is pinned to `write_text(version, encoding="utf-8", newline="")` — bare version string, NO trailing newline ever (`write_text` on Windows otherwise CRLF-converts and the bash string-compare breaks: the same invisible-byte class as the cp1252 bug). Unit test asserts the marker's raw bytes equal exactly the version string.

### Mechanism: third rail on the existing note pipeline

- New module `src/vibe_cognition/whats_new.py` — stdlib-only, mirrors `migrate_mcp`/`update_check` exactly: reads `CLAUDE_PLUGIN_ROOT` (whats-new.json + plugin.json version) and `CLAUDE_PLUGIN_DATA` (marker), prints a block (or nothing) to stdout, always exits 0, top-level try/except Exception → 0. **No network — this feature is fully local.**
- `hooks/session-start.sh`: new step **BEFORE the update-check step** (peer-review: two reasons, both load-bearing — (1) whats-new is fast and local and must not queue behind up to 8s of network I/O on exactly the post-update session where both fire; (2) the fresh-install signal above depends on reading update-check.json's existence BEFORE this session's update-check step can create it. Prime's section ordering is set by prime.py, not by bash invocation order, so AC 12 is unaffected). Bash-side gates, in order: kill switch `VIBE_WHATS_NEW` (tr-lowercased, off|0|false|no) → cheap skip: extract installed version from plugin.json via grep/sed and compare to the marker file's contents; equal → skip the spawn entirely.
- **The extraction line MUST carry the `|| VAR=""` guard (peer-review BLOCKER, empirically reproduced): under `set -euo pipefail`, an unguarded `VAR=$(grep ... | sed ...)` with no match aborts the ENTIRE hook before prime runs — one plugin.json typo would kill migration note, nudge, and all project context, emitting bare `{}`.** Same idiom as the existing MIGRATE_NOTE/UPDATE_NOTE captures. Fallback: extraction empty → spawn the module anyway (module is authoritative; bash is only an optimizer). All new lines `set -e`-safe (explicit if/fi), stdout reserved for the final JSON, `_bc` breadcrumbs `whats_new_start/done`.
- Note forwarded as `VIBE_WHATSNEW_NOTE` env into the prime invocation; `prime.py main()` inserts it **immediately after `update_note`, before `hygiene_line`** (peer review: pin the exact insertion point — prime's real section order is migration → update → hygiene → rehydrate → onboarding-or-digest, and "after the nudge" alone left the hygiene/rehydrate placement to guesswork). `generate_prime()` stays pure — the marker is written only by the module (hook path), same doctrine as last-seen.json (WP-TC14).

### The block (format, an acceptance criterion)

```
INSTRUCTION: Tell the user what's new in this plugin update before continuing.
vibe-cognition updated (v{seen} -> v{installed}) - new since your last session:
- {bullet}
- {bullet}
(Shown once per update. Full details: the plugin's CHANGELOG.md. Disable these notices with VIBE_WHATS_NEW=off.)
```

- The `INSTRUCTION:` prefix line matches ONBOARDING_BLOCK's house style (peer review: the draft folded the directive into prose, weaker for LLM parsing). The block is read by the LLM; the goal is the USER hears it. (`{seen}` renders as `first install` wording when the backlog floor "0.0.0" was used.)
- Pure ASCII end to end (cp1252 pipe lesson, discovery 011831dd7724): plain hyphens, no em-dashes. Unit tests assert `.isascii()` on BOTH every bullet in the shipped whats-new.json AND the module's own static template strings (header/footer literals are exactly where a future em-dash sneaks in). Plus `sys.stdout.reconfigure(encoding="utf-8", errors="replace")` belt-and-braces in main() (getattr form, pyright-clean, as in update_check). Vince's gate additionally includes a real-Windows-pipe smoke run — per discovery 011831dd7724, in-process tests can never catch this class.
- **Caps (anti-spam + prime token budget, cf. the v0.13.0 prime-trim decision):** max 3 versions listed (newest first), max **9** bullets total (3 versions x 3 bullets each — the previous 8 was self-contradictory, peer review). Version-overflow (span > 3 versions with entries): append `(...and older versions - see CHANGELOG.md.)`. Bullet-only overflow (<= 3 versions but > 9 bullets — only possible if the per-version authoring rule was violated): drop trailing oldest bullets silently, NO "older versions" line (nothing older was dropped — the line would lie). Marker still advances to installed in every case — no re-showing later.

### User control

- Config field `vibe_whats_new: bool = True` in config.py — `vibe_` prefix load-bearing (pydantic-settings derives `VIBE_WHATS_NEW`), plain bool, NOT the prime tri-state. Documentation/config-surface for the same env var bash + module read directly (same pattern as `vibe_update_nudge`, same honest-test rule: assert the env derivation end-to-end, no fake config-file channel).
- Kill switch honored bash-side (no spawn) AND module-side (defense in depth). When off: no read, no marker write, no note.

## Out of scope

- Parsing/summarizing CHANGELOG.md automatically.
- Per-project or per-user (email-keyed) seen-tracking — per-machine is the spam-safe unit.
- Re-showing on demand (a `/whats-new` skill could come later; not this WP).
- Any network I/O.
- Marketing copy beyond the maintainer bullets; localization.

## Known-intentional (do NOT "fix")

- `generate_prime()` purity: marker writes live in the module invoked by the hook, never in prime. (WP-TC14 doctrine.)
- Hook stdout reserved for final JSON; `set -euo pipefail` live; `uv run --no-sync` deliberate.
- TC7's decline file lives in `.cognition/` — correct for ITS per-project purpose; do not "align" this marker to it (deliberately per-machine, see above).
- update-check stamp is JSON while this marker is plain text — deliberate divergence: the whats-new bash gate must string-compare the version without spawning anything; the update-check gate only needs mtime.
- CHANGELOG stays the canonical detailed record; whats-new.json is a curated announcement layer, drift between their granularity is expected and fine.

## Acceptance criteria

1. Upgrade path (marker 0.28.0 present, installed 0.30.0, entries for 0.29.0+0.30.0): prime context contains exactly the block above with both versions' bullets (newest first); marker becomes 0.30.0; the NEXT session start neither spawns the module (fake-uv log assertion) nor shows the block.
2. **Rollout-day path (THE launch scenario): no marker, but update-check.json present, installed 0.30.0** → block shown with the capped backlog (floor "0.0.0", `first install`-style wording for {seen}); marker written = installed. This AC must be satisfied WITHOUT artificially pre-seeding a whats-new marker — the state under test is exactly what every existing user's machine will be in.
3. Truly fresh install (no marker AND no update-check.json): no block; marker written = installed.
4. marker == installed: no spawn (bash gate; fake-uv log), no block.
5. Downgrade (marker 0.30.0, installed 0.29.0): no block; marker rewritten to 0.29.0.
6. Version with no whats-new entry inside the span: skipped silently; span of ONLY entry-less versions → no block, marker still advances.
7. Caps: (a) span of >3 versions with entries → newest 3 + the `(...and older versions...)` line; (b) ≤3 versions but >9 bullets → trailing oldest bullets dropped silently, NO overflow line; (c) ≤9 bullets total in every case.
8. `VIBE_WHATS_NEW=off` (any case) and `vibe_whats_new=false`: no spawn, no block, no marker write. Env-derivation test for the config field (env var only — no fake config-file channel).
9. whats-new.json missing, malformed JSON, non-dict, non-list values, non-string bullets: silence, exit 0, marker still advances (treat as "nothing to announce").
10. Bash version-extraction fallback: plugin.json unreadable/unmatched by the bash grep → hook still exits 0, **prime still runs and emits real output (not bare `{}`)**, AND the module is still spawned (fake-uv log) — the wide assertion is the point: a missing `|| VAR=""` guard fails THIS test by killing the hook, not just by skipping the module.
11. ASCII: every bullet in the shipped whats-new.json AND every static template string in whats_new.py `.isascii()`; bullets each ≤ 200 chars (unit tests — enforced at authoring time forever).
12. Prime section sequence asserted IN FULL: migration note → update nudge → whats-new block → hygiene → (rest unchanged); each note its own section.
13. Marker file raw bytes == exactly the version string (no trailing newline/whitespace; `newline=""` pinned). Concurrent session starts racing the marker: no crash, no torn file (temp+rename; last-writer-wins documented).
14. Docs: README section ("What's-new notices" — what/when/once-per-update/how to disable), CHANGELOG entry, version bump both files (0.30.0). Release-procedure doc note that bullets are a release step (the workflow-node update itself is Vince's, post-merge).
15. Tests Windows-safe, no live network anywhere in the new tests.

## Seams checked

- **Nudge co-occurrence:** updated to 0.30.0 while pin already at 0.31.0 → nudge AND whats-new both appear (correct: "here's what you got, and there's more available"). Ordering fixed by AC 11; no shared state (different marker files).
- **Stale-first-session race (43d3c3dab10f):** module reads plugin.json + whats-new.json from CLAUDE_PLUGIN_ROOT on disk (the NEW version's files — hooks always run from the updated root), never from the possibly-stale MCP server. Unaffected.
- **Process leak (d2a62f38cc88):** no interaction; module prints and exits.
- **cp1252 pipe (011831dd7724):** ASCII-enforced at authoring time + reconfigure belt-and-braces (AC 10).
- **Prime token budget (v0.13.0 trim decision):** hard caps (3 versions / 8 bullets / 200 chars each) bound the worst case to roughly a constraint-section's worth of tokens, shown once per machine per update.
- **Journal/graph:** untouched — no cognition state anywhere in this WP.
