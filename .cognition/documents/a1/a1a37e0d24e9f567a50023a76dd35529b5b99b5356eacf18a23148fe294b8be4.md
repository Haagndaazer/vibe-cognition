# Scope: WP-TC4 — Claim-Collision + Reopen Warnings on cognition_update_task (rev 3, APPROVED)

Status: DRAFT rev 3 (rev 1 REJECT: guard interaction + test enumeration; rev 2 REJECT: bare-poke hits the no-updatable-fields error :1673-1679, and 2b had no specified mutation path — both fixed below with a concrete code-path spec; pending focused re-review). Vorpid implements; Vince gates. Epic cedf4a8457e9; task f75cf9fa2b72. Ruling (task detail, Colton): claiming a task already claimed_by someone else returns a warning naming the current claimant and claim age — NEVER blocks; takeover (re-claim over a live claim) requires a transition note; reopening someone else's done/cancelled task gets the same warning shape (promoted from optional per survey synthesis). Composes with TC8 stamped assignment.

## PROBLEM

`claimed_by` is stamped on transition to in_progress and re-claims via blocked/open→in_progress silently RESTAMP it to the new caller (documented cognition_tools.py:1564-1570) — teammate B can grab teammate A's task and A's claim vanishes without a trace beyond the transitions log nobody is told to read. Same-status in_progress calls skip the transition body entirely (the `if status != current` gate at :1605), so they no-op for stamping. No warning exists anywhere. Two agents double-working a task is the top real-world collision in a multi-agent team.

## CURRENT MECHANICS (survey, citations peer-verified)

- `_update_task` at cognition_tools.py:1521-1695. `claimed_by = {name, email}` from `resolve_git_identity` (:1614), set only inside the real-transition body; sticky forever (never cleared).
- Same-status calls: the `if status != current` gate at :1605 skips the transition body — mechanically a stamping no-op (docstring prose at :1566-1567 describes it).
- **Note guard (:1593-1598): `note` with no NEW status is rejected BEFORE the transition block** — "note annotates a status change". This guard interacts with any same-status takeover design and MUST be restructured (see Design 2b).
- Transition audit: `metadata.transitions` append (:1623), entry `{status, at, by, note?}` (:1615-1621). `_add_task` always seeds `transitions` (:1312); read sites default to `[]`.
- Legal transitions `_TASK_TRANSITIONS` (:1220-1226): open→{in_progress,blocked,cancelled,done}; in_progress→{blocked,done,cancelled,open}; blocked→{in_progress,open,cancelled,done}; done→{open}; cancelled→{open}.
- Warning-key precedent: `result["warning"] = "<string>"` on successful `_load_project_core` (:2085); `warnings` list in document storing. Warnings ride on SUCCESS responses.
- **Claim-age precedent ALREADY EXISTS: dashboard/api.py:314-321 `_task_claimed_at(transitions)`** — last-wins scan for the latest in_progress entry, documented as mirroring claimed_by. TC4 does NOT duplicate it (see Design 5).
- TC8: `assigned_to` casefolded-email-or-absent; `assignments` append-only with compare-before-append (:1646-1657). TC4 must not touch either.

## DESIGN

**Definitions (load-bearing).**
- *Caller identity*: the `by` already computed at :1614 (`resolve_git_identity`). No new resolution.
- *Same person*: casefolded email equality. If EITHER side's email is empty → identity unverifiable → ALL machinery below disengages (no warning, no note requirement — "unverifiable ≠ matches", same doctrine as TC10's unstamped-never-excluded). Names never matched.
- *Live claim*: `claimed_by` present AND current status ∈ {in_progress, blocked}. open/done/cancelled = released/finished claim: re-claiming is normal flow, NO warning, NO note (sticky claimed_by is history, not possession).
- *Claim age*: `_task_claimed_at` semantics — the `at` of the LATEST transitions entry with status in_progress (last-wins scan; already proven robust to consecutive same-status entries, which this WP introduces for the first time). Null when no such entry exists (legacy).

**1) Collision warning (never blocks, structured).** Whenever the machinery detects a foreign live claim being displaced or poked (cases 2a/2c below) or a foreign closure being reopened (case 3), the SUCCESS response gains:
```
"claim_warning": {
  "kind": "claim_collision" | "takeover_note_required" | "reopen",
  "claimant": {"name": ..., "email": ...},   # prior claimant (or closer, for reopen)
  "claimed_at": "<ISO or null>",             # claim start (or closed-at, for reopen)
  "assigned_to": "<email>",                  # only when set — TC8 info, zero semantics
  "message": "<one line, human-readable>"
}
```
Key ABSENT when nothing fired (self-actions, released claims, unverifiable identity, fresh claims) — absent-key discipline per TC10. Structured dict (not the bare-string `warning` precedent) because the ruling names data (claimant + age) the caller must relay; `message` keeps the human-readable convention.

**2) Takeover semantics — three shapes, enumerated precisely.**
- **(2a) Transition-based takeover** (open→in_progress or blocked→in_progress) **over a live foreign claim** (i.e., from `blocked` only — an open task's claim is not live by definition): WITHOUT `note` → `{"error": ...}` naming the claimant, claim age, and instructing "pass note= to take over". Validated BEFORE mutation. WITH `note` → proceeds exactly as today (restamp + transitions entry carrying the note) + `claim_warning` (kind claim_collision) in the response. This is the ruling's enforced exception to never-blocks: sentence 2 ("requires a transition note") is meaningless unless enforced somewhere, and this transition is an unambiguous act of claim-taking. One retry with a note always succeeds.
- **(2b) Same-status takeover** (`status="in_progress"` on an already-in_progress task, foreign live claimant, WITH `note`): the single-call takeover — restamps `claimed_by`, appends a transitions entry (status in_progress, at, by, note) recording the seizure, sets `metadata_changed`, returns `claim_warning` (kind claim_collision). Mechanism in §2-MECH below — this is NEW mutation code, not a reroute of the existing transition body.
- **(2c) Same-status foreign poke WITHOUT note** (`status="in_progress"`, already in_progress, foreign live claimant, no note): NO seizure, NO restamp, NO error — the response gains `claim_warning` with `kind: "takeover_note_required"` and a message stating explicitly: "task remains claimed by <claimant>; you did NOT take it over — pass note= to take over." Other fields passed in the same call (owner/priority/etc.) apply normally. **Bare poke included** (status="in_progress" and nothing else): today that shape dies at the no-updatable-fields gate (:1673-1679) — rev-2 BLOCKING #1 — so this shape gets an explicit success path: return the (unmutated) task node + the warning, bypassing that gate, because an attempted claim is not a nothing-call; it deserves an answer, and the answer is the warning. Rationale for no-error overall: this shape is either ambiguous intent (combo update with redundant status) or a natural claim attempt (bare poke, the exact shape of test_task.py:411-415's legitimate first claim) — never-blocks governs both, and the explicit warning prevents the silent-fool outcome (caller believing they claimed what they didn't).
- Same-IDENTITY same-status calls: byte-identical to today in EVERY variant — combo call applies fields with no warning; bare call still errors "No updatable fields" (that error is generic input validation, not a block on claiming, and preserving it keeps the no-op discipline test surface unchanged).
- Unverifiable identity (either email empty): byte-identical to today in every variant, including the bare-poke error.

**2-MECH) Code-path spec (rev-2 BLOCKING #2 — the mechanism, not just the outcome).** All inside `_update_task`, structured as follows:
1. **Resolve identity ONCE, early**: hoist the `resolve_git_identity` call (today at :1614, inside the transition body) to just before the note guard (:1593). One call, result passed down to both the takeover detection and the existing transition body (which drops its own call). `resolve_git_identity` is a pure filesystem read (git_identity.py:131-171) — no cost concern.
2. **Takeover detection, before the note guard**: compute `same_status_takeover_shape = (status == "in_progress" == current status) AND live foreign claim (claimed_by email + caller email both non-empty, casefolded-unequal)`. (Liveness needs no extra check here — current status IS in_progress in this shape.)
3. **Note guard (:1593-1598) carve-out**: its rejection condition gains `AND NOT same_status_takeover_shape` — a note riding a same-status takeover annotates the takeover's OWN new transitions entry (the takeover IS the status event). Every other note-without-new-status call keeps today's rejection verbatim.
4. **New same-status-takeover mutation branch**, parallel to (not inside) the `if status != current` body (:1605): when `same_status_takeover_shape`:
   - WITH note → snapshot prior `claimed_by`/claimed-at FIRST (rev-2 finding #8: the metadata dict at :1587 mutates in place), then restamp `claimed_by` to caller, append transitions entry `{status: "in_progress", at, by, note}`, set `metadata_changed = True`, stash `claim_warning` (kind claim_collision, claimant = the SNAPSHOT).
   - WITHOUT note → no mutation from the status arg; stash `claim_warning` (kind takeover_note_required) — the stashed warning itself is what step 5 keys on to distinguish a bare poke from a true nothing-call.
5. **No-updatable-fields gate (:1673-1679) carve-out**: when a `claim_warning` was stashed and nothing else changed, do NOT error — and do NOT early-return either: fall through to the normal reembed + response assembly (:1684-1693) so the response keeps its full shape (incl. `reembed`; re-upserting an unmutated node is harmless). The gate keeps erroring for every warningless nothing-call (same-identity bare pokes included).
6. **2a enforcement + warning**: inside the existing real-transition body, when new status == "in_progress" and prior status == "blocked" and live foreign claim: no note → early-return error (before any mutation); note present → proceed (existing restamp path) + stash `claim_warning` from the pre-mutation snapshot. Reopen detection (§3) rides the same body for done/cancelled→open.
7. **Response assembly**: `result = _get_node(...)` as today (:1693); attach the stashed `claim_warning` if any.

**3) Reopen warning (same shape, no note requirement).** done→open or cancelled→open where the CLOSING transition's `by.email` (latest transitions entry with status done/cancelled, last-wins scan) is a foreign verified email → success + `claim_warning` kind "reopen", claimant = closer, claimed_at = closed-at. **No closing entry found (legacy/hand-built journal) → warning SUPPRESSED** — cannot attribute the closer = unverifiable, same doctrine as the identity carve-out (rev-1 HIGH #4); acceptance tests this symmetrically with the claim-side null case. No note required (ruling scopes the note to takeover); docstring recommends one. Reopening your own: no warning.

**4) Interpretation record (rev-1 HIGH #5 — flagged, not buried).** "Never blocks" vs "requires a transition note" is a genuine tension. This brief resolves it: the note is ENFORCED (retryable validation) for the unambiguous transition-based takeover (2a) and for the seizure itself in the same-status shape (2b vs 2c split), while every ambiguous or non-takeover shape warns-and-proceeds. Grounds: (i) each ruling sentence must do work — an unenforced "requires" is decoration; (ii) the alternative (silent no-op without note) recreates the exact double-work failure TC4 exists to prevent, unless paired with the 2c warning, which this design does pair; (iii) the rejection is instructive and single-retry-recoverable — a nudge, not a wall. RECORDED as a decision node at handoff and EXPLICITLY flagged for Colton's async review — if he rules the other way, 2a's rejection relaxes to warning-plus-proceed in a patch (the warning plumbing is identical; only the early-return flips).

**5) Claim-age helper: extract, don't duplicate (rev-1 HIGH #3).** Move/extract `_task_claimed_at` into the tools layer (e.g. alongside `_node_email` usage in cognition_tools.py, or a small shared util) and have BOTH dashboard/api.py and `_update_task` consume the single implementation — mechanical conformance change to dashboard imports only, zero dashboard behavior change (its tests unmodified). A second hand-rolled copy is the named drift risk.

**7) What does NOT change.** `_TASK_TRANSITIONS` untouched (same-status was never routed through it; 2b hooks the guard, not the table). Non-collision stamping byte-identical. Assignment ≠ claim (assigned_to never triggers anything; info-only field in the warning).

## ACCEPTANCE

- 2a: blocked→in_progress over live foreign claim rejects without note (error names claimant + age; task state unmutated — claimed_by/transitions asserted unchanged after rejection); succeeds with note (restamp + note in transitions entry + claim_warning kind claim_collision). open→in_progress over a RELEASED foreign claim: no warning, no note needed (liveness boundary).
- 2b: same-status foreign + note seizes (restamp + new transitions entry with note + warning whose claimant is the PRIOR claimant — snapshot-before-mutate asserted); the carved :1593-1598 guard still rejects note+same-status for every NON-takeover shape verbatim (regression test: same-identity note+same-status, and foreign note+same-status on a RELEASED claim, both keep today's error).
- 2c: same-status foreign no-note COMBO call: no error, no restamp, other fields applied, claim_warning kind takeover_note_required with the "you did NOT take it over" message. BARE poke (status only, foreign live claim): succeeds with the same warning and an unmutated node (the :1673-1679 gate bypassed only when a claim_warning was stashed).
- Same-identity: combo same-status unchanged (test_task.py:479 green unmodified); BARE same-identity poke still errors "No updatable fields" (today's behavior pinned by test); self-re-claim after self-block: no warning, no note.
- Reopen: warning on foreign done→open and cancelled→open (closer + closed-at); absent on own-reopen; SUPPRESSED when no closing transitions entry exists (fixture via hand-built metadata, mirroring the claim-side null test).
- claimed_at: null for legacy claimed_by-without-transition fixture; correct after a 2b takeover (consecutive in_progress entries — last-wins).
- Unverifiable identity: empty caller email AND empty claimed_by.email fixtures each bypass all machinery (no warning, no note requirement, today's behavior).
- Helper extraction: dashboard tests unmodified and passing; single shared claimed-at implementation (grep proves no second copy).
- Test-change enumeration honored: test_task.py:418 updated with note + sibling rejection test; no other pre-existing test modified except as enumerated.

## REJECTED ALTERNATIVES (record)

Hard-blocking foreign claims (ACL): rejected by ruling — warning not wall. Silent no-op takeover-without-note (no warning): recreates the double-work failure; rejected in favor of 2c's explicit warning. Auto-releasing stale claims: TC16's territory to SURFACE, nothing expires possession. Clearing claimed_by on in_progress→open: changes stamped-history semantics other consumers read — keep stickiness. Bare-string warning key: loses structured claimant/age. Note required for reopen: ruling scoped it to takeover. Warning on foreign in_progress→open/done/cancelled (closing/releasing someone else's live work): real but unruled — known limit, TC16/dashboard visibility, not more validation here (also the first half of the two-call steal, disclosed below).

## KNOWN LIMITS (document)

Identity is git-config-declared — a collision warning proves stamps differ, not that humans differ. Unverifiable identities bypass by design. The two-call steal (foreign in_progress→open, then open→in_progress) evades the note requirement — both calls are audited (`by` on each transitions entry) but not warned in this WP; disclosed for TC16/dashboard rather than patched with more validation. The 2c warning depends on the caller relaying it — an agent that ignores response keys can still believe it claimed a task it didn't (mitigated by docstring prominence).
