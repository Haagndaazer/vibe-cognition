## PROBLEM

Tasks know who created and who claimed them (server-stamped), but there is no directed "this is YOURS" — a manager cannot assign work such that the assignee's next session surfaces it. `owner` exists but is free text (often an agent name) and deliberately not identity-matched (prime.py:118 records that rejection). Assignment must be email-keyed, stamped, and auditable.

## DESIGN

**Field shape (metadata, additive only).**
- `metadata.assigned_to`: casefolded email string, or absent/"" (unassigned). Client-declared, trust-based (like reports_to_email); the assignEE need not be a registered person node — dangling is legal, mirroring TC5's reports_to semantics.
- `metadata.assignments`: append-only audit list `[{to, by, at}]` — `to` = the casefolded target email ("" for an explicit unassign), `by` = SERVER-RESOLVED git identity of who did it (the unforgeable part — same mechanism as transitions/profile_history), `at` = timestamp. `assigned_by` is not a separate field; it's the `by` of the LAST assignments entry.

**Tool changes (no new tools — surface minimalism; Gate B self-sufficiency audit triggers).**
- `cognition_add_task`: new `assigned_to_email: str | None = None` — assign at creation; when set (non-blank after strip+casefold), seeds `metadata.assigned_to` + the first assignments entry. **Creation-time blank/whitespace value == NOT PROVIDED (peer-review corrected):** skip seeding both the field and the audit entry entirely — never store `""` or a spurious `{to: ""}` entry. (Do NOT copy `owner`'s creation path, which stores the raw value.)
- `cognition_update_task`: new `assigned_to_email: str | None = None` — set/reassign; `""` explicitly unassigns; omit = no change (None-means-no-change, the tool's existing convention). Every effective change (including unassign) appends one assignments entry. A no-op (same email) appends nothing. **No-op guard model (peer-review corrected): pattern on `_update_person`'s compare-before-append reports_to handling (cognition_tools.py:1810), NOT the adjacent `owner` block** — the owner block has no no-op check (it unconditionally sets metadata_changed whenever passed, cognition_tools.py:1521-1523) and is the WRONG template; copying it would spuriously append an audit entry on every same-email call.
- Docstrings must draw the line plainly: `owner` = free-text display, may be an agent, never identity-matched; `assigned_to_email` = the email-keyed HUMAN accountability field that drives session-start surfacing. Self-assignment is fine. Anyone may assign anyone (local trust domain; the audit trail is the control — same doctrine as update_person).
- Emails casefolded at write (the single-normalization convention). Blank-after-casefold explicit email (whitespace) → treated as `""` (unassign), not an error.

**Reads.** `cognition_list_tasks` rows gain `assigned_to` (absent/None when unassigned — never coerced, matching from_agent discipline). `cognition_get_node` already passes metadata through — no change. Optional parity (implementer's call, propose at gate): `assigned_to_registered` bool in update/add returns, mirroring reports_to_registered via the TC5 person-scan helper.

**Prime ("Your Open Tasks" matches assignment — the point of the WP).** Extend `_task_matches_email` (prime.py:169-175) to also match `metadata.assigned_to` (casefolded string compare — note it's a STRING, not a `{name,email}` stamp dict like created_by/claimed_by; don't force it through the stamp-dict branch). Light-import unaffected. Behavior deltas are additive: a task assigned to you now appears in Your Open Tasks even though you neither created nor claimed it — the manager-assigns → subordinate-sees-at-session-start loop. Tasks with no assigned_to behave byte-identically to today.

**What does NOT change.**
- Claiming stays distinct from assignment: status→in_progress does NOT auto-set assigned_to, and assignment does NOT auto-claim. They answer different questions (took it vs. was given it); TC4 (Wave 2) owns collision/takeover warnings, including any assign-vs-claim mismatch warning — do not build warnings here.
- `owner` free-text: untouched, still never matched.
- Dashboard: nothing. WP-TC11 is in flight with a frozen /api/tasks contract; TC8 is metadata-additive and invisible to it. Assignment chips are dashboard V3 territory.
- No new node types, no edges, no prime section additions (assignment rides the existing Your Open Tasks section).

**Concurrent-WP guard.** TC11 (dashboard) is being implemented in parallel and touches NO shared files with TC8 (dashboard/ + tests vs. cognition_tools.py task core + prime.py) — verified: no /api/tasks handler exists yet. Residual (peer-review noted): TC11's /api/tasks contract field-whitelists its rows, so TC8's assigned_to will be additive-but-invisible in dashboard V1 — expected and fine (assignment chips are V3); noting so nobody reads invisibility as breakage. ## REJECTED ALTERNATIVES (recorded)

- Matching assignment on `owner` free text: long-rejected (prime.py:118 — free text is not identity).
- Separate cognition_assign_task tool: rejected — update_task already owns task mutation with None-means-no-change semantics; a new tool widens Gate B surface for one field.
- assigned_by as a standalone metadata field: rejected — derivable from the audit tail; two sources of truth would drift.
- Auto-assign on claim (or auto-claim on assign): rejected — collapses two distinct semantics; TC4 owns the mismatch-warning story.
- Requiring the assignee to be a registered person: rejected — mirrors TC5's dangling-reports_to ruling; a manager assigns before the teammate onboards.

## KNOWN LIMITS (document, don't fix)

assigned_to is client-declared and unverifiable (the server proves who ASSIGNED, not that the assignee accepted). No notification beyond the assignee's next session-start prime (push notifications permanently rejected by Colton). Assignment of a done/cancelled task is legal and unwarned in this WP (TC4 may add a warning).
