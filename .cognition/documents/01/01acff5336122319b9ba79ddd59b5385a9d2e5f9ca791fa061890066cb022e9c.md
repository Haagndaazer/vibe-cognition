## PROBLEM

Prime personalizes tasks/activity per email (P13n-2) but is role-blind: a manager gets no view of what their reports are carrying (in-progress, stale, blocked), and a subordinate gets no standing view of their manager's recent decisions. The org data already exists (person nodes with reports_to_email, casefolded email identity, claimed_by stamps + transitions from TC4-era mechanics) — nothing reads it at session start.

## DESIGN

**Role derivation (one person-node scan per prime).** Resolve the session email → scan person nodes ONCE building: (a) `my_person` (casefolded email match), (b) `direct_reports` = person nodes whose `reports_to_email` casefolds to the session email, (c) `my_manager_email` = my_person's reports_to_email (casefolded, may be empty). Roles: MANAGER iff direct_reports non-empty; SUBORDINATE iff my_manager_email non-empty. Both may hold (middle manager → both sections). Neither, or no session email, or no person node, or `_should_personalize` false → **NO new sections, prime byte-identical to today** (the compat invariant every prime WP has honored).

**Helper relocation (single-implementation doctrine vs light-import).** `_task_claimed_at` moves from cognition_tools to a LIGHT shared home — `cognition/task_meta.py` (new, stdlib-only) or models.py (implementer's pick between those two, nothing heavier; both verified cycle-free for all three consumers, and cognition_tools→prime is already a one-way dep so prime.py itself is NOT a legal home) — with cognition_tools, dashboard/api.py, AND prime all importing from there. **cognition_tools RE-EXPORTS the name** (`from ..cognition.task_meta import _task_claimed_at`) so tests/test_task.py:32-37's direct import keeps working — the OLD path stays valid deliberately; existing tests unmodified. dashboard/api.py:21's import line gets the mechanical one-line path edit (this IS a dashboard file edit — see What-does-NOT-change for the precise wording). Fails-before for the relocation = TDD phrasing: the new shared-module import fails before task_meta.py exists — NOT "old path fails" (it must not).

**Manager rollup section — `## Your Team` (personalized-only, MANAGER role).** One storage.get_nodes_by_type(TASK) scan; for tasks whose `claimed_by.email` casefolds to a direct report's email:
- *In progress*: status in_progress → "- <summary> (<report name>, claimed <age>)" — age humanized from `_task_claimed_at` (days granularity).
- *Stale*: in_progress AND claim age STRICTLY greater than `prime_stale_claim_days` (NEW knob, default 7): stale iff `now - claimed_at > timedelta(days=knob)` — exactly-7-days-old is NOT stale (boundary pinned; the acceptance test asserts both sides). A null claimed_at (legacy, no in_progress transition entry) is NEVER stale — unverifiable ≠ stale, the standing doctrine.
- *Blocked*: status blocked → "- <summary> (<report name>, blocked)".
Capped by `prime_rollup_cap` (NEW knob, default 5) TOTAL rows (stale first — they're the actionable ones — then blocked, then in-progress, recency desc within groups; "+N more" overflow line like existing sections). Unclaimed/unstamped tasks never appear (attribution doctrine). Section placement: immediately after Team Critical (it's the manager's counterpart view), before Your Recent Activity.

**Subordinate section — `## Your Manager's Recent Decisions` (personalized-only, SUBORDINATE role).** Decision nodes whose `_node_email` casefolds to my_manager_email, newest first, capped by `prime_manager_decision_limit` (NEW knob, default 3). NO HEAD-filter — peer-review corrected: the global Recent Decisions model (`_format_decisions`, prime.py:361-368) is a bare recent-nodes list with NO supersedes check (HEAD-filtering exists in constraints :294-296 and workflows :313-315, not decisions); this section mirrors its model exactly rather than inventing divergent semantics. **Placement (peer-review blocking fix): immediately after `## Your Team` — i.e. the pinned personalized order is Team Critical → Your Team (manager) → Your Manager's Recent Decisions (subordinate) → Your Recent Activity; a middle manager gets both new sections adjacent in that order.** "Own claims" (the ruling's other half) is ALREADY delivered by Your Open Tasks' claimed_by branch (P13n-2, prime.py:186-202) — no new section, the brief documents this mapping explicitly so the ruling item is traceably covered, not dropped. Overlap disclosure: a manager's decision can appear in BOTH this section and the global Recent Decisions — deliberate (global semantics untouched), documented in the section docstring; dedup would silently change the global section's meaning. Manager email dangling (no person node registered for it) is FINE — decisions filter by email string; the manager's name renders from the decision stamps.

**Config.** Two/three new knobs (prime_stale_claim_days=7, prime_rollup_cap=5, prime_manager_decision_limit=3) in BOTH PrimeConfig and Settings — the defaults-equivalence test (test_prime.py:93-99) extends automatically-ish; no on/off knob for the role sections themselves (they self-gate on role existence; a roster-less graph never shows them — consistent with the personalization auto philosophy, and one fewer knob to document). prime_personalize=off kills them along with everything else (they are personalized sections).

**What does NOT change.** Global sections byte-identical for role-less users (THE regression fixture). Onboarding, constraints, task split, activity: untouched. No MCP tool changes (prime is hook/CLI surface — Gate B tool-surface audit N/A; battle-plan :61's qualifying set verified to exclude TC16). No dashboard BEHAVIOR or feature changes — but the relocation requires the mechanical one-line import-path edit in dashboard/api.py:21 (in scope as plumbing; dashboard tests unmodified and passing prove behavior held — do not read "untouched" as forbidding that line). No agent roles anywhere (ruling 6be2e867f91e — humans only; teammate-comms owns agent roles). Single-manager: reports_to_email is one string; matrixed orgs out of scope (documented known limit, ruled).

**Prime size discipline (Gate C2 hook).** New sections are capped, fire only for actual managers/subordinates, and cost one extra person scan + reuse of the existing task scan pattern. C2 measures multi-user prime size/latency; the brief pins the shape: no per-report task scans (ONE task pass bucketed by claimant email), no embedding/tools imports, caps enforced before formatting.

## KNOWN LIMITS (document)

Single-manager assumption (ruled; matrixed orgs out of scope). Stale detection requires a transitions-recorded claim — pre-TC4-era claims with no in_progress entry never go stale (disclosed, consistent with unverifiable-doctrine). Role data is trust-declared person-node content (peer-editable; audit trail is the control). The rollup sees only stamped claims — work tracked outside tasks (or claimed by unregistered emails... reports are matched by registered person nodes' emails, so an unregistered claimant is invisible to the rollup even if reports_to_email points at the manager — registration is the onboarding notice's job).
