## DESIGN

**Shape: a NEW dedicated analyzer agent** — `agents/curate-conflict-analyzer.md` — spawned by the orchestrator as a separate pass, NOT extra prose in curate-edge-analyzer. Rationale: the retrieval pattern differs (edge-analyzer relates a new node to its neighborhood; conflict hunting compares stance-bearing nodes against SAME-TOPIC nodes anywhere in the graph, including long-curated ones), and a dedicated propose-only agent keeps the precision measurement attributable. Frontmatter mirrors the sibling analyzers: model pin haiku + read-only cognition tools (get_node, get_neighbors, search, get_history). Propose-only — NEVER writes; the orchestrator remains the sole edge writer (curation containment constraint unchanged).

**Candidate set (peer-review corrected — timing is load-bearing).** Stance-bearing nodes from THIS RUN's worklist: decision, constraint, pattern (assumption when present). NOT episodes/discoveries/tasks/documents (no stance to conflict). **The candidate IDs MUST be captured from the Step-1 `cognition_get_uncurated_nodes` worklist and passed explicitly to the conflict pass — NEVER re-fetched at the pass's insertion point**: Step 2 marks each batch curated as it completes (curate-orchestrator.md:61), so by the time the conflict pass runs, an uncurated-nodes fetch returns EMPTY and the pass silently does nothing every run. For each candidate: search by summary + context terms; compare against returned same-type-or-stance-bearing nodes not already edge-linked to it.

**Lens criteria (the content of the new agent's instructions — contract, wording Vorpid's):**
- Propose `contradicts` ONLY when both nodes assert INCOMPATIBLE STANCES on the SAME SUBJECT (do-X vs never-X; value pinned to V vs W; rule R vs explicit exception swallowing R). Both stances must be current — if either is superseded (HEAD check via get_neighbors), no proposal.
- Propose `supersedes` INSTEAD when it's the same decision lineage evolved (same subject, later node revises stance, especially same recorded_by identity) — temporal evolution is supersession, not conflict.
- NEVER propose for: different scopes (module A rule vs module B rule), refinement (B narrows A without negating it), mere topic overlap, or anything already connected by a supersedes/contradicts path.
- Every proposal carries `reason` PLUS verbatim evidence quotes from BOTH nodes (the stance sentences) — the orchestrator reviews the quotes, not the analyzer's conclusion.
- Output: the established JSON proposal contract with `"source": "curate-conflict"` per edge (distinct from "curate-skill" — provenance for TC15 observability and post-ship precision tracking).

**Orchestrator changes (agents/curate-orchestrator.md).** New step between edge curation and clustering (candidates from the Step-1 capture, per above): spawn `vibe-cognition:curate-conflict-analyzer` (explicit model override param, same mandatory-override rule as the other analyzers — fail f09e770da046) over the stance-bearing candidate IDs; review with conflict-specific hardening ON TOP of the standard rules: (a) both quotes must actually oppose — reject topic-adjacency; (b) HEAD check both endpoints; (c) prefer supersedes when lineage indicators present; (d) suspect-run cap (peer-review pinned): computed over the WHOLE RUN's examined candidates, not per-spawn, and ONLY when ≥15 candidates were examined (below that the ratio is statistically meaningless — small runs rely on the per-proposal review alone); if contradicts proposals exceed 20% of examined candidates, discard the pass's proposals and say so in the report. Commit via existing add_edges_batch. **The canonical final-report template at curate-orchestrator.md:12-14 MUST be edited** to include the conflict clause ("conflict pass: X proposed, Y committed, Z discarded") within the existing one-two-sentence cap — prose guidance alone loses to the "copy this shape" instruction.

**Precision gate (RULED — before ship).**
- Vince authors a labeled evaluation set: ~30-40 node PAIRS as a data file (tests/fixtures or scratchpad JSON): ~10 true conflicts, ~25+ hard negatives (supersession lineages, same-topic-different-scope, refinements, topic-overlap-no-stance) — drawn from the real graph where possible, synthetic where not. Labels are the manager's judgment call; the implementer does not author them (independence).
- Eval mechanism (peer-review corrected — a standalone script CANNOT reach an LLM in this repo: no anthropic dep, no API keys by design, and bench_wave1.py never invokes a model). Split: (a) Vorpid builds scripts/eval_conflict_lens.py which ONLY validates the labeled-pairs fixture and scores a results file (precision/recall/confusion listing); (b) the model-exercising step is AGENT-DRIVEN INSIDE A SESSION — an eval pass (run by Vince at gate, or a spawned eval-orchestrator) iterates the labeled pairs, spawns the REAL agents/curate-conflict-analyzer.md per pair via the Agent tool with the same explicit haiku pin discipline, and writes the propose/no-propose+type results JSON that (a) scores. NO mocked/prompt-approximated stand-in — the thing measured must be the shipped agent definition on the pinned model, or the precision gate is theater.
- GATE BAR: contradicts precision ≥ 0.9 on the labeled set (false conflicts poison trust — the bar is deliberately asymmetric); recall REPORTED but not gated (missing a conflict is today's status quo; inventing one is a regression). supersedes proposals from the lens judged by the same labels.
- If precision < 0.9 at haiku: the remediation knob is escalating the analyzer's model pin to sonnet and re-measuring — model cost is the fallback, not a lowered bar. If sonnet also misses the bar, STOP and report to Vince (design problem, not tuning).

**What does NOT change.** No src/ code (agent-file + SKILL-adjacent docs only — verify zero src diff at gate). Curation containment: orchestrator remains the ONLY writer; conflict analyzer is read-only. SKILL.md launcher unchanged (pipeline boundary holds). Edge idempotency (from,to,type) unchanged. cognition_search/prime unchanged (TC2's territory). No new MCP tools.

## REJECTED ALTERNATIVES (recorded)

Folding hunting criteria into curate-edge-analyzer.md: rejected — different retrieval pattern, unattributable precision, and bloating the general analyzer's prompt degrades its primary job. Auto-committing conflict edges without orchestrator review: rejected (containment + trust). Gating recall: rejected — asymmetric cost; recall improves iteratively post-ship. LLM-labeled eval set: rejected — the labels are the ground truth, they must come from human-delegated judgment (Vince), not the same model class being measured. Marking conflicts via a new node type instead of edges: rejected — contradicts edges already exist, are dashboard-rendered, and are TC2's contract.

## KNOWN LIMITS (document)

The lens sees only uncurated nodes as candidates — two OLD nodes that already conflict are found only if one is re-touched (acceptable: the backlog case is new-work-vs-standing-decision). Precision is measured on a finite labeled set — production monitoring rides the distinct source string + TC15 observability. Haiku judgment variance run-to-run is absorbed by the orchestrator review + the suspect-batch cap.
