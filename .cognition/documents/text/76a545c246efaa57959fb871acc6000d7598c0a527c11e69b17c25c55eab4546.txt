## PROBLEM (verified)

recorded_by/created_by provenance stamps exist only from WP-P13n-1 (2026-07-07). Every node written before that — on ALL existing graphs, including this repo's own pre-July-7 nodes — carries only the free-text `author` field. `_node_email` (prime.py:166-181) reads stamps ONLY and deliberately never `author`. Consequence: legacy nodes are invisible to personalized prime, search identity weighting, exclude-people filters, activity attribution, and the queued unregistered-writers view. New writes work; history is a provenance desert.

## GOALS / NON-GOALS

GOALS: (1) explicit, user-initiated, dry-run-first backfill command stamping legacy nodes with inferred recorded_by/created_by; (2) inferences distinguishable from server-resolved stamps FOREVER; (3) unmappable nodes left untouched — never guessed; (4) append-only journal mechanics, no rewrite.
NON-GOALS: automatic backfill at session start; rewriting `author` fields (verbatim forever); claims/transitions reconstruction (see SCOPE); person-node auto-registration; changes to live stamp minting; MCP-tool delivery in v1 (wrong trust posture — an inference-writer inside every agent's reach; same reasoning as curator-only edge containment).

## NODE-TYPE SCOPE (peer-review H4 + person-scope finding)

- IN: entity nodes (decision/fail/discovery/constraint/pattern/assumption/incident/episode/workflow) → `recorded_by`; TASK nodes → `created_by` (stamp keys verified against the live write paths: cognition_tools.py:278 and :1468).
- OUT, DOCUMENT nodes: the LIVE write path never stamps them — _store_document's metadata (cognition_tools.py:1160-1168) has no recorded_by AT ALL, so there is no live shape for a backfill to match. That live gap is its own bug, filed separately (task 2858ae93bf17); document backfill becomes a v2 line item only after the live path stamps.
- OUT (default), PERSON nodes: their `recorded_by` records the REGISTRAR (who ran register_person), not the person's identity — and because `_distinct_stamped_emails` scans get_all_nodes() unfiltered (prime.py:184-190), backfilling registrar stamps FEEDS THE AUTO-FLIP calculation. Backfilling them buys nothing any consumer reads today and perturbs the flip math. Resolved question 3 confirms.

## MAPPING SOURCES & CONFIDENCE TIERS

The unit of inference: legacy node → (name, email). Three sources, composed in order; a node maps via the FIRST source that produces an unambiguous answer. (AMENDED by ruling Q1: sources 1 and 2 are SUGGESTION GENERATORS only — see RESOLVED QUESTIONS.)

1. **Roster match (HIGH):** node.author (casefolded, trimmed) exactly matches a registered person node's `person.name` (casefolded) → that person's email. Adoption order is register-roster-first, making the roster the explicit source of truth. Collision rule: two registered persons sharing a casefolded name → that name is unmappable by this source, falls through.

2. **Journal git-history match (MEDIUM, doubly gated):** `git blame --line-porcelain .cognition/journal.jsonl`, reading the `author`/`author-mail` fields (peer-review M6). A node maps via blame ONLY when BOTH gates pass:
   - **Name agreement:** the blame author NAME casefold-matches the node's `author` field (guards the flusher-attribution failure: under a shared-worktree flush protocol every line blames to the flusher — this repo's own journal blames Vince-authored lines to "Colton Dyck", verified).
   - **Email consistency (peer-review B1, demonstrated live):** if one author name's blame evidence resolves to MORE THAN ONE distinct email across the node set, that name is Source-2-AMBIGUOUS. (AMENDED by ruling Q6: drift now yields a MOST-RECENT-EMAIL pre-filled suggestion + drift annotation in the skeleton, instead of hard fall-to-manual — safe because nothing auto-stamps under Q1.) Real case: this repo's journal blames "Colton Dyck" to BOTH colton.dyck@acryliccode.com AND colton.dyck@studiomonsoon.net.
   - Additional exclusions: lines introduced by a bulk-rewrite commit — heuristic must be scale-invariant (peer-review M7): judged against the journal's size AT THAT COMMIT with an absolute-line floor, or a content signal, never a bare percentage of today's file; lines predating usable git history fall through.

3. **Explicit user mapping (HIGH, human-asserted):** `--map "Name=email"` (repeatable) and `--map-file <json>` whose shape is MANY-ALIASES-TO-ONE-EMAIL (peer-review L10): `{"email": "...", "aliases": ["Colton Dyck", "colto"]}` — one entry per person, not per spelling.

Unmapped after all three → LEFT UNSTAMPED, listed in the report by name with node counts; empty-author nodes listed as an explicit "(no author)" row (L14). Never guessed, never majority-voted, never similarity-matched.

**Expected-yield honesty (peer-review L11):** on THIS repo's own corpus, manual mapping is the MAJORITY path — ~240 author strings are agent personas ("Vince", "Vorpid", "vince", "curate-orchestrator", …) that no roster or blame gate will match, plus the "colto" name variant. The dry-run report UX must present the unmapped table as the main event, with the --map-file round-trip (emit a skeleton mapping file, user fills emails, re-run) as the primary workflow. Whether agent personas SHOULD map to a human's email at all is the graph owner's call, made explicitly in the map file.

## HONESTY MARKING (doctrine-critical — Colton's review point)

"recorded_by": {"name": "Casey Lead", "email": "casey.lead@x.com", "backfilled": true, "backfill_source": "roster|git-history|manual"}

- Consumers read `.email` and work UNCHANGED — zero consumer changes in v1; the marker is inert. Chroma metadata never carries recorded_by/created_by for ANY node (cognition_tools.py:126-130), so no embedding surface even sees stamps.
- Audits keep the distinction forever. (Ruling Q4: dashboard rendering of backfilled provenance DROPPED — marker stays machine-readable only.)
- **Correction path (peer-review H3):** `--recompute-backfilled` permits overwriting stamps that carry `backfilled: true`, while server-resolved stamps (no marker) remain untouchable under every flag. Default off; dry-run shows what would change.
- REJECTED marking alternatives: separate top-level metadata block (orphanable); global graph-level flag (no per-node granularity).

## WRITE MECHANICS (verified at 862cea4, both reviews)

`CognitionStorage.update_node(node_id, **kwargs)` (storage.py:301-320) journals an `update_node` event (journal-FIRST, C-4) and replaces top-level attrs; replay reproduces via _replay_entry's update_node branch (storage.py:1282-1289). Per node: read node, deep-copy metadata, inject the stamp, `update_node(node_id, metadata=new_meta)`. Append-only; linear journal growth, count disclosed in the report. No reembed needed — embed text is summary/detail(+task status/owner) only (cognition_tools.py:105-109).

**Skip predicate (peer-review H2 — must mirror _node_email, not key presence):** a node is "already stamped" iff its stamp key holds a dict with a NON-EMPTY email. A legacy-era empty-email stamp (resolve_git_identity returns "" when git config had no email, git_identity.py:171) MUST be backfill-eligible. Server-resolved non-empty stamps are never overwritten (absent --recompute-backfilled, and even then only marker-carrying ones).

**Concurrency honesty (peer-review H5):** get_node and update_node are separate lock acquisitions; a live stamp written in the gap would be clobbered. This residual is ACCEPTED and documented — same class as cross-process add_node minting (storage.py:233-236) — mitigated operationally: no-live-session instruction, journal-mtime watch with abort-on-change, re-read each node immediately before writing. No CAS primitive for a run-once migration. (migrate_mcp is precedent for the CLI UX only, NOT concurrency. M9.)

## DELIVERY SHAPE

`python -m vibe_cognition.backfill_identity <project-path> [--map ...] [--map-file ...] [--apply] [--recompute-backfilled]` — dry-run default; `--apply` opts in. Scoped to ONE target project's graph (L13). Dry-run report: per-source mapping table (name → suggested email → node count → source), the unmapped table (primary UX, with skeleton map-file emission), exact journal-line count to append, and the AUTO-FLIP forecast computed over BOTH `_should_personalize` arms (M8): post-backfill distinct stamped emails AND current registered-person-email count (prime.py:209-233). Ships with version bump + CHANGELOG per the release procedure (L12).

## TASK CLAIMS / TRANSITIONS (SCOPE)

v1 backfills task `created_by` only. NO claimed_by / per-transition `by` reconstruction: a claim is an ACTION whose actor the pre-P13n journal genuinely doesn't record — inferring it manufactures history. Read paths already defensive. (Ruling Q5: PERMANENTLY out.)

## PRIME / BEHAVIOR FLIP (disclosed, not "fixed")

Backfilling a formerly-solo-looking graph that truly had multiple writers WILL flip auto-personalize — the point, and why apply is explicit and the forecast is in the report. A genuinely-solo graph backfills to one distinct stamped email → no flip, prime byte-identical.

## TEST PLAN (sketch — the WP brief pins exact cases)

Roster match + casefold + duplicate-name collision → unmappable; blame gated on name agreement (flusher-committed journal fixture) AND email consistency (one name → two emails → drift suggestion + annotation, B1 regression updated per Q6); scale-invariant bulk-commit exclusion; alias map-file many→one; skip predicate empty-email → ELIGIBLE (H2) / non-empty → skipped; idempotency; --recompute-backfilled only marker-carrying; dry-run journal byte-identical; apply appends exactly N update_node events, replay reproduces; end-to-end personalized prime gains Your Activity post-backfill; auto-flip forecast correct in all four quadrants; "(no author)" row; journal-mtime abort. Fixture corpus: snapshot of this repo's own pre-July-7 journal shape.

## RESOLVED QUESTIONS — COLTON'S RULINGS (2026-07-16, decision node 833e9f67de4d)

1. Inference tolerance: MANUAL CONFIRMATION FOR EVERY NAME — auto-stamping rejected even on exact roster hits. Sources 1 (roster) and 2 (blame) become SUGGESTION GENERATORS that pre-fill the skeleton map file; the ONLY stamping path is dry-run → skeleton emission → human confirms/edits → `--apply` with the confirmed map file. Confidence tiers survive as suggestion annotations, not write authority.
2. Marker shape: keys inside the stamp dict ("backfilled": true + "backfill_source"), as proposed.
3. PERSON nodes: OUT of scope, confirmed.
4. Dashboard visual distinction for backfilled provenance: DROPPED — no follow-up task; marker machine-readable only.
5. Claims/transition reconstruction: PERMANENTLY OUT, confirmed.
6. Blame email-drift: auto-pick MOST RECENT email as the pre-filled suggestion (overrides peer-review B1's fall-to-manual). Safe under ruling 1: nothing auto-stamps, so B1's fabricated-fragmentation risk cannot occur — the human confirms the suggested email before any write. The email-consistency DETECTION stays (it triggers the most-recent suggestion + drift annotation).
7. DOCUMENT nodes: OUT until 2858ae93bf17 ships, confirmed.
8. Agent-persona authors: owner's explicit choice via map file, NO default, as recommended.
