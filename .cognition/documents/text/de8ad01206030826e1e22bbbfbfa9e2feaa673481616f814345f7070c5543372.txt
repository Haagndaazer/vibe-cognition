## PROBLEM

A returning user has no "what changed while I was away" view: prime shows recent-N sections regardless of whether the user already saw those rows yesterday. Teams want the delta — teammates' new decisions/constraints/incidents since MY last session — which requires remembering when my last session was, per email, per machine.

## DESIGN

**Marker file — `.cognition/last-seen.json` (NEW, git-ignored, machine-local).** JSON object: casefolded email → aware-UTC ISO timestamp of that email's last session-start, e.g. `{"colton.dyck@acryliccode.com": "2026-07-15T13:00:00+00:00"}`. Read-modify-write preserves other emails' entries (the per-email ruling). Constant lives in prime.py beside ONBOARD_DECLINE_FILENAME. Git hygiene: GIT_HYGIENE_VERSION 3→4, `last-seen.json` entry added to `_write_gitignore` (fails-before: entry absent at v3, present after v4 re-run).

**Who writes the marker (the load-bearing split).** generate_prime STAYS pure read-only. The marker is stamped in `main()` — the CLI/hook path — AFTER the prime output is produced, mirroring where `_consume_rehydrate_flag`'s mutation already lives: `_stamp_last_seen(cognition_dir, current_email)` sets my entry to now (aware UTC isoformat), UNCONDITIONALLY overwriting whatever was there (last-writer-wins; this is also the clock-skew self-heal — a future-dated marker gets replaced with real now on the next session). The read-modify-write cycle is guarded by the repo's established lock pattern (`git_hygiene._acquire_lock`/`_release_lock`, 60s stale detection — peer-review HIGH: two teammates starting sessions on a shared machine concurrently is EXACTLY the per-email ruling's scenario, and an unlocked RMW can lose the other email's fresh entry). Lock-acquire failure ⇒ SKIP the stamp entirely (a missed stamp degrades to the fallback window next session; blocking or failing the SessionStart hook is never acceptable). The write itself is atomic (temp file in the same dir + `os.replace`) so a crash never leaves torn JSON, and the whole stamp is wrapped in `contextlib.suppress(OSError)` — a read-only filesystem must never fail the hook. No stamp when current_email is empty. Empty-graph scope (peer-review): main()'s `if empty:` early-exit (:787-790, .cognition absent or zero nodes) never reaches the stamp — deliberate and acceptable: there is nothing to digest on an empty graph, and the None marker self-heals via the fallback window on the first real session. Other generate_prime callers (dashboard, instructions.py compact-reinject, tests, library use) never stamp — deliberate: only a real session-start consumes news.

**Marker read (tolerance discipline).** `_last_seen_for(cognition_dir, email) -> str | None`: missing file, unreadable file, malformed JSON, non-dict payload, non-string value → None (treated as first-run; never raises — the onboard-declined OSError→empty-set model, extended to json.JSONDecodeError). A None marker NEVER means silent no-op: it means the fallback window (below).

**Digest section — `## Since You Were Gone` (personalized-only).** Fires only when `personalize` is true AND current_email non-empty (it is a per-identity view; "teammates'" presumes a multi-user graph, which is what personalize=auto detects). Content: decision, constraint, and incident nodes with `timestamp > cutoff` where
- cutoff = my marker value when present, else `(now - timedelta(days=prime_digest_fallback_days)).isoformat()` (capped lookback ruling),
- comparison is LEXICOGRAPHIC ISO string compare — the string-compare MECHANISM mirrors `_format_incidents` (:385-389), but the operator is STRICTLY GREATER, a DELIBERATE DIVERGENCE from _format_incidents' `>=` (peer-review corrected: don't claim mirroring): incidents use an inclusive window cutoff where boundary inclusivity is harmless; the marker is a high-water mark where `>` gives exactly-once semantics for a node whose timestamp exactly equals it. No datetime arithmetic anywhere in the section, so the TC16-F2 naive-timestamp crash class is structurally absent (a naive/malformed node timestamp string simply compares low and drops out or shows — degrade, never crash),
- EXCLUDING nodes whose `_node_email(n)` equals my casefolded email — "your own writes are not news to you." Unstamped nodes ("" ≠ my email) are INCLUDED: the section is an awareness view, and new-but-unattributable content is still news; excluding it would silently hide teammates on machines with broken git identity. (Deliberate divergence from the rollup's attribution doctrine, which gates on POSITIVE attribution because it names people; this section reports content, not people.)
- Constraints are HEAD-filtered (mirror the Active Constraints section — the filter is the inline `not storage.get_predecessors(n["id"], CognitionEdgeType.SUPERSEDES)` at prime.py:297-302; the same expression is inlined at :320 workflows and :345 doc-count, there is NO shared helper — a 4th inline copy is fine, extracting a helper is equally fine provided the three existing sites are behaviorally untouched; discovery dfd9f59827b1 records why the filter matters — a superseded constraint version resurfacing as "news" is exactly that inflation bug). Decisions and incidents are NOT HEAD-filtered — mirrors their global sections exactly (the TC16 manager-decisions precedent: mirror each type's existing section semantics, never invent divergent ones).

Rows: newest-first across all three types interleaved, each line `- [decision|constraint|incident] <truncated summary> (<stamped name or "unattributed">, <ISO date>)`, capped at `prime_digest_cap` total with the standard `+N more` overflow line. Empty result (nothing new) → section omitted entirely (""), consistent with every other section's empty behavior — the "never a silent no-op" ruling applies to the FALLBACK-WINDOW mechanism (no marker must not mean no digest), not to a genuinely-empty delta.

**Placement.** Immediately AFTER the TC16 role sections, BEFORE `## Your Recent Activity`: pinned personalized order becomes … Team Critical → Your Team → Your Manager's Recent Decisions → **Since You Were Gone** → Your Recent Activity. Rationale: constraints/tasks are standing priorities and stay pinned up top; the digest is incremental news and reads naturally beside the other identity-derived sections. Role-less users on multi-user graphs still get it (it gates on personalize+email, NOT on manager/subordinate role).

**Overlap disclosure.** A new decision can appear in this digest AND in Your Manager's Recent Decisions AND in global Recent Decisions — deliberate, same reasoning as TC16's overlap ruling: deduping would couple sections' semantics. Documented in the section docstring.

**Config.** TWO new knobs in PrimeConfig AND Settings (equivalence test extends automatically): `prime_digest_cap: int = 5`, `prime_digest_fallback_days: int = 7`. No on/off knob — the section self-gates on personalize (TC16 no-dead-knob philosophy); prime_personalize=off kills it.

**What does NOT change.** generate_prime stays write-free (THE invariant — acceptance test asserts the marker file does not appear after a bare generate_prime call). Non-personalized prime byte-identical. Personalized prime for a user with an up-to-date marker and no news: byte-identical to pre-TC14 (section renders ""). No MCP tool changes (no tool-surface audit — hook/CLI surface only). No dashboard changes, not even imports. Marker file is NEVER committed (hygiene v4 + it must not appear in `git status` of a hygiene-initialized repo — fails-before on the gitignore entry).

## ACCEPTANCE

- Marker lifecycle: main() stamps my email (aware UTC) after output; other emails' entries preserved on rewrite; empty email → no stamp; empty graph (main early-exit) → no stamp; write failure (read-only dir) → hook output still produced, no exception; lock-acquire failure → stamp skipped, hook succeeds.
- Concurrency (peer-review HIGH): a GENUINELY CONCURRENT test of the RMW — two writers racing on different emails must both survive; the test must exercise the interleaving (threads with a barrier, or deterministic stale-snapshot injection), NOT a sequential preserved-entries check, which passes trivially without the lock.
- Marker robustness: null-valued entry ({"email": null}) → treated as missing → fallback window, no crash; marker timestamp in the FUTURE (clock skew/suspended VM) → digest may render empty this session, no crash, and the unconditional restamp overwrites it with real now (self-heal proven by a restamp assertion).
- generate_prime purity: bare generate_prime never creates/updates the marker.
- Digest filtering: node newer than marker shows; node older/equal absent; my own stamped node absent; unstamped node shows (explicit test — pins the inclusion ruling); superseded constraint absent (HEAD-filter); superseded decision SHOWS (mirror-global test, same as TC16's).
- Fallback: no marker → only nodes within fallback window show (older node absent — proves the cap; "never full-history dump"); malformed marker JSON → fallback window, no crash.
- Cap + overflow line at prime_digest_cap; newest-first interleave across types.
- Compat: personalize off → byte-identical prime; single-user graph (auto) → byte-identical.
- Tolerance: naive-timestamp node string in the scan → no crash (string compare); missing timestamp key → treated as "" (drops out, never crashes).
- Hygiene: GIT_HYGIENE_VERSION 4; v3-stamped dir gains the entry on next init (fails-before both sides).
- Knobs: PrimeConfig/Settings equivalence extends; defaults as pinned.
- Fails-before on: marker-cutoff filtering, fallback window cap, own-stamp exclusion, hygiene v4 entry, main()-stamps-marker.
- Docs: README (digest paragraph in the team section), SKILL.md, readme.py COGNITION_GUIDE, CHANGELOG [Unreleased]. No tool-surface audit (no MCP tool changed).
- Gate: standard — isolated worktree, pinned commit, independent pytest+ruff, off-limits untouched, journal untouched on branch. Version bump: HELD — TC14 batches with TC15 as the 0.24.0 candidate; the batch's LAST WP carries the bump commit.

## REJECTED ALTERNATIVES (record)

Marker in the graph/journal: violates machine-local + uncommitted ruling (and would sync one user's "seen" state across machines, which is wrong — seen-ness is per-machine). Marker per machine only (single timestamp): stomps the shared-machine manager/subordinate pair — ruled per-email. Stamping inside generate_prime: breaks the pure-read invariant and would mark "seen" on dashboard/library calls that no human read. A new MCP tool to stamp/read the marker: hook path suffices; tools are heavier surface (audit burden) for zero gain. Including tasks/episodes in the digest: tasks have their own personalized sections and episodes are high-volume noise; the ruled scope is decisions/constraints/incidents. Deduping against other sections: couples semantics (TC16 precedent). Datetime-parse the cutoff comparison: string compare is the in-file precedent and structurally immune to the naive-timestamp crash class.

## KNOWN LIMITS (document)

Per-machine semantics: the same email on two machines sees independent digests (ruled: machine-local). Ephemeral sandboxes re-fallback every session (ruled acceptable: capped window). Marker granularity is session-start, not "actually read" — starting a session marks everything seen even if the human ignored the output. Last-writer-wins on concurrent SAME-email sessions (benign: both stamps are "now" within seconds; the lock protects cross-email entry loss, not same-email ordering). Unstamped-node inclusion means a graph with systematically broken git identity shows more "news" — registration/onboarding is the control, same as the rollup's limit. Lexicographic compare assumes uniform +00:00 offsets — true of every write path at this pin (verified: no Z-suffix or non-UTC writer exists); a HAND-EDITED journal entry with a non-UTC offset (e.g. +05:30) could sort wrongly near the cutoff — degrade (wrong inclusion/exclusion of that row), never crash; shared with _format_incidents, disclosed not fixed. Clock skew: a future marker blanks the digest for one session, then self-heals on restamp.
