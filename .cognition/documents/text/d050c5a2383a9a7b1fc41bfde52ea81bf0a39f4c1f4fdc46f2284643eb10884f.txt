## CURRENT MECHANICS (verified in-session at 38bcb8b)

- Search core: _search_cognition (cognition_tools.py:857) — embeds query, adaptive over-query, then _format_search_results dedupes to best-hit-per-node and drops graph-absent ghosts; each result dict already carries id, node_type, summary, author, timestamp, severity, context, score, from_agent, weight{multiplier, seniority, from_agent, basis}, weighted_score (verified live in Gate D artifacts). Result dicts are plain dict[str, Any] — no typed row model, no pyright union trap.
- Conflict membership semantics are PINNED by pattern 6ed494680fb3: contradicts is one-way/arbitrary-direction → bidirectional check (incoming OR outgoing); supersedes is directional newer→older → INCOMING-only marks the stale side. Canonical implementation: _is_conflicted (dashboard/api.py, WP-DashV3) using storage.get_predecessors/get_successors.
- storage.get_predecessors(id, SUPERSEDES) returns the NEWER node(s) that supersede id (in-edge sources, storage.py:591-613 semantics; direction per the pattern).
- Weight doctrine (TC9, documented in the search docstring): weighting "NEVER hides or wipes" a hit and "never boosts anything above its raw score" — visibility-preserving by principle.

## DESIGN

**Two new ALWAYS-PRESENT fields on every cognition_search result dict** (both derived at format time in _format_search_results' existing per-result loop — one place, no second pass):

- `conflicted: bool` — true iff ≥1 incoming OR outgoing CONTRADICTS edge (bidirectional per pattern 6ed494680fb3). Same rule the dashboard rows use, so the two read surfaces finally agree.
- `superseded_by: str | null` — the node id of the NEWEST superseding node when the hit has ≥1 incoming SUPERSEDES edge, else null. An id (actionable pointer: the consumer can fetch the replacement or walk get_superseded_chain), not a bare bool. Branch case (multiple incoming supersedes edges — peer-review B2 corrected): pick the superseder whose NODE timestamp is lexicographically greatest. This is NEW tie-break logic — queries.py's existing branch handling (get_superseded_chain :100-112, get_workflow_head :140-148) just warns and takes the first by iteration order; do NOT claim or copy a precedent. TIMESTAMP TRAP (pin): storage.get_predecessors returns (source_id, edge_data) tuples and edge_data ALSO carries a 'timestamp' (edge-mint time, CognitionEdge.timestamp models.py:88-102) — the tie-break MUST use the superseding NODE's timestamp (storage.get_node(source_id)['timestamp'], authorship time), never the edge's (a backfilled edge's mint time can differ wildly). OUTGOING supersedes does NOT mark the hit (the newer node is the resolution — it gets `superseded_by: null`).

**Single-implementation relocation (mirrors the task_meta/freshness precedents):** the membership logic moves to a shared light home — `cognition/queries.py` gains `conflict_flags(storage, node_id) -> tuple[bool, str | None]` (conflicted-by-contradicts, superseded_by) — one implementation. **Dashboard wrapper composition (peer-review B1 — THE trap in this WP):** dashboard/api.py's _is_conflicted is NOT the tuple's bool alone — its existing definition ORs incoming-supersedes in, and test_dashboard.py:1109-1115 asserts a node with ONLY an incoming SUPERSEDES edge is conflicted:True. The wrapper is `flags = conflict_flags(storage, id); return flags[0] or flags[1] is not None` — behavior byte-identical, existing DashV3 tests pass UNMODIFIED (that remains the gate criterion; a bool-alone wrapper fails them). Search results keep the two signals SEPARATE (that's the point — the dashboard's single ⚠ collapses them, search consumers need the distinction). tools/cognition_tools.py imports from cognition.queries (already a dependency — get_superseded_chain lives there; queries.py imports only models/storage, stays light; dashboard must NOT be imported by tools).

**Shared-formatter disclosure (peer-review H3):** _format_search_results has exactly one other consumer — cognition_get_workflow's internal top-1 match (via _search_with_embedding, cognition_tools.py:3563-3567), which keeps only the id and discards the rest. The new flags get computed for that one throwaway result and discarded — 3 wasted edge lookups per get_workflow call, accepted (a special-case bypass isn't worth the branch); note it in the cost comment. _format_search_results already takes storage (:657) — no signature change. Multi-project fan-out already threads each foreign entry's OWN storage into the formatter (:3369-3388), so foreign hits get flags computed against the correct graph by construction — no extra work, but the fixture for it is cheap and worth one test (foreign-graph hit with a supersedes edge → flagged from ITS graph).

**Both fields are ALWAYS present** (null/false on clean hits) — matches the weight-field precedent ("ALWAYS present... never silent") and the absent-key ambiguity lesson: a missing key must never be readable as "checked and clean."

**NO ranking change in this WP (pinned decision, record it):** superseded/conflicted hits keep their weighted_score and position. Rationale: (a) the TC9 weight doctrine is visibility-preserving — "never hides"; a supersession down-weight could bury the junior's historically-valuable find entirely, destroying the narrative Gate D S4 wants retrievable; (b) the flag lets every consumer make its own call (an agent can prefer the superseded_by target; a narrative reader keeps both); (c) ranking changes alter search semantics for every existing consumer and deserve their own evaluated WP if real-world use shows flags alone aren't enough. Follow-up candidacy noted in the task, not built.

**Docstring (tool-surface audit applies — return shape changes):** cognition_search's docstring documents both fields, their direction semantics (one line each), the always-present rule, and — closing the Gate B-final disclosure gap for THIS tool — an explicit sentence that search results are otherwise NOT HEAD-filtered (superseded content ranks normally; the flag is the marker). CHANGELOG [Unreleased]. (README/instructions harmonization stays in doc WP 6e54452b9735, which goes last.)

**Cost note (code comment):** up to 3 edge lookups per FORMATTED result (post-dedupe, ≤limit results, so ≤3×limit lookups per search on the in-memory graph) — negligible; no caching.

**What does NOT change:** ranking/weighted_score; get_history/get_neighbors (audit hit search; extending flags there is a possible follow-up, noted in-task, not scope-crept here); prime; dashboard behavior (only the import source of its membership logic); no journal/storage changes; no MCP tool signatures (return-shape addition only).

## ACCEPTANCE

- Hit with incoming contradicts → conflicted true; hit with OUTGOING contradicts → conflicted true (fails-before: incoming-only implementation must fail — the exact V1-banner bug class); clean hit → false, key PRESENT.
- Superseded hit → superseded_by = the newer node's id (fails-before: a bool-only or backwards-direction implementation must fail); the SUPERSEDING hit → superseded_by null; branch case (two superseders) → deterministic newest pick; clean hit → null, key PRESENT.
- Gate D S4 replay shape: seed junior-find + senior-revision + supersedes edge (same shape as the audit scenario); search must return the junior hit WITH superseded_by pointing at the senior node — the assertion that closes the audit finding.

- Single implementation: dashboard tests pass UNMODIFIED after the relocation; grep-proof one membership implementation.
- Docstring names both fields + the not-HEAD-filtered disclosure; CHANGELOG updated.

## REJECTED ALTERNATIVES (record)

Down-weighting superseded hits (rejected this WP — see pinned decision above). HEAD-resolving search results (replacing a superseded hit with its HEAD, like get_workflow does): changes what search RETURNS — a consumer asking "what did we know and when" loses history; get_workflow's HEAD-resolution is right for workflows (you want the current procedure), wrong as a default for evidence search. Bool-only superseded flag: forces a second lookup to find the replacement — the id costs nothing extra and is the actionable half. Flags on get_history/get_neighbors in the same WP: scope creep past the audit finding; noted as follow-up. Computing flags inside adaptive_vector_search (pre-dedupe): wasted lookups on hits the dedupe drops.

## KNOWN LIMITS (document)

Flags are point-in-time graph state — a search cached by a client can go stale (same as every other field). Conflict flag is a bool (names/details via get_neighbors or the dashboard drawer — same division of labor as DashV3 rows). The stale-outranks-correction ranking itself remains (deliberately): the flag is the mitigation, ranking evolution is a possible follow-up with real-world evidence.
