## DESIGN

**Three features. All read-only. All behind existing middleware, sync-def handlers. One new endpoint, two extended row shapes, one new view.**

### 1. GET /api/people + People view
Endpoint: `{people: [{id, name, email, role, seniority, reports_to_email, reports_to_registered, recorded_by, from_agent, timestamp}], count}` — one PERSON-type scan, sorted by name (mirror cognition_list_people's SORT only). reports_to_registered derivation (peer-review corrected — do NOT mirror the tool's implementation): cognition_list_people's own derivation calls _find_person_by_email per row (cognition_tools.py:2281-2299, :2296), which re-runs the full PERSON scan for EVERY row — O(N²). The endpoint must instead build a casefolded-email set once from the single scan and do O(1) membership checks. (Same output, better shape; the tool's O(N²) is a separate pre-existing nit, out of scope here.)

Frontend People view (new nav entry, fetch-on-activation like V2 views, NOT on the 30s poll — same §8 acceptance, unchanged conditions): card grid. Each card: person chip (name), seniority chip (TEXT per mockup line 247: "role · seniority"; STYLE per section 2's pinned override — solid, never the mockup's dashed), email (muted), reports-to line — if reports_to_email matches a roster member, render that person's name as a clickable chip (client-side join over the already-fetched roster; clicking opens THAT person's node in the drawer); if dangling (reports_to_registered false), render the raw email with a muted "(unregistered)" suffix. Card click opens the shared drawer on the person node (GET /api/node works generically today; provenanceHTML renders recorded_by/from_agent). Plus a "view activity" affordance per card: switches to the Activity view and pre-fills the author text filter with the person's name (the existing filter matches author OR recorded_by name/email, app.js renderActivity — no new API). Empty roster → "no people registered" empty-state with a one-line pointer to cognition_register_person.

### 2. Seniority chips on identity renders (the person-chip upgrade, §5 item 1)
Client-side roster join — NO API change to rows: app.js fetches /api/people ONCE lazily (first time any render needs it; cached in a module-level map keyed by casefolded email; no polling, no refetch per view switch — NOTE, peer-review: this is NEW plumbing, not an existing idiom; boardTasksCache/activityCache both refetch per activation, so don't copy them) and identityChipHTML gains an optional seniority lookup: when the identity it renders is RESOLVED (server-stamped) and its email casefolds to a roster entry, append the seniority chip. **Chip style (peer-review HIGH — override the mockup):** the mockup's `.chip.seniority` CSS (mockup line 47) uses `border-style:dashed`, which COLLIDES with the shipped unverified signal (`.chip.person.unverified`, styles.css:144, dashed = untrusted free-text identity) — copying it verbatim would make a fully-verified identity's seniority read as "unverified". Design doc §4.7 (line 71) already instructs treating the mockup as layout, not spec, for exactly this chip-ambiguity class. The seniority chip MUST use a solid border / visually distinct style that cannot be confused with the unverified treatment. Chip text pin: the seniority word alone in rows; the fuller "role · seniority" form only in the drawer provenance block and People cards. NEVER attach seniority to a free-text/unverified author chip — an unverified name string must not borrow a registered person's authority (matches identityChipHTML's existing trust-class doctrine, and prime's never-match-free-text precedent, _node_email :166-181).

Race handling (pin it): renders that happen before the roster fetch resolves render WITHOUT seniority chips and are NOT retroactively re-rendered — the next natural re-render (view switch, drawer open, poll tick) picks the cache up. No render-blocking on the roster fetch, no flicker-inducing global re-render. (A person registered mid-session appears after a page reload — acceptable, document it.)

### 3. List-level conflict flags (§4.7's Board-card warning icon + feed parity)
**Direction semantics (peer-review BLOCKING — pinned):** contradicts edges are stored ONE-WAY with arbitrary direction — cognition_add_edge's docstring says contradicts is "either direction" (cognition_tools.py:3756) and no reciprocal edge is ever created. So conflict membership is BIDIRECTIONAL for contradicts and DIRECTIONAL for supersedes:

`conflicted: bool` in _task_row and _entity_row = (≥1 INCOMING contradicts) OR (≥1 OUTGOING contradicts) OR (≥1 INCOMING supersedes). Concretely: get_predecessors(id, CONTRADICTS) or get_successors(id, CONTRADICTS) or get_predecessors(id, SUPERSEDES) — non-empty checks only; the flag is a boolean, the drawer remains where names/details live. OUTGOING supersedes must NOT flag (the newer node is not in conflict; it IS the resolution).

**Drawer banner repair (same WP, same reason the TC15 Step-4 fix shipped with its counter):** the V1 conflictBannerHTML (app.js:148-162) has the SAME pre-existing incoming-only gap — a node on the outgoing side of a contradicts edge shows no banner. Shipping the row flag without fixing the banner would flag rows whose drawer then looks clean (flag/banner disagreement, worse than either alone). Fix conflictBannerHTML to also scan node.successors for contradicts, rendering "⚠ Conflict: contradicts [id]" for the outgoing side (existing wording "contradicted by [id]" stays for incoming). Supersedes handling in the banner unchanged (incoming-only is CORRECT there).

Board cards and Activity rows render a small "⚠" indicator when conflicted (title tooltip "has a conflict — open for details"); clicking through to the drawer shows the (now direction-complete) banner naming the other node. Overview's constraint/attention lists: NO change (constraints there are already HEAD-filtered — a superseded constraint never appears, and adding the flag to a filtered list invites confusion; documented asymmetry).

Cost note (disclose in code comment): three edge lookups per row (in-contradicts, out-contradicts, in-supersedes) — NetworkX lookups on the in-memory MultiDiGraph, O(degree); at 100-row activity windows this is hundreds of dict lookups, negligible. No caching.

## REJECTED ALTERNATIVES (record)

Server-side seniority join into every row (_entity_row/_task_row gain seniority): bloats every list payload with derived data the client can join from one roster fetch; roster is tiny and stable within a session; and it would put person-scan cost on every list request instead of once. Per-person activity endpoint (/api/people/{id}/activity): the Activity view's existing author filter already answers it client-side over the fetched window; a dedicated endpoint adds surface for a filter convenience. Activity/task counts on people cards: requires a full node scan per roster load for a vanity metric; the "view activity" jump gives the real data. Retroactive re-render when the roster fetch lands: global re-render machinery for a one-session-boundary cosmetic gap. Conflict NAMES in list rows (not just a bool): duplicates the drawer's job into every row payload; the bool is the affordance, the drawer is the detail. Extending conflicted into Overview's HEAD-filtered lists: filtered lists structurally exclude the superseded case, and flagging contradicts-only there would make the same icon mean different things in different views.

## KNOWN LIMITS (document)

Roster is fetched once per session — a person registered mid-session gets chips only after reload (disclosed above). Seniority chips key on server-stamped emails only; graphs with systematically missing recorded_by (pre-P13n history) show few chips — correct behavior, not a bug. The conflicted flag treats contradicts and superseded identically at row level (one icon); the drawer differentiates. People view shows the flat roster, not an org tree — reports-to renders as a chip per card, not a hierarchy visualization (a tree is a future nicety if Colton wants it; the data supports it).
