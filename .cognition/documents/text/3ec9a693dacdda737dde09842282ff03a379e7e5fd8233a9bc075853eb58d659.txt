## DESIGN

**Change 1 — widen the 'auto' heuristic (fixes "registered team, single writer → no personalization"):**

`_should_personalize`'s auto branch becomes:

```python
return len(_distinct_stamped_emails(storage)) > 1 or len(_registered_person_emails(storage)) > 1
```

New helper `_registered_person_emails(storage) -> set[str]`: one `get_nodes_by_type(PERSON)` scan collecting non-empty `metadata.person.email` values (trust stored casefolding, module convention). A SET OF EMAILS, not a node count — duplicate person nodes for one email (journal-replay shape) must not flip a solo graph to multi-user (this is a fails-before acceptance case against a node-count implementation). Rationale line for the docstring: multiple REGISTERED people is a stronger team signal than multiple writers — registration is an explicit team act; the solo byte-identity guarantee is preserved because a solo graph has ≤1 registered person email. All three byte-identity pin tests (:329, :573, :770) must pass UNMODIFIED.

**Change 2 — one-line identity header opening the personalized block (fixes "no registration payoff"):**

- `_RoleContext` gains `my_manager_name: str = ""` — WITH the default (peer-review H1): `_derive_role` has TWO constructor call sites, the normal-path return (:477) AND the early `not current_email` short-circuit (:467); the default keeps the short-circuit valid, and BOTH sites are still updated/inspected explicitly (the short-circuit legitimately passes "" — currently dead code since generate_prime only derives role when personalize is true, which implies a non-empty email, but it must not become a latent TypeError). `_derive_role` resolves the name from the SAME single scan: collect an email→name map (`person.email → person.name`) while iterating; after the loop, `my_manager_name = by_email.get(my_manager_email, "")`. One scan preserved — update the docstring's one-scan sentence rather than contradicting it.
- In `generate_prime`, move `role = _derive_role(storage, current_email)` UP to compute once immediately after `personalize` is decided (inside `if personalize:` — do NOT scan when global). Both existing consumers (:848-849) use the same object; net scan count unchanged.
- New formatter `_format_identity_header(role: _RoleContext) -> str`: returns "" when `role.my_person is None`; else ONE plain line (no `##` heading — it's a preamble, not a section). Format ALGEBRA (peer-review M2 — pinned so punctuation can't drift): start with `You are registered as {name}`; if person.role non-empty append ` — {role}`; if seniority non-empty append ` ({seniority})`; if reports_to_email non-empty append `, reporting to {manager}` where manager = my_manager_name or (fallback) the raw reports_to_email; end with `.`. Literal expected strings (each a test assertion):
  - all fields: `You are registered as Jamie Junior — backend engineer (junior), reporting to Casey Lead.`
  - role empty, rest present: `You are registered as Jamie Junior (junior), reporting to Casey Lead.`
  - only name: `You are registered as Jamie Junior.`
  - manager email unresolvable to a person node: `You are registered as Jamie Junior — backend engineer (junior), reporting to casey.lead@test.local.`
  Never crash, never render "None".
- Placement: FIRST personalized element, inserted immediately before Your Tasks. The pinned order comment (:842-846) is updated to: Identity header → Your Tasks → Team Critical → Your Team → Your Manager's Recent Decisions → Since You Were Gone → Your Recent Activity — and RELOCATED (peer-review L1) to the top of the now-single reorganized `if personalize:` block (role derivation moves up, so the comment must sit above the code it describes, not orphaned above the second half).
- Gating is structural: header requires `personalize` AND a matching person node; the New Here banner requires NO matching person node — mutually exclusive by construction, no new knob. In `prime_personalize="on"` mode a registered user now gains the header too (intended: "on" always meant fully personalized).

This is the same experience any unregistered user of a multi-WRITER graph already gets today — same shape, not a new kind — and it nudges toward registering, which is the point of the banner above it.

## REJECTED ALTERNATIVES (record)

Personalizing on ANY registered person (≥1): flips every solo-registered graph, breaking the byte-identity pin — the whole reason 'auto' exists. Counting person NODES instead of distinct emails: replay-duplicate fragility. Registration acknowledgment via the register_person tool return or a write-time interceptor: tool-scope creep, and the acknowledgment/interceptor feature class is PERMANENTLY rejected by Colton — the header is a passive display line, which is why it's in scope. A `##`-headed "Who You Are" section: heavier than the payoff warrants; one preamble line reads better and costs less context. New config knob to toggle the header: knob proliferation; it self-gates on registration + personalize like every TC16 section.

## KNOWN LIMITS (document)

The header reflects the graph's registration record, not liveness (a stale role/seniority shows until someone updates the person node — same division of labor as everything else in prime). Unregistered users of multi-person graphs get the personalized-but-sparse layout described above. The Gate D S5 finding's second persona wish ("what changed for ME since registering") is Since You Were Gone's job and already ships; the header is the missing anchor line, not a new digest.
