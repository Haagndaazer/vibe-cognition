## PROBLEM

Containment is a documented convention with zero visibility: if a main-session agent hand-authors edges via cognition_add_edge/add_edges_batch (accidental misuse — the exact failure the constraint exists to prevent), nothing surfaces it. The graph's semantic-edge provenance quietly degrades and nobody notices until edge quality complaints appear.

## DESIGN

**Principle: derive, don't track.** The journal already records every edge's source; the count is computed at read time from replayed graph state. No write-path changes, no new persistence, no lifespan state, no run marker — restart-proof and identical on every clone by construction (journal-carried). This is the entire mechanism; anything stateful was rejected below.

**Source taxonomy (pinned).**
- CURATION-LEGITIMATE: "deterministic", "task-parent", "curate-skill", "curate-conflict", "curate-cluster" (new, see below), "curator" (legacy curation default — exempting it prevents every pre-existing graph from showing a scary false baseline).
- OUTSIDE-CURATION (the counted set): "manual", "batch", and ANY unknown/other value (conservative: an unrecognized producer is by definition not a known-legitimate one; a future legit producer must be added to the exempt set deliberately, in code, with a CHANGELOG note).
- Missing source in a replayed entry → replay already defaults it to "curator" (storage.py:1204) → legitimate; no new handling.

**get_status change (the feature).** `cognition_graph` gains TWO keys:
- `edge_sources`: dict — a histogram of ALL edge sources present, dynamic like the existing per-edge-type counts (full transparency; makes any count interpretable at a glance).
- `edges_outside_curation`: int — sum over the OUTSIDE set above (the at-a-glance signal the task text asks for).
Computed in the same pass that already iterates edges for the per-type counts — get_statistics's loop at storage.py:637 (`for _, _, edge_data in self._graph.edges(data=True)`; MultiDiGraph yields every parallel edge with its own source attr) — ONE pass, no second O(edges) walk. Type widening (peer-review HIGH): get_statistics is typed `-> dict[str, int]` (storage.py:615); nesting `edge_sources: dict[str,int]` as a value breaks that annotation and FAILS pyright's ratchet CI (ci.yml:82-83 vs .github/pyright-baseline.txt — reviewer reproduced the exact reportArgumentType error). Widen the return annotation (e.g. `dict[str, int | dict[str, int]]`) or return the histogram via a sibling structure — implementer's choice, but the annotation change is REQUIRED, and pyright parity with main stays a gate criterion. The static `curation` string is UNCHANGED (it documents the launch path, different job).

**Orchestrator Step 4 fix (conflation repair, same WP).** agents/curate-orchestrator.md Step 4 (:111): cluster part_of edges MUST set source="curate-cluster" explicitly (one instruction edit mirroring Step 2's :61 and Step 3's :91 wording). "curate-cluster" joins the exempt set. Going forward, a "batch" edge really is an anomaly. The agent .md change ships in the same WP because shipping the counter WITHOUT it would false-positive the very next cluster-producing curation run. Day-one baseline (peer-review measured, real journal): edges_outside_curation = 85 — 84 "batch" + 1 "manual". Narrative corrected per review: those 84 batch edges are HISTORICAL Step-2-STYLE semantic edges (led_to/relates_to/resolved_by/supersedes, dated 2026-06-22→07-01) that predate consistent curate-skill tagging — NOT cluster edges; the Step 4 fix prevents FUTURE conflation but does not and cannot shrink this baseline. Disclosed, NOT remediated (no journal rewriting, ever); the `edge_sources` histogram makes the baseline legible.

**Trust model (document verbatim).** `source` is caller-declared. A deliberately spoofed source="curate-skill" defeats the counter — IN SCOPE to ignore: the ruled threat model is accidental misuse in a local trust domain (a well-meaning agent calling add_edge with defaults), and the defaults are exactly what the counter catches. Auth/handshake designs are permanently rejected; do not revisit.

**What does NOT change.** No write-path behavior (all tools write exactly as before). No journal format change (no new action types). No dashboard change. No prime change. mark_curated untouched. The containment constraint text itself unchanged (still docs-first; this is a smoke detector, not a lock).

## ACCEPTANCE

- Histogram: graph with edges of every known source value → `edge_sources` exact per-source counts; edge with an unknown source ("mystery-tool") → appears in histogram AND counted in `edges_outside_curation`.
- Count: manual + batch + unknown counted; deterministic/task-parent/curate-skill/curate-conflict/curate-cluster/curator NOT counted; zero-edge graph → `edge_sources` {} and count 0 (keys PRESENT, absent-key discipline does not apply here — get_status always reports both).
- Legacy: replayed journal entry with NO source field → "curator" via existing fallback → not counted (fails-before: this test must fail if someone counts missing-source as outside).
- Single-pass: no additional full edge iteration beyond the existing stats pass (code-review criterion, not a timed test).
- Orchestrator .md: Step 4 sets source="curate-cluster"; wording mirrors Step 2/3's source lines. (Agent .md is prompt-content — the gate check is textual, plus the exempt set includes "curate-cluster" in code.)
- Fails-before on: outside-count computation (manual edge → count 1), unknown-source conservatism, curator-legacy exemption.
- Tool-surface audit: get_status IS an MCP tool and its output shape changes → run the RECURRING tool-surface self-sufficiency audit (workflow node) — the new keys must be self-explanatory from the tool description/docstring alone; update the get_status docstring to name both keys and the exempt/counted taxonomy.

## REJECTED ALTERNATIVES (record)

Token handshake / curation-run auth: permanently rejected by ruling (adversarial model out of scope) — do not revisit. Lifespan-state run flag ("curation in progress"): lost on restart, races multiple sessions, and adds the first stateful coupling between get_status and tool-call ordering — the derive-don't-track principle beats it. New journal action per violation: permanent per-violation records scale with misuse velocity and add a journal format change for what a read-time derivation gives free. Machine-local counter file (.last-rehydrate.json pattern): per-machine view of a cross-machine property (the journal syncs; the counter should match on every clone). Counting mark_curated misuse: out of task scope (task text says edge writes). Timestamp-window heuristics ("recent mark_curated ⇒ inside a run"): racy (survey: Step 2 writes edges BEFORE mark_curated), and the source field makes them unnecessary.

## KNOWN LIMITS (document)

Caller-declared source: deliberate spoofing invisible (ruled out of scope). Historical baseline: pre-TC15 "manual"/"batch" edges count forever — the histogram makes the baseline legible; no journal rewriting. The counter is graph-wide, not windowed — it answers "how many ever", not "how many since yesterday" (a TC14-digest-style delta was considered and dropped: get_status is stateless and the histogram plus journal timestamps let a human answer the delta question when it matters). Unknown-source conservatism means a future legitimate producer shows as violations until its source joins the exempt set in code — deliberate, forces the addition to be a reviewed decision.
