## DESIGN

Three new nav-rail views, matching V1's view-switching + lazy patterns. All read-only. All new endpoints behind the existing middleware, sync-def handlers like their siblings.

### 1. Workflows library — GET /api/workflows + view
Endpoint returns HEAD workflows only, each with its version chain inline: {workflows: [{id, summary, author, timestamp, chain: [{id, summary, timestamp}, … newest→oldest incl. HEAD]}], count} — chain via get_superseded_chain per HEAD (chains are short; one graph walk each, no N+1 HTTP). Frontend: card grid (summary, author, last-updated = HEAD timestamp); clicking a card opens the SHARED drawer on the HEAD node (full body via existing /api/node); the drawer's version-chain strip renders "v3 ⟵ v2 ⟵ v1" clickable (each hop re-opens the drawer on that node). Superseded (non-HEAD) workflows appear ONLY in chains, never as cards — mirrors prime's HEAD-filter semantics (discovery dfd9f59827b1 class).

### 2. Documents table — extend GET /api/documents + view
Existing endpoint gains per-row: freshness and cited_by (int — storage.get_predecessors(doc_node_id, CognitionEdgeType.PART_OF) count, direction pinned above).

Freshness semantics (pinned, peer-review corrected):
- mode == "copy" → null — EXPLICIT mode gate, NOT path-absence: a copy-mode doc stored via file_path RETAINS its path key (cognition_tools.py:1043/1148-1149 set path independent of store_copy), so literal reuse of the inline logic would wrongly compute a verdict for it. The gate is `if mode == "copy": null`, full stop.
- mode == "reference" AND no path key (docs stored via content_text — a COMMON state, not just legacy) → null — no source file exists to check; _get_document's current "unchanged" default for this state implies a check that never ran, and the dashboard badge must not repeat that implication.
- mode == "reference" with path → "unchanged" | "modified" | "missing" via the sha256 re-hash logic REUSED from _get_document as ONE shared implementation (single-implementation doctrine). Relocation target (peer-review): cognition/documents.py — it already owns sha256_file, has zero chroma/embedding imports, and api.py already imports from it (:19); no new module.

Design-doc reconciliation (peer-review B3 — explicit override, not silent substitution): design doc §4.4/§6 says "cheap stat-check variant, like search's staleness" (cheap_staleness_signal, documents.py:259-284 — path_missing/size_changed only, no file read). That helper structurally CANNOT detect a same-size content edit, so it can never produce the "modified" state the design doc's own badge vocabulary promises — §4.4's two clauses contradict each other. This brief resolves the contradiction in favor of the badge vocabulary: full re-hash, honest three-state result. KNOWN COST (disclose in code comment): O(total referenced bytes) per table load; fine at current scale (22 docs, KB-range); NO caching invented now (a stale freshness badge defeats the badge). Flagged for Colton's async review as a design-doc deviation with recorded rationale.

Frontend table columns per design doc: filename, mode, size, stored date, freshness badge (null renders as a muted "n/a" dash, not a state), cited-by count, download button (existing endpoint). Inert-doc flag = DERIVED client-side as cited_by == 0 (muted "uncited" chip) — no separate API field.

### 3. Activity feed — GET /api/activity + view
Endpoint rows REUSE _entity_row (api.py:369-387) verbatim — {id, type, summary, timestamp, severity, author, recorded_by, from_agent} (peer-review B2: the V1 trust-class rendering is CLIENT-side — identityChipHTML(resolved, author), app.js:88-99, always takes recorded_by and author as SEPARATE arguments; a pre-collapsed author string cannot express the trusted-vs-fallback distinction. severity rides along free and is useful on incident/constraint rows). Envelope {activity: [...], count}, newest-first across EPISODE, DECISION, FAIL, DISCOVERY, INCIDENT, CONSTRAINT, PATTERN, ASSUMPTION. Type-set provenance (peer-review H4): §4.5 literally names five (episode/decision/discovery/incident/fail); constraint/pattern/assumption are an EXTRAPOLATION by elimination (no other view claims them; models.py's 12-type enum is fully accounted for: these 8 in, task/document/workflow/person out — Board and libraries own those; no SUMMARY type exists). Deliberate overlap, documented: Overview shows HEAD-filtered active constraints (§4.1); Activity shows ALL constraint nodes unfiltered — a chronological record, not a standing-priorities view; do not HEAD-filter the feed.

Merge shape (peer-review H3, pinned): get_recent_nodes takes ONE node_type or None=all (storage.py:469-475) — the implementation is N per-type calls (one get_recent_nodes(limit=effective_limit, node_type=T) per included type), merged and re-sliced to the overall limit. NOT a single all-types call filtered afterward: excluded types (tasks especially, the most active) would displace genuinely-recent included rows out of the fetch window.

limit query param, default 100, server-capped at 500. Frontend: chronological list, type filter chips + author text filter, BOTH client-side over the fetched window (no server round-trip per filter click); clicking a row opens the shared drawer. Fetch on view activation; NO polling/auto-refresh (see risk acceptance).

### Threadpool risk 4163f54f2848 — explicit acceptance (design doc §8 mandate)
V2 ACCEPTS the risk, recorded here + a code comment at the new endpoints: all three are cheap sync reads over the in-memory graph EXCEPT documents freshness (file IO + hashing), which is bounded by stored-document count and runs in the same threadpool pattern as the existing download endpoint's file IO; V2 adds NO polling loops (every view fetches on activation only), so request volume does not structurally increase. Resolution (restructuring the spawn) stays deferred; if a future WP adds polling, the acceptance must be revisited then.

## ACCEPTANCE

- /api/workflows: HEAD-only cards (superseded workflow absent as top-level, present in its successor's chain); chain order newest→oldest; single-version workflow → chain of 1; branch-warned chains don't crash the endpoint (mirror get_workflow_head's tolerance); zero-workflow graph → {workflows: [], count: 0}.
- /api/documents: freshness per row for reference mode with path (unchanged/modified/missing each exercised — modify + delete a temp file in tests); copy mode → null EVEN WHEN the doc was stored via file_path + store_copy=True and retains its path key (peer-review H1 — this case must be a test); reference mode stored via content_text (no path key) → null (peer-review H2); cited_by correct with a descriptor citing doc_ref (fails-before: direction test must fail if the edge walk is backwards); doc with zero citations → cited_by 0.
- /api/activity: newest-first, type set exact (a task/document/workflow/person node NEVER appears), limit + cap enforced, rows carry recorded_by AND author separately (from _entity_row), from_agent passed through; a recent excluded-type flood does not displace included rows (proves the N-per-type merge — fails-before against a filtered all-types implementation).
- Auth: all three new endpoints return 403 without the token (middleware.py:32/44/48 — the suite's established assertion, tests/test_dashboard.py:151 et al; NOT 401), proving no route bypass.
- Single-implementation: freshness logic has ONE implementation (grep-proof); old import behavior preserved, existing document tests pass unmodified.
- Frontend: three nav entries; each view renders from its endpoint; drawer opens from workflow card, chain hop, document row, activity row; filters are client-side; no polling (code-review criterion).
- Read-only: no new POST/PUT/PATCH/DELETE routes (route-table test).
- Fails-before on: HEAD-only filter, cited_by direction, freshness modified-detection, activity type-set exclusion, auth on new routes.
- Docs: README dashboard section, CHANGELOG [Unreleased]. No tool-surface audit (no MCP tool changed).
- Gate: standard — isolated worktree, pinned commit, independent pytest+ruff (+ pyright parity), off-limits untouched, journal untouched on branch. Version bump: NONE on branch — V2 is a 0.25.0 candidate; bump decision at merge time (may batch with V3 or ship alone; Vince coordinates via the standalone-bump-branch mechanism).

## REJECTED ALTERNATIVES (record)

Client-side activity derivation from /api/graph (design doc left it open): /api/graph strips detail and carries the whole graph — a feed needs a cheap recency window, not a full-graph download; dedicated endpoint matches the per-view pattern V1 established. Freshness caching/mtime fast-path: stores no mtime today, and a stale badge defeats the badge's purpose; disclosed O(bytes) cost instead. cited_by as a stored/maintained counter: derive-don't-track (the TC15 principle) — the graph already has the edges. Server-side feed filters: two filters over ≤500 rows is client-side territory; server round-trips per chip click add threadpool load for nothing. Auto-refresh/polling on the feed: would reopen the §8 risk V2 explicitly accepts as fetch-on-activation-only. People view or any provenance-chip upgrades: V3 by phasing ruling.

## KNOWN LIMITS (document)

Freshness table load is O(total referenced bytes) per request (disclosed, fine at current scale). Activity window is last-N, not paginated — deep history via search/graph views (design doc doesn't ask for pagination). Chain rendering trusts SUPERSEDES chains to be short (they are, by construction — versioning is manual and rare); a pathological 1000-link chain would make one slow card, not a crash (cycle guard exists in the helper). from_agent renders as a plain field in V2 (the badge treatment is V3's).
