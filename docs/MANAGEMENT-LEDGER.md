# Management Ledger

Cross-project manager-craft rules, traded in `#manager-lessons` (vince / Silvie / Reginald, est. 2026-06-10). This file is the citable snapshot for vibe-cognition work packages; the channel history is the canonical record. Numbering is stable — cite as "ledger N".

1. **SHA-pinned approvals.** Sign-off names an exact commit SHA; the merge must be exactly that SHA. Enforce mechanically where possible (`gh pr merge --match-head-commit`).
2. **The voiding clause cuts both ways.** The implementer has the duty (not just the option) to HOLD a merge and void the manager's approval on any material post-sign-off information. Hold first, root-cause, then report — no permission needed. An approval is a snapshot, not a march order.
3. **Verify fixes under their FAILURE condition.** Green-under-clean proves nothing about the dirty environment the fix exists for. Run the fix against the exact condition that proved the bug.
4. **Inherited lines get MORE review suspicion than novel ones.** "Mirror existing pattern X" briefs must cross-check X against known findings first and name what NOT to copy — that's how flagged debt metastasizes.
5. **The defense is also a write to the defended thing.** Any protective change gets asked: does applying this protection itself touch the protected artifact? (Three instances day one: the journal-commit rule, the `-text` cut-over, the torch-swap upgrade brick.)
6. **Single-run green proves nothing timing-dependent.** Concurrency-touching work needs N consecutive green suite runs (we use 5) before pinning; tests where a worker thread can die must collect worker exceptions and assert the collection empty as its own labeled assertion.
7. **Decorrelated review layers.** Manager-read + adversarial subagent catch different classes; the cross-project manager trade is the only reviewer whose priors we can't contaminate — it has repeatedly falsified fixes the originating manager had approved.
8. **Pre-written acceptance criteria.** Written before the implementer starts; review checks against commitments, not re-derived correctness.
9. **Right commands on expired evidence.** Destructive-op bans exist because "it's a no-op" claims go stale between checking and acting. Verify seconds before acting, not minutes.
10. **Accidental correctness is an unverified dependency.** When you survive by sequencing luck, write the procedure down before you need it deliberate.
11. **Composition review.** Review the fixes within one work package against EACH OTHER, not just each against main — two individually-correct changes can compose into a regression the diff-vs-main view cannot show.
12. **The tautology check (on tests).** Of every new test ask: would this pass against the REVERTED code? Run the revert, don't reason about it. A green tautological test converts "untested" into "falsely verified."
13. **Process rules vs non-code outputs.** Every process rule gets checked against the implementer's side-channel state (graph records, logs, fixtures, generated docs). A rule that keeps such state out of durable storage is a data-loss bug wearing a tidiness costume.
14. **Multi-source briefs need a weighting rule.** When consulting multiple veteran sources, say whose testimony governs which domain — and weight by CONTEXT-MATCH, not seniority ("which of you was solving my topology?"). A weighting rule is the question "which parts of this source's context do I share?" made explicit. Disagreement resolution: primary artifact wins; reputation never does.
15. **Seam-gate the PLAN.** Inherited constraints arrive load-tested individually and untested pairwise. Before the first WP brief, run an adversarial pass seeded with named suspected seams ("here are four ways these decisions collide; find the fifth").
16. **Evidence expires across waits.** Any check-then-act where the act can BLOCK (lock waits, workflow resumes) must re-gather its evidence after the block, inside the exclusion that makes the re-read trustworthy. The block is exactly the window where the world changes.
17. **Check what your green-detector actually measures.** A pass-signal that is accidentally decorative (e.g., an exit code read after a pipe) converts "unverified" into "verified by luck." Gates assert on the program's own exit or structured output, never on what survived a pipe.
18. **The ledger's rules compose like any other constraints.** A fix that exists because of rule X gets its review seeded with "which OTHER ledger rule could this fix itself violate?" Rules 11/15 apply reflexively to our own remedies.
19. **An upgrade is also a write to the thing currently running.** Every release's constraint inventory includes the RUNNING copy of the previous version — its open handles, locks, and half of any shared state — because the diff cannot show a party that isn't in the repo. When a failure self-heals, ship detection + a clear message, not a code fix that adds its own seams.
20. **A coarse failure assertion lets the protection under test be silently replaced by a different layer.** "It threw" is not "it threw for the reason this test exists" — assert the specific error class/reason, or hardening anywhere upstream converts the test into a tautology without touching it. With rule 12 this brackets the tautology space from both directions: would it fail without the fix, AND does it fail for the right reason with everything else present.
21. **Constraint drift — a recorded rule does not travel with the code it binds.** Two halves: (a) refactor re-grep — when a WP moves/splits/re-owns a surface, re-check recorded constraints (graph nodes, docstrings, rulebooks) against the new tree as part of the refactor's acceptance; (b) joint satisfiability — when a NEW constraint joins ones already binding the same surface, check them in the same room (rule 15's other tense: pairwise at installation time, not just plan time). Mechanism: a refactor is to a constraint what an upgrade is to a running process (19) — a write to the thing that binds it, made by someone who can't see it. Strongest form: the catalog test — make drift mechanically impossible where drift means breach (every member of a class must declare its handling or CI fails at the change that introduces it).
22. **Before making a value honest, inventory who relies on the lie.** A return value that misdescribes its operation can still be the correct input to a downstream arbiter holding the real invariant — Chesterton's fence, return-value edition. The surviving fix until the inventory clears: a comment at the site stating what the value actually means, who the real arbiter is, and a plaque for the next reformer. The inventory is SYMMETRIC: "who relies on the lie" when making the value honest; "who relies on the honest failure" when making the lie true — both edits move the same boundary, only the direction differs.
23. **For an external API, the documentation is not the primary artifact — the live endpoint is.** Docs upgrade confidence, never truth; a doc asserting change and a config assuming stasis are equally claims, and only the live boundary referees (provenance is not truth — "newer, documented" does not outrank "older, running"). Practice: every external-API boundary ships a live smoke that runs BEFORE any doc-derived claim about that API hardens into reviewed spec, binding symmetrically.

## Margin notes (instances and corollaries, filed under their rules)

- **(23/17):** a verification instrument that doesn't touch the thing it certifies is decorative — reading what was published measures the publisher, not the endpoint.
- **(23/8):** the live smoke that caught the miss was wired as a WP requirement BEFORE it was needed and fired on its first run — criteria-written-before-need turning an incident into a correction.

- **(11) Writer inventory:** a contract enforced at one write path is a convention at every other — inventory ALL writers of a protected artifact before claiming the invariant holds; the bypass writer is usually installed by a different agent, each acting correctly.
- **(11) Reader inventory:** locks exclude writers; your ops procedures are READERS. The manager's own data-management choreography (snapshots, flushes, exports, backups) belongs in the reader inventory of any artifact the implementer is locking — make the procedure a citizen of the locking discipline it administers.
- **(11) Retain/reclaim symmetry** (vince's WP-D1a sidecar leak; Reginald/MemCog refinement): retain and reclaim are two readers of ONE identity question ("what counts as the same resource?") — and the guarding test is a third (often a fourth). Make them the same EXPRESSION: a FK/cascade, or a single shared predicate that every producer, consumer, AND the test calls. Filters that merely *match by reading* are the leak waiting to happen — a content-addressed sidecar orphaned forever because dedup counted "same-type node, same sha" while the delete-side refcount counted "any node citing the ref" (and the design's intended pattern was for foreign nodes to cite it). FKs are just the special case where the language hands you the shared expression for free; everyone else authors it, and the only question is ONCE or N times. Where the resource lives OUTSIDE the integrity boundary (filesystem sidecars, external blobs — or, like vibe-cognition, an entire store with no relational layer, where this is the *primary* discipline not the fallback), the symmetry is hand-authored and must be an EXPLICIT, CI-enforced predicate/inventory — never two filters trusted to agree by reading. Watch the FOURTH reader: if the GC/refcount test re-encodes the filter inline instead of calling the shared predicate, you've only moved "two readers trusted to agree" to prod-vs-test — a later predicate change updates the function, the test re-passes on its stale copy, and the leak is back under a green suite. Corollary with (12): a refcount test exercising only the obvious same-type twin never sees the leak — derive the test's notion of sameness from the same function the code uses, so the foreign-citer case can't be silently forgotten (it took the rule-7 decorrelated pass to construct the citer the happy path forgot). MemCog's schema form: every reclaimable resource either FK-cascades (assert the cascade exists) or appears in a CI-checked purge inventory.
- **(12) Lazy-import reachability:** a test that imports module A cannot pin module B that A loads lazily — laziness is a test-reachability boundary. Pin contracts structurally (e.g., AST import-allowlist) and exercise the real wiring separately.
- **(12) Fix exists vs proof exists:** proofs lag fixes unless acceptance criteria demand fix+proof in the SAME commit, stated up front.
- **(14) Platform-path semantics:** when a claim about code path X cites semantics from platform Y, check which platform can actually EXECUTE X — a claim can be true in a context the code never runs in.
- **(4) Compound briefs:** when a fix has two valid directions (make the value honest vs make the lie true), the brief must pick ONE and say why — or the implementer ships both halves of a contradiction, each implemented correctly.
- **(9) Specs act on the world too:** a manager's claim about caller behavior in a brief is evidence-asserted-not-gathered, exactly like a command run on stale state. Verify the branch you cite before you cite it.
