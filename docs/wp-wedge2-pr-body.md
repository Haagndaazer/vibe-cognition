## §W2-b: INV-1, spawn-free event loop
- **Dispatch seam** (`tools/dispatch.py`, new): every registered tool (all 29) now goes through `@dispatch_tool(mcp)` instead of `@mcp.tool()` — a `functools.wraps`-preserving async wrapper that routes the sync tool body to a dedicated, pre-started `concurrent.futures.ThreadPoolExecutor` (4 workers) via `run_in_executor`, instead of FastMCP's own anyio-backed `call_sync_fn_in_threadpool` (the mode-(b) mechanism). `prewarm_dispatch_executor()` runs pre-`handshake_yield` (same section as the existing anyio warm-batch) and forces all 4 threads to exist via a `threading.Barrier` rendezvous — a naive N-submissions warm-up is NOT reliable here (`ThreadPoolExecutor._adjust_thread_count` skips spawning when an idle thread is available, and a near-instant no-op can free its thread before the next submission lands); the barrier keeps every thread "busy" until all 4 have started, guaranteeing genuine concurrency. Empirically verified: prewarm creates exactly 4 distinct threads, and a 10-call storm against the 4-worker pool after prewarm causes zero further `Thread.start()` calls. Guard: `dispatch_tool` raises `TypeError` at registration time if handed an already-`async def` function — `run_in_executor` would otherwise silently return a never-awaited coroutine as the tool's "result."
- **Coverage disclosure (gate finding, MINOR)**: the WP2-AC3 storm test drives dispatch through `fastmcp.Client(mcp)`'s IN-MEMORY `FastMCPTransport` (anyio memory streams) — it proves the DISPATCH half of INV-1's zero-spawn claim only. The transport-WRITE half (every real stdio response write needing an anyio worker) has zero coverage in that harness; it stays covered ONLY by the retained warm-pool/heartbeat below, not by a zero-spawn proof.
- **Real bug found and fixed during integration**: FastMCP's `Context.request_context` reads a `contextvars.ContextVar` that anyio's `to_thread.run_sync` propagates into its worker thread automatically — a bare `ThreadPoolExecutor.submit()` does not. Every tool call failed with `RuntimeError: no request context` until `dispatch_tool` explicitly captures `contextvars.copy_context()` on the event-loop side and runs the tool body through it (`call_context.run(fn, ...)`) on the worker thread.
- **Transport cover**: WP-Wedge v1's `_worker_heartbeat`/`_warm_worker_batch` (anyio pool, 4 workers) is left UNCHANGED — tool dispatch no longer uses anyio's pool at all, so this machinery now exists solely to keep the stdio TRANSPORT's reader+writer warm (`mcp/server/stdio.py` wraps stdin/stdout via `anyio.wrap_file`, whose `AsyncFile` routes every readline/write/flush through `anyio.to_thread.run_sync`, confirmed in the pinned venv). 4 warm workers is more than the 2 the transport needs. Per the brief, removing this before independent transport coverage would be a regression — it was NOT removed.
- **Blast radius on the existing test suite**: converting all 29 tools to `async def` broke every test calling `mock_mcp.tools[name](ctx, ...)` synchronously (hundreds of call sites, two `_MockMcp` definitions — the shared one in `conftest.py` and an un-migrated duplicate in `test_xp2_routing.py`). Fixed at the ONE choke point in each: `_MockMcp.tool()` now detects a coroutine function and wraps it in a sync shim (`asyncio.run(fn(...))`) — zero changes needed to the hundreds of downstream test bodies. Verified no test calls `mock_mcp.tools[...]` from inside an already-running event loop (would break `asyncio.run`'s no-nested-loop rule).

## §W2-c: INV-2, import-free tool surface
- New AST guard (`test_wp_wedge2.py`, distinct from WP-Wedge v1's five-package heavy-chain guard): flags ANY function-body `Import`/`ImportFrom` under `src/vibe_cognition/`. Fixed the four known sites by hoisting to module level: `service_tools.py:87` (`project_registry.LoadedProjects`), `cognition_tools.py`'s `import json as _json`, `dashboard/api.py`'s `starlette.concurrency` import, `dashboard/server.py`'s `webbrowser`/`time` imports.
- Three whole-file exclusions (documented, same style as the heavy-chain guard's own exclusion): `embeddings/generator.py` (the sanctioned bg-thread-only backend-construction site, shared with the heavy-chain guard), `dashboard/cli.py` (standalone console-script entry point, never imported by the server), `instructions.py` (`main()`'s lazy imports are for the SessionStart compact-hook entry point, a separate process invocation).
- One per-line exception (`wp2-import-free: sanctioned` marker, mirrors ruff's `noqa`): `server.py`'s shutdown-path `from .dashboard.server import stop_dashboard`. Hoisting this ONE to module level creates a genuine circular import (discovered when the guard's own wiring test failed on it) — it's provably safe as a function-body import since `dashboard_tool.py`'s own top-level import already fully loads that module during normal startup, well before shutdown code runs.
- Pre-exercise: `lifespan()` now calls `count_documents()` (bare + `is_chunk`-filtered — the same two calls `get_status` already makes in production) against the opened home collection before yielding, exercising chromadb's own conditional-import paths the four known sites couldn't cover by grep alone. Best-effort + budgeted (<200ms, logged if exceeded); a failure never blocks startup.

## §W2-d: watchdog margin
`_WATCHDOG_TIMEOUT` 120s → 300s (2.5x the observed healthy max of 119.7s). New `Settings.wedge_watchdog_timeout` field, env-overridable via `WEDGE_WATCHDOG_TIMEOUT`, wired into `lifespan()`'s `_watchdog(...)` call. Late-recovery semantics unchanged.

## §W2-e: breadcrumb fidelity
- `stamp_and_flush()` (new primitive) replaces bare `stamp()` at every bg-thread-only call site in `_run_subprocess_import_probe` and `_load_embeddings_and_sync` — a mid-wedge breadcrumb file now reflects every stamp made so far, not just the last checkpoint (Incident B's exact defect: the file ended at `handshake_yield` even though the stack proved later stamps had already happened in memory). Loop-side (`watchdog_fired`) and pre-yield-path stamps stay stderr-only, unchanged.
- `tool_served_degraded`: one stamp, first occurrence per process, fired from `get_lifespan()` (the one choke point every tool calls at entry) when `embedding_error` or `watchdog_fired` is set — distinguishes "degraded but serving" from "hung" in fleet logs. Never flushes inline (worker-thread-context safe, same as `stamp()`).

## §W2-f: dispatch-stall self-forensics (replaces WP2-AC2 per the §W2-a negative)
- New FastMCP middleware (`_DispatchStallForensics`, first-party public `on_call_tool` hook, no monkeypatching): races each tool call against a per-call timeout (default 30s, env-overridable via `DISPATCH_STALL_THRESHOLD`) ONLY during the load window or a degraded state (zero overhead on healthy serving). On a stall, dumps every live thread's stack to stderr (`sys._current_frames()` + `traceback`, not `faulthandler.dump_traceback` — the latter requires a real OS file descriptor, which a captured/wrapped stream doesn't always have) — once per process, never touching disk. The in-flight call is NEVER cancelled; the real result is still returned once it finishes. This converts the next real production mode-(a) occurrence into a pinned stack in the client's MCP logs.
- The §W2-a repro script (`tests/wp2_mode_a_forensics.py`) is wired into the suite as regression coverage for the plain-import-lock CLASS — passes on current main for the four cleared sites. Its docstring is explicit that it does NOT claim to reproduce Incident A.

## Honesty bar (per brief §6)
This WP bounds DISPATCH during a wedge. The embedding backend itself remains unavailable for the wedge's full duration (24+ min observed) — degraded responses are the ceiling until the sidecar embedding process epic lands. A fixed 4-worker dispatch pool bounds concurrency at 4 simultaneous in-flight calls; a storm beyond 4 queues (bounded latency), it does not scale. No "fixed the hang" language: mode (a)'s exact site stays unpinned; §W2-f gives production a way to pin the next occurrence itself.
