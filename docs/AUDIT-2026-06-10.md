## 1. Core cognition (storage, journal, concurrency)

**Provenance:** the lazy byte-offset catch-up under the in-process lock was a deliberate v0.7.0 decision (graph node `7330e0252c8a`), explicitly chosen over a background watcher or sqlite. The torn-trailing-line handling came out of peer review (pattern `62c1782b4fcd`). **No node anywhere considers inter-process append atomicity or file locking** вЂ” findings C-1..C-4 are blind spots *within* the chosen design, not decided trade-offs, and all fixes below stay inside it.

### C-1. No cross-process append atomicity вЂ” interleaved large writes corrupt the journal В· **bug, HIGH** В· verified
`storage.py:607` appends via a buffered `TextIOWrapper`:
```python
with open(self._journal_path, "a", encoding="utf-8") as f:
    f.write(line + "\n")
```
A line larger than the ~8 KiB buffer (episodes вЂ” "verbose is fine" per the tool docstring) flushes as multiple OS writes. With two processes appending concurrently (two sessions, or session + post-commit hook), chunks can interleave; `_catch_up` then skips both mangled lines as malformed (`storage.py:663`) вЂ” permanent silent data loss. There is no `fcntl`/`msvcrt` lock anywhere (grep-verified). *Fix direction:* single `os.write()` on an `O_APPEND` fd, optionally plus a byte-range lock around the append. Untested scenario (concurrency tests are single-process only вЂ” see T-1).

### C-2. A crashed writer's torn tail eats the next writer's valid entry В· **bug, MED** В· verified
The offset correctly parks before a torn tail (`storage.py:648-650`), but a *different* process's subsequent append concatenates onto the torn bytes; everything up to the next `\n` parses as one malformed line and is skipped вЂ” the new, valid entry is lost for all readers.

### C-3. Shrink-only replacement detection В· **bug, MED** В· verified
`storage.py:637` `if size < self._offset:` triggers a full rebuild only when the journal *shrinks*. A journal replaced with equal-or-larger content (git pull/merge of a committed `.cognition/journal.jsonl` вЂ” which the design intends to support: `server.py:49` "teammate pulled new JSONL entries") resumes replay from a stale byte offset into unrelated content: first partial line skipped, then entries double-applied or replayed after their tombstones (deleted nodes resurrect). *Fix direction:* store a cheap identity check (first-line hash / file ID) next to the offset.

### C-4. Mutate-then-journal with no rollback В· **bug, MED** В· verified
All write ops mutate the in-memory graph **before** appending (e.g. `add_node`, `storage.py:96-108`). If the append raises (disk full; Windows AV transiently locking the file), the in-memory graph keeps a phantom write no other process вЂ” nor this one after restart вЂ” will ever see.

### C-5. Tool-level TOCTOU + ignored `add_edge` return В· **bug, LOW** В· verified
Each tool call makes several independently-synced storage calls; a concurrent delete between them makes `storage.add_edge` decline (returns `False`), but `cognition_tools.py:364` ignores the return and reports `{"created": True}`. Same in the batch tool (`:460`, `created += 1` regardless).

### C-6. Self-replay log noise В· **inconsistency, LOW** В· verified
`_append_journal` doesn't advance `_offset`, so every process re-reads its own appends and logs "caught up: +1 entries" (`storage.py:666-671`). Benign (idempotent replay is what guarantees journal-order convergence) but misleading when diagnosing convergence вЂ” worth a comment or advancing the offset on self-writes.

### C-7. `get_reasoning_chain` mislabels diamonds as cycles В· **bug, LOW** В· verified
`queries.py:29,47`: `visited` is global to the traversal and never popped, so Aв†’Bв†’D, Aв†’Cв†’D marks the second D `"cycle": true`. Only linear chains are tested.

**Done right (keep):** binary-mode byte-exact offset reads (CRLF-safe), torn-tail offset parking, shrink в†’ full rebuild incl. reference index, idempotent replay of every entry type, single-tombstone delete with replay-side edge re-cascade.

## 2. Embeddings & search

**Provenance:** `embeddings/` carries visible code-search heritage (default collection name `"code_embeddings"`, `repo`/`file_path`/`content_hash` metadata) predating the cognition graph вЂ” no graph nodes cover its design. The graph/Chroma drift was *named* in the v0.7.0 discovery (`55b6740e42f8` mentions ChromaDB) but the shipped fix deliberately covered only the graph вЂ” so E-2 extends a known-deferred item rather than re-opening a settled one.

### E-1. ChromaDB telemetry is ON В· **bug, MED-HIGH** В· verified
`embeddings/storage.py:28`: `chromadb.PersistentClient(path=str(persist_directory))` вЂ” no `Settings(anonymized_telemetry=False)`, so PostHog telemetry phones home from every server start in every user's project. One-line fix; also removes background network chatter from the MCP process.

### E-2. Structural nodeв†”embedding drift (the live 37 vs 35) В· **gap, MED-HIGH** В· verified
Embeddings are written in exactly one place at record time and backfilled exactly once at startup (`server.py:128-135`). Paths that create nodes-without-embeddings:
1. **`post-commit.py` writes journal lines directly** вЂ” replay (`storage.py:719-777`) never embeds. Most likely cause of today's gap (2 commits since last restart = 2 missing).
2. **Model-loading window / embedding_error**: `cognition_tools.py:64-65` silently skips embedding, and the tool result doesn't say so.
3. **Embedding failure after journal append**: `cognition_tools.py:67-79` runs bare вЂ” an exception leaves the node journaled but unembedded, and a retry creates a duplicate node (new timestamp в†’ new ID). The delete path treats Chroma as best-effort (`operations.py:58-61`); the create path should too.
4. **Cross-process replay never embeds** вЂ” convergence assumes the writer embedded; see 1 and 2.
5. **Reverse drift**: replayed `remove_node` tombstones never purge Chroma in other processes в†’ orphan embeddings surface in search as dead node IDs.
*Repair lever today is only a restart.* Cheapest fixes: re-run the missing-embedding sync from `cognition_reload` (its docstring doesn't mention embeddings aren't resynced), and/or opportunistically embed after `_catch_up` ingests `add_node` entries.

### E-3. Documents embedded with the query prefix В· **bug, MED** В· verified
Both write paths call `generate_query_embedding` for *documents* (`cognition_tools.py:67`, `server.py:80`), while `generator.py:37-38` defines asymmetric nomic prefixes and `generate(text, input_type="document")` exists unused. Self-consistent (search works) but discards nomic-embed-text-v1.5's asymmetric-retrieval training в†’ degraded ranking. Fixing requires a one-time collection re-embed (vector spaces incompatible).

### E-4. Concurrent `PersistentClient`s on one ChromaDB dir В· **gap, MED** В· inferred
Both sessions open the same `.cognition/chromadb` (`server.py:165-168`); Chroma's PersistentClient is not multi-process-safe вЂ” sqlite lock errors (feeding E-2.3) and stale per-process segment caches.

### E-5. Backend/model mismatch is undetectable В· **inconsistency, LOW-MED** В· verified
`OllamaBackend.encode` ignores `is_query` and applies no prefixes (`generator.py:98-116`) though the default ollama model is also nomic; switching backend/model against an existing chromadb mixes incompatible embedding spaces вЂ” nothing records model/dimensions in the collection to detect it.

### E-6. Code-search heritage dead code В· **dead-code, MED** В· verified, zero callers
`embeddings/storage.py`: `bulk_upsert` (:55), `delete_by_file` (:131), `delete_by_repo` (:152), `get_by_id` (:169), `get_content_hashes` (:192); `vector_search`'s `repo`/`file_path_prefix` params (:218-220) filter on metadata that is never written (passing `file_path_prefix` would filter out *everything*); default `collection_name="code_embeddings"` (:19) unused. Either prune, or put `bulk_upsert`+`generate_batch` to work (see E-8).

### E-7. Misc В· LOW В· verified
- `embeddings/storage.py:99` `datetime.utcnow()` вЂ” deprecated, naive timestamp; everywhere else uses `datetime.now(timezone.utc)`.
- Error-shape mix: `vector_search` swallows all exceptions в†’ `[]` (outage looks like "no results", `:257-259`); `delete_embedding` swallows в†’ `False` (and returns `True` even when nothing was deleted); `upsert`/`get_*` raise.
- `trust_remote_code=True` with no pinned `revision=` (`generator.py:49`) вЂ” unpinned remote code from HF hub at model load; **this is also why `einops` must stay a dependency** (nomic's remote code imports it) вЂ” deserves a pyproject comment or it will look removable forever.
- Dimension truncation skips nomic's recommended layer-norm (`generator.py:76-77`) вЂ” harmless at default 768.

### E-8. Slow startup sync В· **improvement, LOW** В· verified
`_sync_cognition_embeddings` embeds one node per loop (`server.py:78-92`) while `generate_batch` and `bulk_upsert` both exist, both dead. Also reaches into `embedding_storage._collection` (private; `server.py:66`) and its `except Exception: pass` (`:68-69`) makes a real Chroma failure silently re-embed everything.

## 3. MCP tools & server

**Provenance:** node deletion (v0.7.1, `6c8337d42c79`) and the FastMCP instructions baking (`b840621ce2d2`) are settled; no graph history covers the tool-layer contracts below вЂ” genuine gaps, not residue.

### T-1. Zero tool-level tests В· **gap, MED** В· verified
Nothing exercises any of the 15 `@mcp.tool()` closures вЂ” validation, error envelopes, return shapes, the batch JSON parsing, `require_embeddings`. Every bug in this section lives in that untested layer.

### T-2. `total_uncurated` silently caps at 500 В· **bug, MED** В· verified
`cognition_tools.py:527` asks storage for `limit=999999` but `storage.py:382` hard-caps `uncurated[:min(limit, 500)]` вЂ” the reported backlog count can never exceed 500.

### T-3. `add_edges_batch` crashes mid-batch on a non-dict element В· **bug, MED** В· verified
`cognition_tools.py:413-417`: a string/number/null in the JSON array raises `AttributeError` on `.get` **after earlier edges were already journaled** вЂ” partial commit + exception, while every other malformed input is skipped-and-reported. Needs an `isinstance(e, dict)` guard.

### T-4. Curation `reason` is requested, then discarded В· **gap, MED** В· verified (two agents independently)
`edge-analyzer.md:36` instructs subagents to produce a reason per edge; `cognition_add_edge` only logs it (`cognition_tools.py:366-367`), `add_edges_batch` drops it without error (`:414-417`), and `CognitionEdge` (`models.py:49-59`) has no field for it. The rationale `/vibe-curate` works to produce is never persisted. Also: the skill says pass `source: "curate-skill"` at the top level (`vibe-curate/SKILL.md:44`) but the tool reads `source` per-edge-object вЂ” literal compliance tags every curated edge `"batch"`.

### T-5. No `cognition_get_node`, no node UPDATE tool В· **gap, MED** В· verified
Search results omit `detail` (not in Chroma metadata) and `get_neighbors` returns summaries only вЂ” after a search hit there is no direct way to read a node's full narrative. `storage.get_node` exists (`storage.py:282`). Likewise `storage.update_node` is implemented, journaled, replayed, and tested (`storage.py:142,771`; `test_cognition.py:160`) but unexposed вЂ” fixing a typo requires delete+re-record, losing ID, edges, and curation state. Caveat for any future update tool: **nothing re-embeds after `update_node`** вЂ” search would serve the stale summary forever (pairs with E-2).

### T-6. Error-contract split across tools В· **inconsistency, MED** В· verified
Three behaviors coexist: error-dicts (most tools), raised exceptions (`get_uncurated_nodes` on bad `node_type` вЂ” `cognition_tools.py:523` bare enum call vs `get_history`'s handled version at `:288-293`; `cognition_dashboard` with no try/except вЂ” `dashboard_tool.py:40`), and silent wrong answers (`get_chain` treats unknown `direction` as incoming, `queries.py:52-57`; nonexistent node yields a fake `{"type": "unknown"}` stub instead of the error-dict `get_neighbors` returns; `get_edgeless_nodes` with invalid type returns `[]`). `node_type` is validated three different ways вЂ” one shared `_parse_node_type` helper fixes most of this.

### T-7. Documented-but-unenforced input contracts В· **gap, LOW-MED** В· verified (two agents)
"summary: MAX 250 chars" appears in the docstring, SKILL.md, and cluster-analyzer.md, but `models.py:41` is bare `summary: str` and `_record_node` validates nothing вЂ” 5,000-char summaries and `severity="banana"` are accepted and journaled (the post-commit hook *does* truncate to 250, `post-commit.py:86` вЂ” the two writers disagree). `limit` params accept 0/negative. Either enforce (`Field(max_length=250)`, episodes exempt; severity enum) or re-label as guidelines.

### T-8. Search can't see other sessions' nodes until restart В· **gap, MED** В· verified
Graph tools converge per-op; `cognition_search` queries Chroma directly (deliberate) but `_sync_cognition_embeddings` runs once at startup. Combined with E-2 this means cross-session search staleness with no in-session repair lever.

### T-9. Pyright `lifespan_context` noise вЂ” root cause + clean fix В· **improvement, MED** В· verified
fastmcp types `Context.request_context` as `... | None` and its own docs say to None-check. Every tool does `ctx.request_context.lifespan_context[...]` un-narrowed (~22 sites). One accessor in `tools/utils.py` (`raise RuntimeError if rc is None; return rc.lifespan_context`) routed through all sites kills all 22 pyright errors and dedupes the pattern.

### T-10. Server startup/shutdown niggles В· **improvement, LOW** В· verified
- Deterministic-edge backfill is skipped if the embedding model fails to load вЂ” it's inside the same `try` (`server.py:107-140`) though it needs no embeddings; reorder.
- `setup_logging` runs after `Settings()` (`server.py:147-153`); `basicConfig` is a no-op if any import installed handlers вЂ” `log_level` may silently not apply.
- `.env` resolves against CWD = plugin root under `uv run --directory` (`config.py:27`) вЂ” a project `.env` is never read; a stray plugin-cache `.env` would be.
- Stale comment `server.py:146` still says REPO_PATH is "set by per-project .mcp.json".
- `__init__.py:3` `__version__ = "0.1.0"` vs 0.7.1 вЂ” stale and unread by anything; wire to `importlib.metadata` or delete.
- `bg_thread.join(timeout=5.0)` can abandon a mid-sync embedding backlog (harmless; next start re-syncs).
- `get_status`'s `cognition_embeddings` is an int on success, dict on failure (`service_tools.py:46-50`).
- `get_edgeless_nodes` / startup edge-backfill are O(N) lock round-trips (`_synced` per neighbor call); one storage-level pass under one lock would be atomic and faster.

### T-11. Dead from the product surface В· **dead-code, LOW** В· verified
- `queries.get_superseded_chain` / `get_incident_resolution` вЂ” exported, tested, called by nothing (and `get_incident_resolution` contains an if/else with identical branches, `queries.py:178-182`). Note the remove-node tool itself recommends `supersedes` chains that no tool can then traverse вЂ” surface or prune.
- `storage.redirect_edges` (`storage.py:228`) + `CognitionEdgeType.DUPLICATE_OF` (blocked by both edge tools) вЂ” merge-logic residue of the removed curator. **Provenance:** curator removal (`e8861a18195e`) deliberately kept journal-compat (`source="curator"` default is documented-deliberate); the merge pair is unowned residue вЂ” prune, or build the dedupe/merge tool it implies.
- `EmbeddingGenerator.generate_batch` вЂ” zero callers (see E-8 for its natural home).

## 4. Dashboard

**Provenance:** no graph history covers dashboard design choices вЂ” initial-build era. Security posture verified good: loopback-only binds, token auth + host-header check (DNS-rebinding-proof), every `innerHTML` sink escaped, delete CSRF-protected by custom header. Findings are correctness/staleness, not exposure.

### D-1. `start_dashboard` never verifies the server started В· **bug, MED** В· verified
`server.py:164-174` starts uvicorn in a daemon thread and stores state immediately; `:139-146` short-circuits all later calls with `already_running`. If bind fails (incl. the probeв†’bind TOCTOU), the thread dies silently and the tool reports a dead URL for the rest of the session. Check `server.started`/`thread.is_alive()`.

### D-2. `port=0` returns a broken URL В· **bug, MED** В· verified
`_find_free_port` (`server.py:75-79`) returns the literal `0` instead of `getsockname()[1]` on its preferred-port path (the ephemeral fallback at `:82-84` does it right). `port` is reachable from the MCP tool. The idempotency test only compares the two returned URLs to each other.

### D-3. UI staleness В· **gap, MED** В· verified
`/api/graph` is fetched exactly once at page load (`app.js:451`) вЂ” no refresh button, no polling; the primary use case (dashboard open during an active session) goes stale immediately. Related smaller items: `--no-embeddings` leaves the UI in "Loading embedding modelвЂ¦" forever (no `disabled` state, `cli.py:36-57` / `api.py:23-31`); search wiring only attaches if the initial graph fetch succeeds (`app.js:451-468`); a concurrently-deleted episode stays in the sidebar (`app.js:166-169` misses `removeEpisodeFromList`).

### D-4. CDN-loaded cytoscape with no SRI В· **improvement, MED** В· verified
`index.html:8-11` loads cytoscape/fcose from jsdelivr with no `integrity`/`crossorigin` вЂ” a CDN compromise runs arbitrary JS in a page that can DELETE nodes; also breaks the dashboard offline. Vendoring into `static/` (already wheel-included) fixes both.

### D-5. Minor В· LOW В· verified
Search endpoint 500s on malformed bodies instead of JSON 400s (`api.py:126-128`, no limit clamp); host check requires a port suffix and rejects `[::1]` (`middleware.py:28`); token compare isn't `secrets.compare_digest` (`middleware.py:34,37`); delete response discards the helper's `edges_removed` detail (`api.py:114`); duplicate `type`/`edge_type` keys in neighbor payloads (`api.py:87-93`); unused `context`/`severity` shipped in the graph payload against its own "keep payloads small" docstring; `stop_dashboard` not exported from `dashboard/__init__.py` while `start` is; `stop_dashboard` closes the static-dir `ExitStack` even on join timeout (`server.py:196-198`); default port 7842 hardcoded in three places; `webbrowser.open` inside the MCP process could theoretically leak child stdout to the stdio channel on Linux (`server.py:179`); unused `Response` imports (ruff-confirmed).

## 5. Hooks, migration, distribution

**Provenance:** the plugin-declared MCP + venv relocation is settled v0.6.0 work (`0ef59e508d6d`), with a recorded constraint that install mechanics can't be self-verified (`b8ec24fe9107`) вЂ” H-3/H-4 below are flagged for the next human-gated release test rather than as code-only fixes. The post-commit hook predates the graph; its SHA-format scare was investigated and closed as a false alarm (`352a79a64406` вЂ” matcher prefix-normalizes), but its *wiring and duplication* were never examined.

### H-1. `post-commit.py` runs on bare `python` В· **bug, HIGH** В· verified wiring, inferred impact
`hooks/hooks.json:31`: `"command": "python \"${CLAUDE_PLUGIN_ROOT}/hooks/post-commit.py\""`. The plugin only guarantees `uv` exists; macOS has no `python` by default, Windows often only `py`. Where missing, this PostToolUse hook fails silently on **every Bash call**. It also bypasses the synced venv (stdlib-only today, so it needs a *system* Python в‰Ґ3.10). Route through `uv run --no-sync` like the other hooks.

### H-2. `post-commit.py` forks the journal-writing logic В· **gap, MED** В· verified
It re-implements `generate_node_id` and the journal line format by hand (`post-commit.py:24-27,83-93`), byte-matching `models.py`/`storage._replay_entry` today вЂ” but nothing pins that contract, it never embeds (E-2.1), and it has zero tests. Trigger heuristics are loose: matcher is `"Bash"` only (misses PowerShell commits вЂ” relevant on this very machine), fires on any command *containing* `git commit`, and doesn't check command success (idempotency scan mostly compensates). Stale docstring shows a pre-plugin install path (`post-commit.py:6-12`).

### H-3. session-start.sh: silent `uv sync` failure В· **bug, MED** В· verified
`session-start.sh:50` under `set -euo pipefail` with `2>/dev/null` and no `||` guard вЂ” a sync failure (network, lock) kills the hook with zero diagnostics and no fallback `{}`, unlike every other failure point in the script (all guarded). Related: prime/reinject failures discard stderr to `/dev/null` вЂ” the user gets no context with no clue why; a breadcrumb log under `.cognition/` or plugin-data would make these diagnosable.

### H-4. First-install race: server `uv run` and hook `uv sync` race on the same venv В· **gap, MED** В· inferred from verified wiring
`plugin.json` launches with `uv run` (no `--no-sync`) so a fresh install self-syncs вЂ” but the dep set includes torch (multi-GB); the MCP startup timeout will likely kill the first launch while session-start.sh concurrently syncs the same env. Also `plugin.json:22` hard-codes `${CLAUDE_PLUGIN_DATA}/.venv` with no fallback while the hooks defensively fall back вЂ” and their fallback `${PLUGIN_ROOT%/*}` breaks on backslash paths (no `/` to strip в†’ venv lands back inside the version-pinned cache dir, resurrecting the Windows EPERM the relocation was built to fix). Gate on the recorded constraint: verify on a human machine at next release.

### H-5. Both SessionStart entries fire on compact В· **inconsistency, LOW** В· verified
`hooks.json:4-24`: the first entry has no matcher (matches `compact` too), the second is `matcher: "compact"` вЂ” post-compact runs the full session-start (uv stamp check, migrate_mcp, prime) *plus* reinject. Re-priming may be intentional (**provenance:** the reinject decision `7e9665028bb1` says only that instructions may not survive compaction); re-running migration is waste. Either scope the first matcher or comment the intent.

### H-6. Hygiene В· verified
- **No LICENSE file** (MIT declared in `pyproject.toml:6` + `plugin.json:6`) вЂ” real distribution gap for a marketplace plugin. **MED**
- **No CI** (no `.github/`) вЂ” 129 tests exist; the release procedure is push-and-pin with nothing running them. **MED**
- `httpx` is a dead direct dependency вЂ” zero imports repo-wide (grep-verified; satisfied transitively by fastmcp/chromadb anyway). Likely auto-curator-era residue. **LOW**
- Dev deps duplicated: `[project.optional-dependencies].dev` в‰Ў `[dependency-groups].dev` (`pyproject.toml:23-30` vs `:48-55`); `uv sync` only reads the group вЂ” drop the extras copy. **LOW**
- `config.py:103` docstring claims the chromadb path is "gitignored" вЂ” `.gitignore` has no `.cognition` entry at all; `config.py:98` calls `.cognition` "Git-committed" (and `server.py:49` assumes teammates pull the journal). The intent (commit journal, ignore chromadb) is half-implemented: **`.cognition/chromadb/` ignore line is missing**, and `.cognition/` sits untracked. Decide and align docstring + .gitignore + README. **MED (decision needed вЂ” yours)**
