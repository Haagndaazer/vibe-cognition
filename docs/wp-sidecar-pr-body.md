WP-Sidecar (P0 endgame, docs/wp-sidecar-plan.md rev 3): the heavy torch/scipy/sentence_transformers import leaves the server process entirely, moving into a supervised child subprocess. This is the WP that actually ends the wedge — everything since v0.15.3 bounded its symptoms (dispatch survives, orphans die, watchdogs fire); this removes the wedge *source* from the process that matters.

## §S-a: sidecar entry module + named-mutex load serialization
New `src/vibe_cognition/embeddings/sidecar.py` — THE sanctioned heavy-import site going forward, run via `python -m vibe_cognition.embeddings.sidecar` (spawned directly by the server via `subprocess.Popen`, no uv/shell intermediary needed since `sys.executable` inside the already-running server process IS the resolved venv interpreter). `SentenceTransformersBackend` moved here verbatim from `generator.py`. Single-threaded request/response loop over stdin/stdout JSON lines (`_sidecar_protocol.py`: `load`/`generate`/`ping` + unsolicited `lock_wait`/`lock_acquired`/`lock_acquired_abandoned`/`lock_wait_expired` events — the supervisor's load-timeout clock starts at whichever lock event arrives, never at request submission, so queueing behind another session's load never counts as wedged). Arms WP-Lifecycle's own `lifecycle.arm_ancestor_watch(depth=1)` — direct reuse, no intermediary, exactly the parameterization that WP built for this.

`_load_mutex.py`: `Local\vibe-cognition-model-load` named mutex (ctypes, no new dependency) serializing the heavy import + model load machine-wide. `WAIT_ABANDONED` is treated as successful acquisition (breadcrumbed `sidecar_mutex_acquired_abandoned`, distinct from plain `sidecar_mutex_acquired`) — a supervisor killing a wedged holder mid-load is designed behavior, not an edge case. Acquire-timeout expiry proceeds without the lock (breadcrumbed `sidecar_mutex_wait_expired`) — stampede risk beats never loading. Empirically validated (not just theorized) against real cross-process contention and a real kill-while-holding abandonment, before wiring into production code — same discipline as WP-Lifecycle's mutex/handle work.

## §S-b: server-side sidecar client + supervisor
New `sidecar_client.py`: `_SidecarProcess` (spawn, dedicated reader + writer drain threads, bounded outbound queue, request/response correlation, event dispatch) + `SidecarSupervisor` (in-budget kill+respawn+backoff, degrade, lazy-on-demand + slow-periodic recovery) + `SidecarBackend` (the `EmbeddingBackend` proxy `EmbeddingGenerator.from_config` constructs for the non-ollama case). IPC non-negotiables from the brief, all present: dedicated reader AND writer threads; a bounded outbound queue (`maxsize=16`) so a caller thread's own `queue.put` timeout — not a raw blocking pipe write — is what it depends on; round-trip timeout owned by the supervisor (`generate()`'s `except SidecarError: proc.kill()`), never the requesting thread; `kill()` (probe-style kill+wait, WP-Wedge's now-subsumed pattern) demonstrably closes handles and unblocks a stuck write (WPS-AC5, verified for real). stdin/stdout are PIPEs — the sanctioned exception to the DEVNULL-only subprocess rule, *because of* these dedicated drain threads (documented at the spawn site); stderr stays DEVNULL.

`EmbeddingGenerator.from_config`'s non-ollama branch now constructs `SidecarBackend(config)`, which blocks (bounded by the retry budget) via a **standalone** module-level supervisor singleton — used by `dashboard/cli.py`'s dev tool and any future direct caller. The **real MCP server never calls `from_config` for this branch**: `lifespan()` constructs a **context-attached** `SidecarSupervisor(config, context)` directly, and `_load_embeddings_and_sync` drives it via `ensure_ready()` — this is the sole writer of `embedding_generator`/`embedding_error`/`embedding_ready`, replacing WP-Wedge's `_wedge_lock` atomicity discipline (two racing writers, bg thread + watchdog) with a single owner (no watchdog exists anymore to race). `ensure_ready()` always assigns `embedding_generator` **before** setting `embedding_ready` — by construction, not by a lock, since there is no second writer left to interleave with.

Recovery has two independent legs, both required by the brief: a dedicated `_recovery_loop` thread runs for the server's whole lifetime, waking either on its own slow periodic schedule (`sidecar_periodic_retry_interval`, default 300s) or immediately via `notify_demand()` (an actual embedding request arriving while degraded) — either path retrying a fresh spawn+load and, on success, clearing `embedding_error`/installing a working generator. Production evidence says wedged loads eventually complete; this makes recovery never permanent, matching (and generalizing) WP-Wedge's late-recovery guarantee.

**Real bug found and fixed during integration** (caught by my own end-to-end testing, not the mocked unit tests): `_SidecarProcess.__init__`'s `subprocess.Popen` call was missing `text=True` — every write to the child's binary stdin stream raised `TypeError`, silently swallowed by the writer thread's `except Exception: return`, so *nothing* ever reached the sidecar and every caller hung for its full timeout with no indication why. Fixed, and the writer's exception handler now logs unexpected failures (any exception other than a clean broken pipe) to stderr instead of swallowing them silently — the exact silence that hid this bug from me for longer than it should have.

## §S-c: AST guard flip + runtime sys.modules assertion
`test_wp_wedge.py`'s heavy-chain AST guard flips its one sanctioned site from `generator.py` to `sidecar.py` — the server process may no longer import torch/scipy/sentence_transformers/transformers/sklearn *anywhere*. `generator.py` no longer defines or imports `SentenceTransformersBackend` at all (moved wholesale); it keeps only `OllamaBackend`'s lazy `import ollama` (unrelated, untouched). The `EmbeddingBackend` ABC moved to a new leaf module (`_backend.py`) so `generator.py` and `sidecar_client.py` don't need to import each other at module level for it — a genuinely circular dependency existed for `SidecarBackend`/`EmbeddingGenerator` specifically (documented, `wp2-import-free: sanctioned`-marked, same precedent as server.py's existing dashboard-shutdown circular import). WP-Wedge-2's *separate* import-free guard (INV-2, bans ANY function-body import outside three files) gained a fourth whole-file exclusion for `sidecar.py` — it runs in a separate subprocess, never imported by the server, so INV-2 doesn't even apply to it structurally.

New `_heavy_import_guard.py`: a runtime companion to the static AST guard — checks `sys.modules` for the heavy chain and logs loudly (never raises; the cost is already paid if it ever fires) at two moments: the MCP handshake (`lifespan()`, right before yield) and the first tool call (`get_lifespan()`'s existing choke point, first-occurrence-gated). Catches what the AST guard structurally can't: a future third-party dependency transitively pulling the chain in behind our backs.

## §S-d: embedding_status UX parity
`get_status`'s `embedding_status` now defers to the sidecar supervisor's own `.status()` (`spawning`/`loading`/`waiting-for-load-lock`) while not yet ready, for the non-ollama backend; ollama (no supervisor) keeps its unchanged, simpler `"loading"`. `syncing`/`ready`/`error: ...` states are unchanged. `get_status` itself stays instant (a supervisor status check is a few attribute reads, no IPC).

## Probe/watchdog removal (subsumed)
`_run_subprocess_import_probe` and `_watchdog` (plus `_IMPORT_PROBE_*`/`_WATCHDOG_*` constants and the `wedge_watchdog_timeout` Settings field) are deleted entirely — the sidecar subprocess IS the probe now, and the supervisor's own kill/respawn/degrade state machine replaces the watchdog's fixed-timeout polling with something that actually *knows* the sidecar's real state.

WP-Wedge's five AC2 watchdog/late-recovery tests are removed from `test_wp_wedge.py`, replaced by `test_wp_sidecar.py`'s supervisor-equivalent coverage (mapping documented in that file's own module docstring, WP2-AC6 precedent): kill-mid-request-then-respawn, wedged-load-degrade-after-budget, lazy-recovery-clears-error. Two of the five have no direct replacement because their premise is now structurally impossible rather than merely guarded against: the clobber-guard and stranding-interleaving races both existed *because* a second writer (the watchdog) could race the bg thread; with the watchdog gone, there is only one writer, full stop.

`tests/wp2_mode_a_forensics.py` (WP-Wedge-2's §W2-a repro, the plain-import-lock-class guard) is **deleted**, and its wrapper test with it — its entire premise (a function-body import colliding with a bg-thread import lock, *in this process*) is now structurally impossible, since the heavy chain is never imported in the server process at all. The underlying regression class it guarded stays covered by the (unaffected) import-free AST guard.

`test_ac1_every_tool_returns_within_bound_in_the_watchdog_fired_state` (test_wp_wedge2.py) is replaced by `..._in_the_sidecar_degraded_state`, driving the new degrade mechanism (monkeypatched `SidecarSupervisor._attempt_load`) instead of the deleted watchdog.

## Honesty bar (§S-e, per the brief)
1. **Per-server sidecars still cost one torch heap per session.** This is per-server, not per-machine — N concurrent sessions still mean N sidecar processes, N torch heaps, even though only ONE loads its model at a time (the mutex bounds the *load*, not the memory footprint). A shared machine-wide daemon (one embedding process serving every session) is the real fix for that cost, deliberately deferred as a follow-up epic. Nothing here precludes it: the protocol (`_sidecar_protocol.py`) is already framing-only, transport-agnostic — a future daemon variant could reuse it over a different transport (a named pipe or localhost socket) without redesigning the wire shape.
2. **Serialization bounds the wedge, not time-to-ready.** A session queued behind N other sessions' loads still waits roughly N × healthy-load-time, surfaced honestly via `waiting-for-load-lock` rather than a falsely-instant status. Nothing in this architecture bounds time-to-ready under fleet-wide load — that is the shared daemon's job, not this WP's. No "embeddings become promptly available" language appears anywhere in this WP's code or comments.
3. **`dashboard/cli.py`'s standalone path is untested in this WP** (its own dev-only entry point, `EmbeddingGenerator.from_config` → the standalone module-level supervisor). It is structurally identical to the WPS-AC2 test's code path (same `from_config` → `SidecarBackend` → real subprocess flow), just without a request-scoped context to attach recovery updates to — flagging the gap rather than silently assuming coverage.
4. **The grandparent-degraded-arm path** WP-Lifecycle already flagged as unit-tested-only (not a real four-tier topology) applies identically here via reuse — no new gap introduced, restating for completeness since this WP reuses that exact mechanism.

## Gate fix round (93a447a → this commit)
1. **BLOCKER 1 — `notify_demand()` was dead code**: the plan's "lazy respawn on next embedding demand" leg didn't exist — only the 300s periodic tick did. Wired into `get_lifespan()` (`tools/utils.py`), the universal per-tool-call choke point every tool already passes through: when `embedding_error` is set, it now pokes `lc["_sidecar_supervisor"].notify_demand()` (guarded for the ollama backend, which never sets that key) right alongside the existing `tool_served_degraded` breadcrumb. `require_embeddings()` calls `get_lifespan()` first, so it gets the same poke for free — no second call site needed despite `notify_demand()`'s own docstring naming both. New tests: `test_get_lifespan_pokes_sidecar_supervisor_notify_demand_when_degraded`, `..._does_not_poke_notify_demand_when_healthy`, `..._tolerates_missing_sidecar_supervisor_key` (test_tool_utils.py).
2. **BLOCKER 2 — `generate()`'s live-failure path didn't write `embedding_error`**: confirmed via the reviewer's repro and an independent read — `_degraded` flipped True but `context["embedding_error"]` stayed whatever it was before (often `None`), so `get_status()` kept reporting `"ready"` and `require_embeddings()` never gated, letting every subsequent embedding call raise `SidecarError` raw through the tool body. This is the post-ready flavor of the lying-status-tuple class `_wedge_lock` existed to prevent. Fixed: `generate()`'s `except SidecarError as e` now writes `self._context["embedding_error"] = str(e)` under `_state_lock` in the same block that flips `_degraded`. `test_sidecar_killed_mid_request_returns_error_then_respawn_recovers` now asserts `context["embedding_error"] is not None` and `"error:" in supervisor.status()` after the live kill, not just `_degraded is True`.
3. **BLOCKER 3 — `wp_sidecar_test_stub.py` hardcoded my implementation worktree's absolute path**: `sys.path.insert(0, r"C:\Users\colto\...\Worktrees\vibe-cognition\src")` only worked because that specific worktree still existed on disk. Dropped entirely — confirmed `uv run python -c "import vibe_cognition; print(vibe_cognition.__file__)"` resolves the package via the editable venv install with no path manipulation needed; every AC2–AC6 test spawns this stub via `sys.executable`, which already is that venv's interpreter.
4. **MAJOR — protocol version-skew guard was untested, and "reject + respawn" over-promised**: a server-side skew (an old server process outliving a plugin update on disk) can never self-heal by respawning the *sidecar* — every fresh spawn re-reads the same current-on-disk code (and therefore the same `PROTOCOL_VERSION`), so it hits the identical mismatch every attempt; only the server *process itself* restarting clears it. Fixed the over-promising wording in `_sidecar_protocol.py`'s `PROTOCOL_VERSION` comment to say so explicitly. Added two tests (`test_wp_sidecar.py`): `test_protocol_version_mismatch_rejects_without_the_heavy_import_and_stays_alive` (monkeypatches the CLIENT side's `PROTOCOL_VERSION` only, talks to a REAL freshly-spawned `embeddings/sidecar.py` process — confirms the rejection happens before `_do_load` ever runs, so no heavy import is triggered, and the process stays alive) and `test_protocol_version_skew_degrades_bounded_and_never_self_heals_via_respawn` (full supervisor path, real unmocked `_spawn`, confirms bounded degrade and that a follow-up `_attempt_load()` fails identically). Writing the first test surfaced a real, related bug: `send_load()` unconditionally waited for a lock event before ever checking the response, so a rejection answered *before* the sidecar ever touches the mutex (no lock event fires) would misread as a wedged lock wait for the full `mutex_wait_timeout + 15.0`s instead of surfacing the real error promptly. Fixed by racing the lock event against the response event instead of sequencing them.
5. **MINOR — dead `_loading` field**: removed from `SidecarSupervisor.__init__` (confirmed zero other references).
6. **NIT — `ensure_ready()`'s "Never raises" claim**: `_spawn()`'s `Popen` call sat outside `_attempt_load()`'s try, so a transient `OSError` (e.g. resource exhaustion) would have propagated out of `ensure_ready()` rather than being consumed by the retry budget like any other failed attempt. Moved the spawn inside the same failure boundary — a spawn `OSError` is now just a failed attempt, retried like a timeout or an error response, making the docstring's claim actually true rather than needing to be softened.

Verified post-fix-round: full suite 754 passed / 1 skipped (net +5 tests), `uv run ruff check .` clean.

## For-the-record
- Ollama stays completely structurally untouched: `_load_embeddings_and_sync`'s ollama branch still calls `EmbeddingGenerator.from_config(config)` directly, unchanged, no supervisor involved. Verified the watchdog's removal doesn't leave ollama without ANY load-time bound: ollama's `OllamaBackend.__init__` doesn't itself validate connectivity (lazy, per-call, unchanged from before this WP) — a genuinely hung/unreachable ollama daemon was never bounded by the old watchdog either (the watchdog bounded the *sentence-transformers* import path specifically; `bg_model_load_start_time` was set before `from_config`, but ollama's `from_config` call was never slow in the first place, since it doesn't touch the network at construction time) — this WP does not regress that pre-existing shape.
