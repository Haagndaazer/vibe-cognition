## Cross-cutting: ONE shared document-identity predicate (Vince + Reginald, from the F1 post-mortem)
Today the identity question "is this a document node carrying sha X?" (`type=="document" AND metadata.sha256==sha`) is written TWICE — the dedup loop in `_store_document` (cognition_tools.py:146-148) and `_other_document_has_sha` in the delete path (operations.py:20-30). They agree only by coincidence of reading; that exact "two filters trusted to match" asymmetry CAUSED the F1 leak. D1b adds a THIRD reader (blob refcount-delete). **Before adding the third site, extract ONE shared helper** — `storage.documents_with_sha(sha) -> list[node_id]` (or a `_is_document_with_sha` predicate) — and route dedup, sidecar-delete, AND blob-refcount-delete through it. Retain and reclaim become the SAME expression structurally; with no FK to enforce it (JSONL + networkx + filesystem), the shared function IS the structural binding. **Mode refinement is caller-side, not a second predicate:** the shared helper answers identity (document + sha); the *sidecar* reclaim purges when NO document with the sha remains (any mode), while the *blob* reclaim purges when no **`mode=="copy"`** document with the sha remains (a reference-mode twin has no blob stake — peer-review B3). The mode filter lives at the call site as a one-line `.filter`, so the identity question stays single-sourced. This extraction is its own first step in Commit 3 (refactor + re-green D1a's existing delete tests, NO behavior change) before the blob logic is added.

**The guarding test calls the predicate too (Vince addendum, ledger 11):** the refcount/GC tests must assert "still referenced?" by calling `documents_with_sha` — NOT by re-encoding `type==document AND sha==` inline. One predicate expression, called by every PRODUCER (dedup), every CONSUMER (sidecar-delete, blob-delete), AND the guarding test. If the test re-encodes the filter, it becomes a drifting FOURTH reader: a future predicate change updates the function while the test re-passes on its stale inline copy — the F1 leak reborn under a green suite. Deriving "sameness" in the test from the same function the code uses also keeps the foreign-citer case (the F1 case) inside coverage automatically.

## Commit 1 — matcher pair rules (replace the D1a inert guard)
Replaces the pair-level `if DOCUMENT in (node_type, other_type): continue` guard (storage.py ~589-597) and the PART_OF-only existing-edge check (~614-617) with the full truth table from §1/§9 S4.

**Six-pair truth table (direction + ref gate):**
| pair | edge | direction | ref gate |
|------|------|-----------|----------|
| entity ↔ episode | `part_of` | entity → episode | ANY ref (unchanged) |
| entity ↔ document | `part_of` | entity → document | **`doc:`-prefixed key ONLY** |
| document ↔ episode | `relates_to` | document → episode | **`doc:`-prefixed key ONLY** |
| document ↔ document | skip | — | — |
| episode ↔ episode | skip | — | — |
| entity ↔ entity | skip | — | — |

- The ref gate is checked on the **shared key currently being iterated** (`key.startswith("doc:")`), not on the node's whole ref list — a document-involving pair links ONLY when the shared ref is a `doc:` key. This is the §9 S4 vacuum defense: an entity + document sharing only `issue:X` must NOT link.
- **Idempotency / manual-edge coexistence (§9 S4) — keyed per (from,to,EDGE_TYPE), NOT per pair (peer-review A5):** the matcher now mints TWO edge types (`part_of` for entity↔doc, `relates_to` for doc↔episode), and `add_edge` uses `edge_type` as the MultiDiGraph key (storage.py:161-164), so a same-type add OVERWRITES — a per-pair skip would let a deterministic mint clobber a curator's same-type manual edge and destroy its provenance. Correct rule: **skip if an edge of the SPECIFIC type this rule would mint already exists between (from,to), of ANY source.** That makes it idempotent (no double-mint of the deterministic edge) AND non-destructive (a manual `part_of` E→D is left untouched — the matcher does not re-mint over it). A different type on the same pair does not block — the case that matters is a manual `relates_to` not blocking a deterministic `part_of` (the two *deterministic* types live on different (from,to) pairs, so they never collide with each other). Implement by checking `self._graph.has_edge(from_id, to_id) and <minted_type>.value in self._graph[from_id][to_id]` — the existing key-based check at storage.py:614-617, generalized from the hardcoded `PART_OF` to the per-rule type. (This is simpler AND safer than the source-based skip the first draft proposed; the source field is not needed for the skip.)
- `doc:` keys pass through `_normalize_refs` as exact (they don't start with `commit:`, so no prefix expansion) — commit-SHA prefix behavior untouched (rule 21 / node `99b8a9f50164`).
- `relates_to` is recorded one-directional (document → episode). `seen_pairs` (storage.py:575) dedups only WITHIN a single call; cross-call idempotency (the store tool calls the matcher for the doc; `cognition_record` calls it for the episode) rests on the per-(pair,type) existing-edge skip above — the second call sees the first call's edge of that type and skips. Tests must assert that cross-call mechanism, not `seen_pairs`.

**Tests (rule 20):** entity→document `part_of` on a shared `doc:` ref; document→episode `relates_to` on a shared `doc:` ref; **§9 S4 vacuum — document + episode sharing ONLY `issue:X` → ZERO edges** (RUN fails-before against an any-ref impl, rule 12); doc↔doc / episode↔episode / entity↔entity skip; positive control entity↔episode `part_of` still fires (no regression); manual `relates_to` E→D present → deterministic `part_of` E→D still minted (different type, not blocked); manual `part_of` E→D present → matcher does NOT re-mint/clobber it (same-type skip, provenance preserved); re-run matcher → no duplicate edge (per-(pair,type) idempotency); cross-call order — record doc then episode vs. episode then doc → identical single `relates_to`. Remove/replace the D1a `test_document_is_graph_inert_*` test (its guarantee is intentionally superseded — note that in the commit msg).

## Commit 2 — copy mode (blob write + sanitization + size policy), PAIRED deletion lands in Commit 3
`documents.py` gains the blob path + the agent-controlled path component that D1a deferred.
- `blob_path(cognition_dir, sha, ext)` → `.cognition/documents/<sha[:2]>/<sha><ext>`.
- **Extension sanitization = whitelist-or-DROP:** keep `ext` only if it matches `^\.[A-Za-z0-9]{1,10}$`, else drop (store still succeeds — never fail on a hostile/odd MIME). The `<sha>` stem is server-generated; only `ext` is agent-controlled. This is THE agent-controlled path component (deferred from D1a precisely because D1a composed none).
- **Write-once + atomic:** if the blob path exists, skip the write (content-addressed = identical bytes; dedup/integrity-free). Write to a temp file **in the SAME directory as the final blob** (same filesystem, or `os.replace` can fail cross-device — peer-review B2) then `os.replace` onto the final name, so a crash can't leave a half-written blob under the sha name (binary mode — Windows `os.open` TEXT-mode doubling lesson from WP-4). Two concurrent writers of identical bytes both replacing is harmless (content-addressed). The TOCTOU on the exists-check is the accepted non-transactional model (Vince backlog #2) — note it in the docstring.
- `_store_document(..., store_copy=False, local_only=None)`:
  - `store_copy=True` → copy bytes into the blob (reference path still records path+sha; copy mode ADDS the blob + sets `mode="copy"`). content_text copy writes the encoded bytes.
  - **Size policy (§9 S1), at store time not push time:** `≥50 MB` → force `local_only` + warn; `≥95 MB` → default-commit REFUSED, `local_only` required (no hard cap — §8(b)). Thresholds are constants.
  - `local_only=True` → ignore the blob from git. **`.gitignore` scheme PINNED (peer-review A1):** a dedicated **`.cognition/documents/.gitignore`** (NOT the repo-root `.gitignore` — root is human-curated + git-tracked, and writing machine-generated content-addressed paths there churns a shared file). **One line per blob, the blob path relative to the documents dir (`<sha[:2]>/<sha><ext>`)** — NOT per-`<sha[:2]>` shard (a shard line would wrongly ignore a sibling default-commit blob AND breaks Commit 3's per-blob removal). Add is idempotent (no dup lines). The file is **LOCAL per machine, self-ignoring** (it includes a `.gitignore` line for itself): `local_only` is inherently a per-machine choice, so the ignore-set and the S3 promote/demote transitions stay local and need no cross-machine coordination (a committed shared list would force a teammate's `default` store of the same sha to fight machine A's committed `local_only` line). Default (not local_only) → blob committed, no `.gitignore` line.
  - Result (§9 S5): `mode`, `blob_bytes`, blob path, `local_only`/`already_committed`, alongside `indexed_text_chars`.
- **S3 dedup git-policy compare:** re-store of an existing sha — `local_only → default` **promotes** (remove the `.gitignore` entry, report `promoted: true`); `default → local_only` warns `already_committed: true` (can't un-publish). Dedup still returns the existing node (does not merge new context — state it).

**Tests (rule 20):** sanitization keeps a good ext, DROPS traversal/`..`/separators/reserved/over-long while the store still succeeds; write-once skips a second identical write (blob mtime/content unchanged); size thresholds (50/95 MB) flip git policy with the stated warning, no hard cap; `local_only` writes the `.gitignore` entry; S3 promote/demote transitions. Use small byte payloads + monkeypatched size where needed (no real 95 MB files in CI).

## Commit 4 — N1 ghost-search fix (general; own commit, rule 20/12)
The pre-existing live bug: `cognition_search` formats hits straight from Chroma with no graph-existence check, and cross-process `remove_node` replay never un-embeds. Escalated by documents (verbatim deleted client text) but fixed generally for ALL embedded nodes.
- **(a) search drops graph-absent hits — THIS is the correctness guarantee (always safe, never deletes):** in `cognition_search`, strip the chunk suffix (`_id.split("#chunk-")[0]`) and drop any hit whose stripped id fails `storage.has_node`. Requires the cognition graph in the tool — fetch `cognition_storage` from the lifespan (key `"cognition_storage"`, the same key `_sync_cognition_embeddings`'s caller uses — verify in server.py at wiring time) and route the tool's lifespan access through `get_lifespan(ctx)` (the D1a accessor). NOTE (peer-review B5): `require_embeddings(ctx)` still does its own direct `request_context` access, so the pyright baseline may not drop by the full count — **measure, do not pre-commit a number;** lower the baseline file only by the amount actually observed.
- **(b) startup reconciliation sweep — reclamation, NOT the correctness path; ordering-hardened (peer-review A4):** `_sync_cognition_embeddings` gains a pass that deletes orphan Chroma ids (incl. `#chunk-*`) absent from the graph, so storage self-heals instead of accreting orphans. **Hazard the first draft missed:** a blind "enumerate all Chroma ids, delete graph-absent" WILL delete a live embedding written by a concurrent same-machine writer whose `add_node` journal line this process hasn't caught up on — reintroducing an N1-class loss from the deletion side. Mitigations, all required: (1) force `cognition_storage` catch-up (`reload()`/`_catch_up`) IMMEDIATELY before enumerating, so the graph reflects the latest journal; (2) compute the orphan set, and **only call `delete(ids=orphans)` when `orphans` is non-empty** — `_collection.get(ids=[])`/`delete(ids=[])` RAISE `ValueError` on empty lists (peer-review A3; the existing sync is saved only by its `if not all_nodes: return` guard at server.py:60); (3) enumerate via `_collection.get()` (no-arg = all, safe) NOT `get(ids=[])`. A residual cross-process TOCTOU remains (a write landing between catch-up and delete) — document it as the accepted non-transactional-model race (Vince backlog #2); the query-time filter (a) means it never causes a WRONG search result, only a transiently-late reclamation.
- **(c) chunk purge** already wired in Commit 3.

**Tests (rule 20, the §9 N1 harness):** two `CognitionStorage` over ONE journal with INDEPENDENT real ChromaDB stores in tmp (Chroma is gitignored/local; a fake can't prove the cross-process replay path) — record+embed a NORMAL node on A; B syncs (embeds into B's Chroma); replay a `remove_node` tombstone into B's graph (graph-only, mirrors cross-process — `remove_node` never un-embeds); **assert B's `cognition_search` returns nothing** (RUN fails-before against current `main` — the ghost surfaces, rule 12). Separately: seed an orphan id in a Chroma store whose graph LACKS it, run the hardened sweep, assert it's deleted; AND a guard test — a node present in the graph (catch-up applied) is NOT swept (proves the ordering fix, not just the happy path). Focused unit: fake embedding storage returns a hit whose id isn't in the graph → search drops it (backstops (a)).
