WP-Lifecycle (P1, docs/wp-lifecycle-plan.md rev 3): orphan servers must die with their parent. Belt-and-suspenders, both server-side, per the brief's four scope items.

## §L-a: ancestor-death watch (primary, works even mid-wedge)
New module `src/vibe_cognition/lifecycle.py` (ctypes over pywin32, no new runtime dependency). Topology fact driving the whole design: on Windows there is no exec, so `uv run ... python -m vibe_cognition.server` means our direct parent is uv, and uv never dies first (watching only the direct parent would deadlock the pair forever). `arm_ancestor_watch(depth=2)` walks the chain via `NtQueryInformationProcess(ProcessBasicInformation)` (own pid -> parent, then parent's own PBI -> grandparent), opens `SYNCHRONIZE | PROCESS_QUERY_LIMITED_INFORMATION` handles on both, and blocks a daemon thread in `WaitForMultipleObjects` — either dying forces `os._exit(0)`. `depth` is a parameter (not hardcoded), per the brief's explicit reuse requirement for WP-Sidecar (depth=1, parent=server, no intermediary).

OpenProcess semantics implemented exactly as specified (not looser):
- NULL on the **direct parent** (pid already gone) → genuinely orphaned, exit now.
- NULL on the **grandparent** → NOT fatal (a launch shim that legitimately exits right after spawning uv is a normal topology); degrades to uv-watch + pipe-watch only, breadcrumbs the resolved chain (pids + image names via `QueryFullProcessImageNameW`).
- `ACCESS_DENIED` on a live process → falls back to slow polling (`_pid_is_alive` via `GetExitCodeProcess`), never exits, never crashes the watch.
- PID-reuse guard (`is_younger_than_self`): validates via `GetProcessTimes` that an opened ancestor's creation time is EARLIER than our own start time; a reused pid (necessarily younger) is treated as "gone," not trusted. Unit-tested directly even though the race itself is hard to drive for real.

POSIX fallback: `os.getppid()` polling — deliberately not over-engineered for a platform the fleet doesn't run. Pure Python (no ctypes), so its test (`test_wp_lifecycle.py::test_arm_ancestor_watch_posix_detects_reparenting`) runs on every platform, not gated behind the Windows-only skip marker the rest of `lifecycle.py`'s coverage carries.

## §L-b: pipe-closure watch (secondary, loop-independent by requirement)
`arm_stdin_watch()`: a dedicated daemon thread polls stdin via `PeekNamedPipe` (detects `ERROR_BROKEN_PIPE` without consuming data), gated on `GetFileType(stdin) == FILE_TYPE_PIPE` so a console/dev run (FILE_TYPE_CHAR) is skipped entirely, never exits on it. On a confirmed broken pipe: sleeps the 5s grace (letting the loop-riding MCP-conventional EOF path finish on its own if the loop isn't actually frozen), then `os._exit(0)` unconditionally — if graceful shutdown already completed, the process is already gone and this line never runs. Any other `PeekNamedPipe` failure is NOT treated as a confirmed closed pipe (avoids a false-positive exit on a transient error).

## §L-c: startup ordering guarantee
Both watches are armed in `lifespan()` immediately after `prewarm_dispatch_executor()` and strictly before `bg_thread.start()` — same reasoning as WP-Wedge's warm-spawn and WP-Wedge-2's dispatch-executor prewarm: the watch threads' OS threads must exist before any loader-lock wedge can block further thread creation. New test `test_wp_lifecycle.py::test_both_watches_armed_before_bg_thread_starts` extends the WP-Wedge-2 ordering-test pattern; failure-drive confirmed (temporarily moved the two `lifecycle.arm_*` calls after `bg_thread.start()` — test failed with the exact expected diff, then reverted).

## §L-d: breadcrumb-retention sweep
Verified the existing v0.15.3 sweep (`_startup_timing.prune_old_logs`, age + keep-N rules) against this incident's evidence. Bumped `_PRUNE_KEEP_RECENT` 50 -> 200: the orphan-churn incident produced far more startup events than 50 assumed (4 wedged pairs within 50 minutes via pure hang->reconnect churn); each file is a few bytes of text, so the wider margin is free forensic headroom. No new retention machinery, per the brief. New test pins the default (`test_prune_default_keep_recent_is_200`).

## Journal/chroma hard-exit safety (§5, cited not re-proven)
The three real guarantees the brief calls out were already covered by existing tests, cited rather than duplicated: torn/malformed trailing journal lines are skipped-and-later-ingested (`test_cognition.py::test_torn_trailing_line_is_eventually_ingested`, `::test_corrupt_complete_line_is_skipped`, `::test_malformed_journal_lines_skipped`); chroma writes are sqlite-transactional (rollback on interruption, unchanged, no new test needed); the additive startup backfill sync self-heals a committed-journal-but-missing-vector state (pre-existing WP-4 coverage, unchanged). No flush choreography added to any death path, per the brief.

## Honesty bar
This WP bounds the SERVER's own exit guarantee two independent ways; it does not and cannot fix the client-side reap (out of scope per the brief, evidence already filed in the graph). The grandparent-degraded-arm path (a launch shim that exits right after spawning uv) is exercised only via unit-level mocking (`test_lifecycle.py`) — driving it with a REAL three-tier disposable-shim-exits-immediately topology would need a fourth process tier in the integration harness, which the brief's three ACs don't call for; flagging this as a coverage gap, not a defect, in case a future WP wants that specific real-topology proof. `os._exit(0)` remains deliberate throughout — no death path was "improved" into a graceful one.

## Gate fix round (3a6f82f → this commit)
1. **MANDATORY — silent WAIT_FAILED busy-loop**: `_watch()`'s `WaitForMultipleObjects` call never distinguished `WAIT_FAILED` from a benign timeout; with no poll fallback present, the loop re-issued an INFINITE wait in a tight silent retry (a `WAIT_FAILED` result returns near-instantly), permanently disabling the primary guarantee with zero signal. Fixed: `watched_handles` entries now carry `[kind, handle_or_None, pid]` uniformly, and a `WAIT_FAILED` result degrades every currently-waited handle to polling by pid (can't tell which one failed) with a stderr line, rather than retrying blindly. New regression test; failure-drive confirmed it hangs (pytest-timeout has to kill it) without the fix. Discovered as a real side effect while fixing this: `GetCurrentProcess()`'s pseudo-handle is invalid input to `WaitForMultipleObjects` (`ERROR_INVALID_HANDLE`) — several existing unit tests were unknowingly relying on the OLD bug (a busy-loop that never called `exit_fn`) to "pass"; fixed those test doubles to use a real `OpenProcess`-obtained handle instead.
2. **MANDATORY — timeout-budget self-consistency**: `test_wp_lifecycle_integration.py`'s `_ARM_TIMEOUT_SECONDS` equaled pyproject's global `pytest-timeout=60`, and each AC calls the wait-helper 2-3x sequentially — on a slow runner, pytest-timeout would hard-kill the test opaquely before any individual wait's own bound was reached. Dropped to 20s and added explicit `@pytest.mark.timeout` overrides (90s/120s) sized for each AC's own worst-case sum.
3. **CHEAP — missing grandparent-REUSED unit test**: added the sibling of the existing grandparent-GONE test (`test_arm_ancestor_watch_grandparent_reused_is_degraded_not_fatal`), completing the degrade-matrix coverage.
4. **CHEAP — PR body numbers**: corrected the WPL-AC5 count above.
5. **CHEAP — POSIX test wrongly gated behind the Windows-only skip marker**: moved `arm_ancestor_watch_posix`'s test out of `test_lifecycle.py` (whose module-level `pytestmark` skips non-Windows) into the already-platform-independent `test_wp_lifecycle.py`, where it now genuinely runs everywhere as its docstring always claimed.
6. **Your-call — defensive degrade around the two arm calls**: agreed and implemented. `lifespan()` now wraps each `lifecycle.arm_*` call in its own `try/except Exception` + `logger.warning`, consistent with this WP's own established philosophy elsewhere (NULL-grandparent, ACCESS_DENIED) of "even if part of this doesn't work, keep serving" — an unexpected failure arming an optional safety net (e.g. thread creation itself failing) must not crash the entire server startup. New test proves startup still completes when both arm calls raise; failure-drive confirmed it crashes lifespan() without the guard.

Verified post-fix-round: full suite 740 passed / 1 skipped, ruff clean (see WPL-AC5 above for the flake note).

## For-the-record
- `uv run` cold-resolve cost (`--no-sync` still needs to locate the venv) made the integration tests' arm-wait helper timeout (20s per call as of the gate fix round) necessary as a generous-but-budget-consistent ceiling; observed actual arm times were well under 5s once `uv run` itself started, on this machine with a warm `.venv`.
- The `.venv/Scripts/python.exe` uv creates on Windows is its own small trampoline binary (257KB vs the base CPython install's ~105KB) — an early smoke-test iteration that invoked it directly (bypassing `uv run`) produced a SEPARATE grandchild pid from `subprocess.Popen`'s reported pid, which briefly looked like a bug in the ancestor-walk before being traced to the trampoline itself adding an unplanned process tier. Not a defect in this WP's code; noted here since it's a nonobvious Windows/uv interaction that could resurface elsewhere.
- A second nonobvious Win32 constraint, this one caught by the gate: `GetCurrentProcess()`'s pseudo-handle (documented as valid for many process APIs) is flatly rejected by `WaitForMultipleObjects`. Production code was never exposed to this (the real code path always uses genuine `OpenProcess`-obtained handles); only my own test doubles' convenience shortcut hit it. Detailed in fix item 1 above.
